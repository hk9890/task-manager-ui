## Description

Gate task for epic beads-workbench-r1ou. Verifies the restructured test architecture lands cleanly with no coverage loss and hits the stated performance targets.

## Acceptance Criteria

- [ ] **All child implementation tasks closed**
- [ ] **Unit tier perf**: `mise run test` completes in <10s wall time on the baseline hardware (was 18.6s post-i1th)
- [ ] **Integration tier perf**: `mise run test:integration` completes in <30s wall time (was 96s post-i1th, dominated by per-test fixture rebuilds)
- [ ] **Coverage parity**: every observable behavior asserted by the old Bucket A integration tests is still asserted somewhere — either in a unit test (using `FakeBeadsGateway`) or in the contract suite (against real bd). Audit by re-reading the deleted `internal/app/model_integration_test.go` from git history and confirming each `t.Fatalf`/`t.Errorf` site has a counterpart.
- [ ] **Gateway contract coverage**: every read method on `BeadsGateway` (12 methods) is exercised by `RunReadContract`. Verify by grepping for `t.Run` blocks in the contract package and matching against the interface definition in `internal/gateway/beads/interface.go`.
- [ ] **Fake/real parity**: both `TestFakeGatewayReadContract` and `TestRealGatewayReadContract` pass. If any fake adjustments were needed during contract implementation, they were captured as fix commits or tickets — no hidden fake drift remains.
- [ ] **Mutating coverage**: all 4 write methods (CreateIssue, UpdateIssue, CloseIssue, AddComment) are exercised by mutating scenarios. Link/dep mutations are exercised by scenario 2.
- [ ] **No silent test deletions**: total test count (counting subtests where appropriate) does not drop in a way that represents lost coverage. Net count may go down due to consolidation, but the *behavioral surface* covered must not shrink.
- [ ] **Docs in sync**: `docs/TESTING.md` reflects the new tiering, fixture-sharing rules, and contract-test policy.
- [ ] **`mise run quality` green** end-to-end.

## Notes

- A small net-negative test count is expected (we're consolidating 8 app-layer integration tests into ~12 contract subtests + a couple of scenarios + unit-test ports — the actual count depends on how much existing unit coverage already overlapped).
- If the integration-tier target (<30s) is missed by a small margin, document the cause before accepting — likely either a bd subprocess that's slower than expected, or a scenario that grew beyond ~10 steps.
