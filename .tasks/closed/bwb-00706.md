## Problem

After [[h2xu]] landed, cold-start placeholders have clear-shaped pulsing skeletons. But **refresh** state on board, search results, and details panes still shows stale data + header braille spinner only — no per-surface motion signal that "this surface's data is being replaced". On a slow refresh the user has no local feedback that anything is happening.

## Surfaces affected

- `internal/ui/board/board.go:209-225` — refresh path keeps stale rows visible.
- `internal/ui/details/details.go:108-110` — refresh path falls through to normal detail layout.
- `internal/ui/search/search.go:283-285` — skeleton path only fires when `Loading && len(Results) == 0` (cold-start); refresh-with-existing-results has no in-rail indicator.

## Pinned by tests (must update, not remove)

- `internal/mode/details/model_test.go:722-779` — `TestRefreshSameIssueKeepsStaleContent`
- `internal/ui/search/search_test.go:340-363` — `TestRenderRefreshKeepsStaleResults`

## Design decision (resolved during plan-review pass)

**Chosen approach:** Dim the stale rows by one shade, foreground-only, driven by the existing `loading.SkeletonPhase` value (same source as the h2xu.3 cold-start pulse). The dim cycles through `styles.SkeletonShades` so refresh motion matches the established cold-start animation language.

**Rejected:**
- Bottom-edge progress line — too intrusive; conflicts with status bar.
- Cycling pane border color — board/details/search do NOT have borders; only toaster/modal do (`internal/ui/toaster/toaster.go:64`, `internal/ui/modal/modal.go:349`). Adding borders to satisfy this animation would be a major layout change out of scope.

**Selection-conflict rule:** On a selected row, dim foreground only — never the selection background — so the selection highlight remains visually dominant during refresh.

**Header spinner kept:** The header spinner is the global aggregate; per-surface dim is the local signal. Both serve distinct purposes (e.g., board on screen while details refreshes in background). Removing the header spinner is out of scope.

**Animation tick source:** Reuse existing infrastructure. `loading.TickMsg` already advances `spinnerFrame` unconditionally (`internal/app/model.go:419-421`), and `SkeletonPhase(m.spinnerFrame)` is already passed into mode `View()` calls (`internal/app/model.go:948-958`). No new tick source needed.

## Out of scope (deferred)

- Narrower in-flight states (e.g., single-issue placeholder at `internal/app/model.go:881`) — already substitutes a cold-start-like placeholder; piling on a refresh dim would be a third visual language. File a follow-up if needed.
- Renaming the board mode logger to `component=board` — tracked separately in [[okmo]].
