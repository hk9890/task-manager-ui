## Symptom

Pressing Ctrl+R quickly multiple times in the dashboard causes the board to
"not work properly anymore" — UI feels stuck/laggy, columns flicker, selection
jumps unexpectedly, and bd subprocess invocations balloon to 10-30s durations.

## Evidence

`~/.local/state/bwb/bwb.log` shows session `d3bad156` with stacked overlapping
`bd` subprocess calls and abnormal durations:

- `bd ready --explain --json` → 12495ms, 7120ms, 29043ms, 18271ms
- `bd query status=in_progress --json` → 12821ms, 13990ms, 18050ms
- `bd query status=closed --json` → 12539ms, 15826ms, 18271ms, 9089ms

These are normally sub-second operations. The pattern is consistent with many
batches of 4 in-flight bd subprocesses queued by repeated Ctrl+R presses.

## Root Cause

`internal/mode/board/model.go`:

- Line 233-234: BoardActionReload handler calls `startReload(refreshModeManual)`
  **with no in-flight guard**.
- Compare line 322-328: `AutoRefresh()` correctly guards with
  `if m.IsLoading() { return nil }`.

Each Ctrl+R press immediately dispatches a fresh `tea.Batch` of 4 gateway
commands (`loadReadyExplainCmd`, `loadInProgressCmd`, `loadClosedCmd`,
`loadClosedCountCmd`) even when a previous batch is still in flight.

Resulting failure modes:

1. **Subprocess stacking** — N presses ⇒ 4N in-flight `bd` subprocesses;
   duration grows from contention.
2. **`pendingResults` goes negative** — `startReload` resets the counter to 4
   and clears the partial slots, but old-batch result messages keep arriving
   and decrement past zero (line 163, 171, 179, 187). Once negative,
   `maybeCompose()`'s guard `if m.pendingResults > 0` (line 377) still passes,
   so composition fires repeatedly with stale slots overwritten by trickling
   newer-batch results — flicker / wrong data.
3. **Selection lost on every press** — for `refreshModeManual` lines 358-363
   reset `focusedColumn=0` and `selectedRow={}`. Hammering keeps yanking the
   user back to column 0 row 0.

## Acceptance Criteria

- Ctrl+R while a reload is in flight is either ignored, coalesced into a single
  pending reload, or cancels the in-flight batch — *not* stacked.
- `pendingResults` cannot go negative; results from a superseded batch are
  discarded (e.g., via a batch generation/epoch counter).
- Selection state is not gratuitously reset when the user just wanted to refresh
  (consider reusing the auto-refresh anchor for manual reloads too).
- Repro: pressing Ctrl+R 10 times in 1 second should produce at most one
  active batch of 4 gateway calls in the persistent log.
