## Description

Each dashboard section currently requests a hardcoded 25 items (`defaultSectionLimit` in `builtin_provider.go`) regardless of terminal height. At a 24-line terminal a section can show ~21 rows. The limit should be computed from the board's known `height` and applied to section queries before they are dispatched â€” not baked into the section definition.

## Context

- Board model tracks height: `internal/mode/board/model.go` field `height int`, set via `SetSize(width, height int)`
- Section queries dispatched in `loadSectionCmd` at `internal/mode/board/model.go:470`
- Section definitions come from `builtin_provider.go` with hardcoded `Limit: 25`
- Each issue row is exactly 1 terminal line tall (`RenderCompact` returns a single line)
- **Actual rendering math** (verified): `columnHeight = max(3, height-1)` (`board.go:65`), then `innerHeight = columnHeight - 2` (`section.go:55-58`). Real per-section capacity = `max(1, height-3)` â€” NOT `height-2`.
- All 3 query types (`IssueListQuery`, `ReadyIssuesQuery`, `BlockedIssuesQuery`) have a `Limit int` field

## Instructions

1. Remove `defaultSectionLimit = 25` from `internal/dashboard/builtin_provider.go`. Set `Limit: 0` in all 4 section factory functions (meaning "caller will set").

2. In `internal/mode/board/model.go`, add:
   ```go
   func (m *Model) sectionItemCapacity() int {
       rows := m.height - 3 // matches actual FormSection inner height: max(3,h-1)-2
       if rows < 1 {
           rows = 1
       }
       return rows
   }
   ```

3. **Handle zero-height at startup:** `Init()` fires before the first `tea.WindowSizeMsg` so `m.height == 0`. Default to a sane minimum when height is unknown:
   ```go
   func (m *Model) sectionItemCapacity() int {
       if m.height == 0 {
           return 20 // safe default before first WindowSizeMsg
       }
       rows := m.height - 3
       if rows < 1 {
           rows = 1
       }
       return rows
   }
   ```

4. In `loadSectionCmd` (or the dispatch site), apply the capacity to the section query before firing:
   - `QueryTypeListIssues`: `section.Query.ListIssues.Limit = capacity`
   - `QueryTypeReadyIssues`: `section.Query.ReadyIssues.Limit = capacity`
   - `QueryTypeBlockedIssues`: `section.Query.BlockedIssues.Limit = capacity`

5. **Resize policy (explicit):** Do NOT re-issue section queries on terminal resize. Resize adjusts display truncation only â€” the section already clips content to height via `FormSection`. Section queries only fire on initial load, explicit reload, and post-mutation reload. This avoids a subprocess storm when the user drags a tmux pane.

6. Update tests that reference the old constant:
   - `internal/dashboard/builtin_provider_test.go` lines 58, 59, 66, 67, 77, 78, 94, 95 â€” assert `Limit == 0` (provider no longer sets it)
   - `internal/mode/board/model_test.go` lines 56 and 526 â€” assert limit is set by the board before dispatch

7. **Document the provider contract change** in a code comment on `dashboard.Section.Query` (or in the `Limit` field godoc): "Limit==0 means the caller (board model) will set an appropriate display limit before dispatching. Custom dashboards must not assume Limit>0."
