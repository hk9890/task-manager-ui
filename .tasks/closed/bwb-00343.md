What I did: In search mode (590-issue scale fixture) typed query text and observed the Results pane before pressing Enter; then pressed Enter; cross-checked internal/mode/search/model.go and docs/user-guide/key-bindings.md.

Expected (source: docs/user-guide/key-bindings.md Search Mode — 'Typing while the query field is focused updates the search query directly'): typing a query produces results for that query, or it is clear that it has not run yet.

Actual — two coupled problems:

1. Stale results look authoritative under a draft query. After typing 'zqx99' (a 0-match query) the Results pane still shows the previous query's 25 rows, header still reads 'Search: 25 results', badge still '25 capped'. Nothing visually ties the displayed results to the *previous* query. The ONLY indicator that these results do not match the typed query is a 5-char border label flipping 'shown' -> 'draft' on the Search rail. A reasonable user typing 'zqx99' sees 25 real issue rows and concludes those 25 match 'zqx99'. They do not. Search actually executes only on Enter (model.go:186: Enter && focus==FocusQuery -> triggerSearch); after Enter the results correctly become '0 partial / No matches for zqx99' and the badge flips to 'shown'.

2. Enter-to-search is undocumented. docs/user-guide/key-bindings.md Search Mode lists 'enter' ONLY as 'open selected result in detail mode' and never states that Enter submits/runs the query when the query field is focused. The phrase 'updates the search query directly' implies a live/immediate search, which is false. Enter is overloaded (submit query vs open detail by focus) and the submit half is absent from both key-bindings.md and the configurable search actions in docs/CONFIGURATION.md. A user following the docs has no documented way to discover how to actually run a search.

Note: the in-app empty-state DOES say 'Type query, then press Enter.' — but that hint is only shown while results are empty; once stale results occupy the pane the hint is gone.

Severity: medium. Core browse flow; misleads the user's mental model (FEW HICCUPPS: World, Explainability, Claims) and the canonical keybindings doc is wrong/incomplete.

Suggested direction: while draftQuery != appliedQuery, visibly mark the Results pane as stale (dim it / overlay 'Press Enter to search for "<draft>"') rather than relying only on the rail badge; and document Enter's submit-query role in key-bindings.md.
