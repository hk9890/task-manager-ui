## Description
Direct user runtime feedback found that search query entry still does not behave like a real text field. In the current search controller, several letter keys (`j`, `k`, `h`, `l`, `r`, `/`) are reserved for search-mode navigation or actions even when the query field is focused. This means users cannot type normal text containing those letters into the search query.

## Evidence
- User asked whether `j`/`k` can be typed into the search field; current implementation swallows them
- `internal/mode/search/model.go` handles `j`, `k`, `h`, `l`, `/`, and `r` before normal query insertion logic
- Current behavior makes the field feel broken as a text input

## Instructions
1. When the query field is focused, normal text entry must win over pane-navigation hotkeys.
2. Keep result navigation shortcuts available when results pane is focused.
3. Ensure query editing behaves like a normal text field for ordinary letters.
4. Update search tests to cover previously swallowed letters.
5. Re-verify runtime search interaction after the fix.

## Files to Inspect
- `internal/mode/search/model.go`
- `internal/mode/search/model_test.go`
- related app/search tests

## Acceptance Criteria
- [ ] Users can type ordinary letters like j/k/h/l/r into the focused search query
- [ ] Result navigation shortcuts still work when results pane is focused
- [ ] Search tests cover the focused-query text entry behavior for previously swallowed letters
