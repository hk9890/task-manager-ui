## Problem

The gateway read cache (`internal/gateway/beads/cache.go`) keys its change-token on the mtime of `.beads/last-touched`. When that file does not exist, `currentToken()` returns `(zero, false)`, which causes `get()` to always miss and `set` to never store â€” the cache is silently disabled for the runner's lifetime.

`.beads/last-touched` is written by bd only on `create/update/show/close` operations (upstream commit a34f189). Projects where bd has never been used in those ways from the CLI never get the file, so bwb's cache stays dead.

## Empirical evidence

Comparing two bwb sessions, same 60s board auto-refresh, same 4 commands:
- `ai-marketplace` (no `last-touched` file): 4 bd calls per minute, every minute, every tick. Hit rate ~0%.
- `beads-workbench` (file exists from dogfood usage): whole idle minutes show zero gateway calls. Hit rate ~100% during idle.

## Scope

1. In `NewCommandRunner` (or in `newReadCache`), bootstrap `.beads/last-touched` to an empty file if missing. Verified empty is safe: bd treats empty file same as 'no last touched ID' for its Get path and does not reject it.
2. Add per-cache-hit logging at INFO level so we can measure hit/miss ratios going forward from the existing bwb-*.log files. Mirror the `bd command finished` slog format with a `bd command cache hit` message and the same operation/argv fields.
3. Tests for both: bootstrap creates the file when missing, leaves it alone when present, and cache hits actually fire after a project starts with no token file. Logging tests follow the pattern of existing `TestCommandRunnerRunLogsExecutionTrace*` tests.
4. Update `docs/MONITORING.md` with the new log message so the diagnostics surface stays in sync.
