## Description
Verifier review of beads-workbench-ekp found that architecture docs still do not explicitly describe row responsibilities, optional list responsibilities, panel responsibilities, and the separate dedicated detail component in terms of the implemented code.

## Evidence
- docs/CODING.md package layout still describes only generic ui reuse and loading/modal/toaster primitives
- grep across docs found no explicit row/list/panel responsibility documentation for this epic decision
- project-plan/ARCHITECTURE.md remains high-level and does not document the implemented board/search shared-row-or-not and shared-list-or-not outcome
- internal/ui/details/details.go still exists as the dedicated detail component, but docs do not call out that separation in the architecture language expected by the epic plan

## Why
beads-workbench-ekp.2 and .3 require docs that reflect the implemented architecture, not an aspirational one. Without explicit documentation, acceptance review cannot verify whether list extraction was justified or intentionally rejected.

## Acceptance Criteria
- [ ] docs/CODING.md explicitly describes row, optional list, panel, and detail responsibilities as implemented.
- [ ] If no shared list component exists, the docs say row-level sharing is sufficient for now and explain why.
- [ ] The docs clearly preserve internal/ui/details as the dedicated detail component.
- [ ] Acceptance review can point to concrete documentation matching the code tree.
