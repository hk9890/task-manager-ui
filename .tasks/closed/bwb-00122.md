Introduced today by commit 4ad4cee (epic 9x70 write contract). Fails ~50% of runs locally with race detector enabled.

## Symptom
```
contract_write.go:230: CloseIssue (second, idempotency): expected nil error, got close issue: command exited with code 1: Error closing bd_001-<suffix>: issue not found: bd_001-<suffix>
```

## Reproduction
```
go clean -testcache
for i in 1 2 3 4 5; do
  CGO_ENABLED=1 go test -tags=integration -race -run 'TestRealGatewayWriteContract/CloseIssue/Idempotency' ./internal/gateway/beads/contract/ 2>&1 | tail -3
done
```

## What's known
- TestRealGatewayWriteContract uses datasets.WritableTempFixture (fresh t.TempDir + bd init) inside each subtest factory call, so subtests run on independent bd databases. Inter-subtest interference SHOULDN'T be possible by construction.
- Real factory at internal/gateway/beads/contract/real_write_contract_integration_test.go:30-40 creates a new gateway+workdir per subtest.
- Yet the second CloseIssue call on the same just-created ID sometimes returns 'issue not found' from bd 1.0.4.
- Suggests either: (a) bd close-then-close is not actually idempotent in 1.0.4, (b) timing/fsync race where the issue write hasn't fully landed before the second close reads, or (c) the t.Parallel() siblings somehow share state via a process-global default.

## Impact
- Blocked CI on the 4 dependabot PRs this session; PRs #1-4 were merged on red CI as a workaround.
- Main CI is currently red ~50% of the time on Ubuntu (this is the only post-fix failure left).

## Suggested next steps
1. Reproduce in isolation: `go test -count=20 -run 'TestRealGatewayWriteContract/CloseIssue/Idempotency$' ...` to confirm rate.
2. Check whether bd 1.0.4 documents close-on-already-closed as idempotent.
3. Instrument the second close to inspect the on-disk state of the .beads dir between the two calls.
