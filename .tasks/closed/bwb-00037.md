## Description
Analyze and redesign how Beads Workbench handles potentially incomplete result sets returned from capped `bd` queries, with particular focus on search semantics and the data-access contract between the UI/domain layers and the beads gateway.

This is not a narrow bug fix. It is a product and architecture initiative to ensure BWB can reason correctly about result completeness instead of inferring completeness from capped item counts. The work may require adapting the data access interface, gateway result metadata, UI contracts, and runtime behavior.

## Success Criteria
- [ ] The product has an explicit design for handling search/result completeness.
- [ ] The chosen design explains the data-access contract for complete vs partial/capped results.
- [ ] Follow-on implementation tasks are created with clear scope and dependencies.
- [ ] An acceptance review confirms the design outcome and any resulting implementation work.

## Scope Notes
- Focus on search/result completeness first; board/dashboard list caps may be evaluated separately if the design indicates shared primitives are needed.
- Evaluate whether the current gateway/domain interfaces can express truncation/partial-result state without fragile stderr coupling.
- Prefer a ground-up analysis over a tactical limit bump.
