## Outcome (per review finding B3)

The argv-level runner cache (`internal/gateway/beads/cache.go` and surrounding plumbing) is deleted. Runs AFTER 8pxi.5's perf validation so we can isolate any regression: if perf drops here, the cache was load-bearing and we have a documented choice (revert, or accept the delta with justification).

## Why this is separate from 8pxi.3
8pxi.5 measures post-refactor perf against the pre-refactor baseline (~5s cold, ~85% hit). That baseline INCLUDES the argv cache. Deleting the cache and measuring in one shot confounds "E2 broke perf" with "argv cache was load-bearing."

## Files to delete
- `internal/gateway/beads/cache.go`
- `internal/gateway/beads/cache_test.go`
- Cache-related plumbing in `internal/gateway/beads/runner.go` (the cache field, `logCacheHit` method, the bootstrap call, the cache-check fast path)
- Cache-related tests in `internal/gateway/beads/runner_test.go` (TestCommandRunnerLogsCacheHit, TestCacheHitsAfterBootstrap, etc.)
- `.beads/last-touched` bootstrap (in cache.go) -- becomes dead code once cache is gone

## Validation
- Re-run `scripts/analyze_dashboard_perf.py --project beads-workbench` against a fresh session AFTER deletion
- Compare to the post-refactor baseline captured in 8pxi.5
- Acceptable delta: cold-load no more than 1.5x of 8pxi.5 number; hit rate not applicable (cache is gone)
- If unacceptable: file a follow-up bug to investigate, or revert the cache and document the requirement to keep it

## Documentation updates
- `docs/MONITORING.md` -- remove the "Cache hits emit bd command cache hit" section; replace with whatever the repository-layer cache emits (if anything)
- Update `scripts/analyze_dashboard_perf.py` log strings if the new repository-layer cache uses different message names
