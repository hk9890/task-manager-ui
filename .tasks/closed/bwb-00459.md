## Description

The fatal-error screen (shown when bd's startup health check fails — "no beads project here" or "beads is not available") does **not** honor `ctrl+q`, the documented global quit shortcut (`docs/user-guide/key-bindings.md` line 11). The fatal-screen key handler hard-codes `q` and `ctrl+c` as quit triggers and ignores everything else, so `ctrl+q` is silently swallowed.

The pinning test `TestFatalErrorScreen_CtrlQQuits` in `internal/app/fatal_error_quit_test.go` is currently marked `t.Skip` referencing this issue. Companion tests `TestFatalErrorScreen_QKeyQuits` and `TestFatalErrorScreen_CtrlCQuits` pass — confirming `q` and `ctrl+c` work and the user is never trapped on the fatal screen.

(The original filing also claimed `q` and `ctrl+c` were broken — those parts are **withdrawn** as PTY harness artifacts. The remaining real bug is the `ctrl+q` swallow.)

## Expected vs Actual

- **Expected**: `ctrl+q` is the documented shell global quit. The fatal-error screen, being part of the app shell, honors it.
- **Actual**: At `internal/app/model.go:354`, the fatal-screen handler matches only `msg.String() == "q"` or `msg.String() == "ctrl+c"`. `ctrl+q` falls through. The early-return on the `fatalErrTitle != ""` branch also makes the normal shell's quit handler unreachable, so the `m.keys.Match(config.ShellContext, config.ShellActionQuit, msg)` path that DOES bind `ctrl+q` never sees the keypress.

## Root Cause (confirmed)

`internal/app/model.go:347-358`:

```go
if m.fatalErrTitle != "" {
    switch msg := msg.(type) {
    case tea.WindowSizeMsg: ...
    case tea.KeyMsg:
        if msg.String() == "q" || msg.String() == "ctrl+c" {
            return m, tea.Quit
        }
    }
    return m, nil
}
```

`ctrl+q` is not in the set; the early `return m, nil` prevents fall-through to the normal shell quit handler.

## Instructions

1. Replace the hard-coded `q` / `ctrl+c` check with a call to the same predicate the rest of the shell uses for quit:
   ```go
   if m.keys.Match(config.ShellContext, config.ShellActionQuit, msg) {
       return m, tea.Quit
   }
   ```
   This routes `q`, `ctrl+c`, **and** `ctrl+q` through the configured quit binding.
2. Verify the predicate is reachable from the fatal-screen state — `m.keys` is initialized before the fatal screen can be rendered, so this should already hold; confirm by reading the construction order in `NewModel` / `Init`.
3. Un-skip `TestFatalErrorScreen_CtrlQQuits` in `internal/app/fatal_error_quit_test.go`.
4. Confirm the three fatal-screen quit tests all pass (`q`, `ctrl+c`, `ctrl+q`).
