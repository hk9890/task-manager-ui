## Description

- `bd ready --explain --json --limit <N>`

## Contract restated (key clauses — see interface.go for full text)

**Preconditions:**
- ctx must be non-nil.
- `opts.Limit > 0` → forwarded as `--limit`; effect on blocked list unspecified by bd.

**Postconditions:**
- `result.Ready` = issues with no unresolved dependency blockers.
- `result.Blocked` = issues with ≥1 dependency blocker; `BlockedBy` populated as enriched `IssueReference` objects (ID, Title, Priority, Status — RICHER than `BlockedIssues`).
- `result.TotalReady` = `len(result.Ready)` (no separate pre-limit total).
- `result.TotalBlocked` = number of dependency-blocked issues.
- `result.CycleCount` = number of dependency cycles detected (typically 0).
- `result.Blocked[i].Issue.Status` is typically "open" (not "blocked").

**Error semantics:**
- `ErrorCodeCommandFailed` on non-zero exit.
- `ErrorCodeDecodeFailed` on unparseable stdout.

**bd quirks observed:**
- 2026-05-16: returns top-level JSON object (not array, unlike `bd ready --json`); decoder path differs.
- 2026-05-16: `schema_version` field present in payload; decoder ignores.
- 2026-05-17 (faif.5 scale audit): IGNORES the default 100-item cap that `bd ready --json` applies. At ~590 issues with 507 ready, explain returns all 507 while plain `bd ready --json` returns only the first 100. ReadyExplain is authoritative for the board model.
