Follow-up from beads-workbench-oo2b latency investigation. When sibling child issues are loaded concurrently, each ShowIssue independently re-fetches the same parent via parentChildSiblings because parentSiblingCache (internal/gateway/beads/read_gateway.go ~836-878) is only populated AFTER the fetch returns. Concurrent loads all cache-miss and spawn duplicate 'bd show <parent>' subprocesses, amplifying a fan-out that pushed observed latency to 6.1s.

Fix: wrap the cache-miss path in parentChildSiblings with golang.org/x/sync/singleflight keyed on parentID so concurrent sibling loads collapse onto a single parent fetch. Optionally add a small read-concurrency semaphore in CommandRunner.Run as broader hardening.
