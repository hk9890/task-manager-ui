## Description

Running `mise run bwb` against a multi-issue DB shows TWO rows of column-top borders stacked, each with a different Done count (e.g. one row "50", the next row "56+"). Subsequent column body rows render correctly underneath. The duplicate header rows appear in the same `View()` capture, not as terminal scrollback.

## Expected vs Actual

- **Expected**: Exactly one row of column-top borders (`╭─ Not Ready ─╮  ╭─ Ready ─╮  ╭─ In Progress ─╮  ╭─ Done ─╮`).
- **Actual**: Two consecutive rows of column-top borders, with different Done counts on each row.

## Root Cause Hypothesis (not confirmed)

The two different Done counts (50 then 56+) suggest two `View()` outputs from different model states are being stitched into one frame. Likely path: first render happens before/with default `Height=20` → `closedLimit=50`; a `WindowSizeMsg` then enlarges the terminal and `closedLimit` recomputes to 56; a second render happens but the previous frame's headers aren't replaced. Verify by reproducing under `--debug` and checking app/model.go View() for partial-state composition.

## Minimal Repro

```
cd /home/hans/dev/github/beads-workbench
mise run bwb   # in a terminal taller than ~60 rows
```

## Impact

Cosmetic but glaring — undermines confidence in the UI. Self-resolves on next refresh, but it is the first thing a user sees on launch in a tall terminal.

## Where Discovered

User-reported during release prep on 2026-05-16. Output pasted into the planning conversation; alt-screen is enabled (`cmd/bwb/main.go:53`), so this is not terminal scrollback — bwb is genuinely rendering two header rows.
