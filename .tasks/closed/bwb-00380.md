## Description
Discovered from direct manual verification after closing `beads-workbench-2shr.6`. The previous fix appears to stop the worst ANSI/pane-corruption overflow, but the real detail view for `beads-workbench-2shr` is still not acceptable when the comments section contains long pasted output. The content now stays more bounded, yet the rendered result remains messy and hard to read in practice.

## Expected Behavior
After the overflow fix, the comment-heavy detail view should be readable in a normal terminal. Long pasted output should remain visually contained and also present as clean, comprehensible content rather than a cluttered wall of awkwardly wrapped lines.

## Actual Behavior
In direct runtime verification of `beads-workbench-2shr`, the detail view is still ugly in the comment-heavy area. The worst bleed appears reduced, but the content pane remains cluttered and difficult to scan. Wrapped command/test output still produces a poor operator experience and is not at an acceptable quality bar.

## Reproduction
1. Build the app: `go build -o /tmp/bwb ./cmd/bwb`.
2. Run `env BD_NON_INTERACTIVE=1 /tmp/bwb` from the repo root.
3. Open epic `beads-workbench-2shr` in detail mode.
4. Page down into the historical comments/test-output area.
5. Observe that the content remains difficult to read even though the previous overflow bug was partially mitigated.

## Impact
Acceptance review for `beads-workbench-2shr` should remain blocked. The previous fix addressed part of the technical defect, but the user-visible rendering quality is still not good enough for the real comment-heavy detail workflow.

## Decisions
- Long pasted-output comments may be elided/collapsed inline with a clear marker such as `… (+K lines elided)` when needed to restore readability.
- Comment ordering should switch to newest-first.

## Instructions
1. Reproduce the current state in the built app using the real `beads-workbench-2shr` comments area.
2. Improve comment-heavy detail rendering so pasted command/test output is readable and visually well-structured, not merely width-bounded.
3. Implement inline elision/collapse for very long pasted-output comments when needed to preserve readability, using a clear visible marker.
4. Switch detail comments to newest-first ordering.
5. Add focused regression coverage for the accepted presentation behavior where practical.
6. Re-verify in the built app and confirm the result is actually readable to a human, not just mechanically bounded.
