## Description
Update repo-visible docs so operators can discover the new CLI flag surface and understand which paths are non-interactive versus TUI startup paths.

## Instructions
1. Update top-level docs to list the supported `bwb` flags and explain the intended minimal CLI surface.
2. Document `--config`, `--cwd`, `--print-config`, and `--check-config` with concrete examples.
3. Document version output/build metadata expectations so release builds and local dev builds have understandable `--version` behavior.
4. Document the exit-code table for the non-interactive CLI paths.
5. Document the `--debug` line prefix and the categories of debug events emitted.
6. Update any coding/releasing/agent-facing docs that currently imply `bwb` always boots directly into the TUI with no preflight flag handling.
7. Keep documentation aligned with the final implemented semantics; do not document `--beads-dir` if the implementation ships only `--cwd`.

## Files to Modify
- `README.md`
- `AGENTS.md`
- `docs/OVERVIEW.md`
- `docs/CODING.md`
- `docs/CHANGE-WORKFLOW.md` (if startup/usage guidance needs CLI examples)
- `docs/RELEASING.md` (if version/build metadata behavior needs documenting)

## Acceptance Criteria
- [ ] README or equivalent top-level docs list the shipped flag surface
- [ ] `AGENTS.md` and `docs/OVERVIEW.md` describe the pre-TUI CLI layer and which flag paths exit before Bubble Tea starts
- [ ] Docs include working examples for `--config`, `--cwd`, `--print-config`, and `--check-config`
- [ ] Docs include the exit-code table for the non-interactive CLI paths
- [ ] Docs describe the `[bwb-debug]` line prefix and which events are emitted
- [ ] Docs match the final `--version` behavior and build metadata source
- [ ] No docs claim a `--beads-dir` flag if the implementation does not ship one
