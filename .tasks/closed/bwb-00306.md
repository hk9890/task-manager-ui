opener_test.go only tests the splitEditorCommand helper (2 cases). There is zero test coverage for:

1. The `editorCommand()` fallback chain: explicit command → $EDITOR env → 'vi' default. This is a documented config behavior (CODING.md lines 97-99) with no verification.
2. The `ProcessOpener.Open` method itself — no test verifies it wires command+args+path correctly or handles exec errors.

The splitEditorCommand tests are fine but they're testing the least interesting part. The fallback chain is where config-related bugs would actually hide.

Suggested fix: Add tests for the fallback chain using t.Setenv to control $EDITOR, and a test that exercises Open with a trivial command (e.g., 'true' or '/bin/echo') to verify the path argument is appended correctly. The Opener interface already exists as a seam for the service layer — the opener itself is the boundary that should have its own coverage.
