## Symptom

The shared user-scoped log at `~/.local/state/bwb/bwb.log` (path documented in `docs/MONITORING.md`) violates its own JSON Lines contract when multiple BWB processes write to it concurrently.

Observed in a real log file (~/.local/state/bwb/bwb.log, 3152 lines, 3 concurrent sessions on 2026-05-18):

- Exactly one corrupted line whose JSON prefix (`{"timestamp":"…","level":"INFO","message":"b`) is missing; the line begins mid-record at `,"--sort","closed","--limit","56"],"exit_code":0,"duration_ms":2570}`.
- The argv signature matches a `query issues` call from session `c9d3a32b` (build `dev`, project `beads-workbench`), which was writing concurrently with sessions `f2d9cfcf` (build `0.4.0`, project `dtctl`) and `ba37d264` (build `0.4.0`, project `~/setup`).
- Downstream impact: `jq -c .` aborts at the bad line and silently drops every record after it. Any streaming JSONL consumer will under-report data.

## Root cause (likely, not yet proven)

`internal/logging/logging.go:150-171` builds the persistent sink as a `lumberjack.Logger` per process with no inter-process locking. With three concurrent writers against a shared file, the structural risks are:

- Concurrent `write(2)` calls — should be atomic under `O_APPEND` on a local FS for sub-PIPE_BUF writes, so probably not the primary cause for records under ~1 KB.
- Rotation race — each `lumberjack.Logger` keeps its own size counter, close/rename/reopen sequence, and file handle. Two processes both deciding to rotate (or one rotating while another writes) is the most plausible mechanism for a torn record.

Exact mechanism is worth confirming with a reproducer (two writers + small `MaxSize` to force rotation collisions) before locking in the fix.

## Acceptance

- A torn record like the one in the captured log cannot occur with N>=2 concurrent BWB processes writing the persistent sink, including across a rotation boundary.
- `docs/MONITORING.md`'s JSON Lines guarantee remains accurate (every line is a complete JSON object).
- Existing rotation defaults (10 MB / 5 backups / 30 days / compress) are preserved unless the chosen approach requires changing them.

## Candidate approaches (decide during implementation)

1. Per-process filename: `bwb-<pid>.log` or `bwb-<session_id>.log`. Eliminates the cross-process race entirely. Trade-off: more files in the rotation directory; `tail -f bwb.log` capture command in `docs/MONITORING.md` needs updating.
2. `flock`-based wrapper around the sink's `Write` and rotation. Keeps the single-file contract; needs care to also hold the lock across lumberjack's close/rename/reopen path.

## Evidence

- Captured log: `~/.local/state/bwb/bwb.log` at the time of analysis.
- Rotation history present: `bwb-2026-05-17T09-20-08.674.log.gz`, `bwb-2026-05-17T15-04-49.006.log.gz`, `bwb-2026-05-18T05-20-31.946.log.gz`.
