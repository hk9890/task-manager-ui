## Description

Two related correctness issues in `internal/gateway/beads/read_gateway.go`:

1. **G5 — Exact completeness claim is wrong for capped backends.** `searchIssuePageFromRecords` (`:364-378`) always marks results `Exact`. But `bd ready --json` and `bd blocked --json` themselves have backend-imposed caps. If `bd ready` returns N issues and N equals an internal cap, the gateway falsely claims complete results.

2. **G7 — Empty text + WorkState gives misleading results.** `searchIssuesFromList` (`:214`) early-returns only when text is empty AND `WorkState == WorkStateAny`. Empty text + `WorkStateReady` falls through to `searchIssuesFromReady` and returns the full ready queue marked as a search result. Operator has no signal that "no text filter was applied".

## Files to Modify

- `internal/gateway/beads/read_gateway.go`
- new test: extend `internal/gateway/beads/read_gateway_test.go`

## Instructions

1. Route `searchIssuePageFromRecords` through `searchMetadataFromLimitedBackendResults` (already used elsewhere) so the cap is honored. Add a test simulating a capped upstream response and asserting completeness is reported as `Capped` not `Exact`.
2. Either:
   - Require non-empty text when `WorkState` is set (return a validation error from the gateway), OR
   - Attach a typed notice in the result indicating "no text filter applied; returning entire ready/blocked queue" and have the UI render it.
3. Add tests for both paths.

- [ ] Completeness reporting is accurate for capped backend reads
