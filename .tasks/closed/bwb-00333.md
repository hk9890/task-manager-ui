## Description

`internal/repository/caching/caching.go:437` (`Catalogs`) is TTL-only (default 5 min). `CreateIssue`/`UpdateIssue`/`CloseIssue`/`AddComment` never invalidate `catalogsCache`. When a user creates an issue with a new label `urgent-2026`, the next create-form opening within 5 minutes does not list it — the user re-types, risking typos that fork labels across the project.

The deleted argv-level cache invalidated every entry (including catalogs) on every `IsWrite=true` call, so this regression was introduced by the new CachingRepository.

## Instructions

1. On any successful CreateIssue / UpdateIssue / CloseIssue / AddComment that could add a new label or type — i.e. when `input.Labels` is non-empty in Create/Update — invalidate `catalogsCache` so the next Catalogs() refetches.
2. The cheapest correct invalidation: clear the entire catalogsCache on any successful write. This is what the old argv-level cache did. Document that catalogs are cheap and infrequent enough to justify full invalidation.
3. Keep the TTL as a safety net for EXTERNAL catalog changes.

## Test

Add `TestCreateIssue_InvalidatesCatalogs`:
1. Build CachingRepository over a fake backing whose Catalogs returns `{Labels: ["a","b"]}` initially and `{Labels: ["a","b","new"]}` after a write.
2. Call Catalogs() — expect ["a","b"].
3. Call CreateIssue(ctx, input{Labels:["new"]}).
4. Call Catalogs() — expect ["a","b","new"].
