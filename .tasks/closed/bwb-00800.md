## Description
Implement the standalone app flows for issue creation and quick issue mutations. This task covers create, update, close, and comment operations that should be available directly from the product UI while keeping the ownership boundary with the rich editor flow explicit.

## Instructions
1. Implement the chosen create-issue UX explicitly: either editor-based create documents or a dedicated inline create flow. Do not leave creation ownership ambiguous.
2. Add UI actions for update issue metadata, close issue, and add comment.
3. Route all write operations through the new gateway.
4. Use gateway-provided status/type/label catalogs for UI controls where needed.
5. Keep inline quick actions focused on small mutations while leaving rich editing to the external editor flow.
6. Integrate success/error feedback through the shared app-shell UI primitives.
7. Add automated tests for all supported mutation flows using non-interactive UI/test seams.

## Files to Modify
- `internal/app/` (update command routing if needed)
- `internal/mode/details/` and/or create-related UI packages (create or update quick-action flows)
- `internal/ui/` (update action prompts/components if needed)
- `internal/domain/` (adjust mutation input models if needed)
- mutation-related `*_test.go` files (create or update)

## Acceptance Criteria
- [ ] The create-issue UX is explicitly chosen and implemented
- [ ] Users can create issues from the app
- [ ] Users can update issue metadata from the app
- [ ] Users can close issues from the app
- [ ] Users can add comments from the app
- [ ] UI controls use gateway-provided catalogs where needed
- [ ] All write actions route through the gateway and surface clear feedback
- [ ] Automated tests cover the supported mutation flows without manual UI inspection
