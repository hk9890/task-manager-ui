## Description

Dashboard startup currently fires 7 sequential `bd` subprocesses (~3 s wall time) due to per-section + per-count queries plus a global runner mutex. Empirical investigation shows we can cut this to 3 parallel `bd` calls (~700 ms wall time) by replacing the per-section/per-count machinery with two generic query primitives plus a dashboard-specific composer.

## Problem evidence
- 7 serial bd calls on startup, each ~200–700 ms (see `internal/gateway/beads/runner.go:103-107` for the serialization point and `internal/mode/board/model.go:174-179, 599-624` for the section + count fan-out).
- `bd ready --json` and `bd blocked --json` are issued twice (once for the section, once for an uncapped recount) — pure duplication.
- `Gateway.HealthCheck` runs a full `bd list --json` (~680 ms) when `bd ping --json` (~210 ms) is the documented health probe.
- `bd query --json` and `bd list --json` responses do NOT carry per-issue dependency state (only `dependency_count`), so the dependency-blocked vs ready split must come from `bd ready --explain --json` (single call returns both sets + summary counts).

## Target design
- Add a generic `Query(expr, opts)` gateway method wrapping `bd query --json`. Reusable beyond the dashboard (in_progress, closed, future search).
- Add a `ReadyExplain()` gateway method wrapping `bd ready --explain --json` — single call returning ready + dependency-blocked + counts.
- Build a pure, table-tested dashboard composer that takes the 4 result groups (Ready, Blocked, InProgress, Done) and produces typed `Column` data with consistent sort + per-column caps.
- Rewrite the board data layer to fan out 3 parallel calls (ReadyExplain + Query in_progress + Query closed-recent) and feed the composer.
- Split the runner mutex into `sync.RWMutex` so reads run in parallel (writes still serialized).
- Swap health check to `bd ping --json`.

## Fetch contract (locked by user during planning)
- Closed column: backend sort + limit (DB has 1000+ closed; cap = `max(50, height-3)`).
- Ready, Blocked, In Progress: **no limit on the read side**. Active population is expected to be <100. If any of these groups returns >500 items, the gateway/composer must log a structured warning so the cardinality regression is observable.
- All 3 dashboard reads dispatched in parallel; render gates on completion of all 3.
