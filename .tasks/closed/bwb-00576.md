## Description

Catch the class of bug where a dashboard column displays a capped slice length as the "total" instead of the real DB count. Today the Done column shows 50/56+ on a DB with 452 closed issues; no test catches this because `TestCompose` asserts `Total == len(Closed)` (the bug encoded as correct).

## Instructions

1. Create `internal/dashboard/parity/count_parity_integration_test.go` (build tag `integration`).
2. For each `Dataset` returned by `datasets.AllAvailable(t)`, run the full dashboard fetch pipeline (the same code path bwb uses on startup — `app/services.go` factories or direct gateway calls), capture the resulting `dashboard.Columns`, and compare:
   - `Columns.NotReady.Total` vs `bd count --status blocked --json`
   - `Columns.Ready.Total` vs the count from `bd ready --json | length`
   - `Columns.InProgress.Total` vs `bd count --status in_progress --json`
   - `Columns.Done.Total` vs `bd count --status closed --json` (this is THE failing case)
3. Fail with a clear diff: `column=Done dataset=this_repo bwb_total=56 bd_count=452 delta=-396`.
4. Mark the Done assertion as `t.Run("DoneTotalMatchesBdCount", ...)` so it can be triaged independently.
5. The test should currently FAIL on `this_repo` and `ai_config_manager` — that is the desired state until [[bug-done-total-wrong]] is fixed. Use `t.Skipf` only when env gate is off, NEVER mask the failure.

## Acceptance Criteria

- [ ] Test compiles and runs under `go test -tags=integration ./internal/dashboard/parity/...`
- [ ] On fixture: all 4 column totals match bd output (currently passes by accident)
- [ ] On `this_repo` (env-gated): Done assertion fails with a clear delta message exposing the bug
- [ ] On `ai_config_manager` (env-gated): Done assertion fails with a clear delta message
- [ ] When the bug is fixed (separate task), all assertions pass on all datasets without test changes
