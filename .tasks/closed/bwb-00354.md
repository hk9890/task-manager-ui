## Description

`internal/gateway/beads/runner.go:114` detects the "no beads database" error via `strings.Contains(stderr, "no beads database found")` — a plain substring match against arbitrary `bd` localization/wording. Any future `bd` CLI release that rephrases this stderr silently downgrades the error to `ErrorCodeCommandFailed`, and the UI's first-run "no database here" guidance (`internal/app/model.go:235`) stops firing. The path has zero test coverage today.

Major finding G3 from the 2026-05-16 audit.

## Files to Modify

- `internal/gateway/beads/runner.go`
- new test: extend `internal/gateway/beads/runner_test.go`

## Instructions

1. Investigate stable signals from `bd`:
   - Does `bd` use a dedicated exit code (e.g., 2) for missing-db? Run `bd ready --json` in `/tmp` and inspect exit code + stderr.
   - Is `bd doctor` a stable check? `bd ping --json`?
2. Pick the most stable signal. Order of preference: dedicated exit code > stable error JSON > stable stderr substring as last resort.
3. If `bd` does NOT currently expose a stable signal, file an upstream `bd` request and document the current best-effort detection with a pinning test that fails loudly on rewording.
4. Add a test exercising the missing-db code path: invoke runner with `WorkDir` pointing at a dir with no `.beads/`, assert returned error has code `ErrorCodeNoDatabaseFound`.
