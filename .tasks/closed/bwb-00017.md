## Description

`board.Model.closedLimit()` (`internal/mode/board/model.go:257-263`) imposes a hard `min(50)` floor on the Done column cap regardless of terminal height. The client should ask for exactly what fits on the current viewport — no constant floor. The caching layer is responsible for over-fetch / dataset retention separately; the client should not encode that policy.

## Expected vs Actual

- **Expected**: `closedLimit` returns the section item capacity for the current terminal height. Client asks for what it needs; caching handles dataset retention.
- **Actual**: `closedLimit` returns `max(50, sectionItemCapacity())`. At small terminal heights this over-fetches by ~43 issues; at large heights it has no effect.

## Why the floor is wrong

1. **Client-side policy that belongs in caching**: the UI's job is "ask for what I can display". The decision to keep data warm across resizes is a cache concern.
2. **Blocks honest resize-refetch semantics** (novz): the floor means small-terminal grows below height ~53 silently no-op, hiding bugs and creating an asymmetry between "small-screen" and "large-screen" code paths.
3. **Magic number** with no test pinning the 50 to anything user-visible.

## Fix

- Remove `closedLimit()` entirely
- Update the one call site at `internal/mode/board/model.go:346` to use `m.sectionItemCapacity()` directly
- Update `TestClosedLimit` (`internal/mode/board/model_test.go:631-652`) → either remove (subsumed by `TestSectionItemCapacity`) or rewrite to assert the new behavior
- Update comments in `internal/dashboard/parity/sort_parity_integration_test.go:48` and `scale_parity_integration_test.go:12` that reference the old min-50 floor

## Out of scope

- novz's resize-refetch design (separate in_progress epic; this fix simplifies one of its premises but doesn't replace its work)
- Caching policy for closed-issue retention (handled by the caching layer, future epic)
