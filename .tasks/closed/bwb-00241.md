## Summary

Latent bug: `buildFilterArgs` at `internal/gateway/beads/read_gateway.go:625` joins `query.Statuses` with comma and passes as a single `--status open,in_progress` flag. bd 1.0.4 treats this as a literal status name match and returns `{"groups":[],"total":0}`. Multi-flag `--status open --status in_progress` also broken (only last value wins for bd count).

## Discovered during

`beads-workbench-048i` Read-counts audit (parent epic: `beads-workbench-yspw`).

## Impact

LATENT. The only production caller is `board/model.go:670-672` which always passes a single status (`["closed"]`). No user-visible regression today. The clause `query.Statuses set → only matching statuses counted` in `interface.go:464-465` is partially unsatisfiable for multi-status queries.

## Caveat

`buildFilterArgs` is SHARED with other methods (ListIssues, etc.). bd's `--status` flag semantics differ per subcommand: `bd list --status open,in_progress` works (union), `bd count --status open,in_progress` does not, `bd search --status open,closed` returns empty (INERT — different behavior again). Any fix must audit per-subcommand bd flag semantics before changing the shared helper.

## Recommended fix

Either:
(a) Per-subcommand argv builders that emit the correct flag shape for the target bd verb (cleanest but more code).
(b) Restrict multi-status filtering to subcommands that support CSV (list, query); fall back to in-memory filter for count/search; document constraint.

Add a contract test exercising multi-status CountIssues to pin whichever behavior is chosen.

## Acceptance criteria

- [ ] CountIssues with multi-status query returns correct counts (sum of matching groups).
- [ ] Per-subcommand argv encoding documented in ARGV_CONTRACT.md.
- [ ] Test pins the chosen behavior.
