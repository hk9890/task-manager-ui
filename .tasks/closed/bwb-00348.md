## What

Add model-level regression tests that drive the Update loop with two reload key events in sequence while the first refresh's gateway Cmds are still pending. Tests must be written FIRST (TDD), MUST FAIL against current unfixed code, and MUST PASS after the sibling fix tasks land.

## Why

The bug is silent in normal smoke tests because the gateway in tests returns "fast enough" that double-refresh races don't surface in a single Update pass. Without a regression test driving the message order explicitly — by capturing the in-flight Cmd without draining it before firing the second keypress — this class of bug will return.

## Test approach (simplified — no fake changes needed)

The existing infrastructure is sufficient:

- `internal/app/model_test.go:3028` defines `runBatch(cmd tea.Cmd) []tea.Msg` — recursively expands tea.BatchMsg into individual msgs.
- `internal/app/model_test.go:3054` defines `applyMessages(t, model, msgs)` — runs Update against each msg, recursively expanding returned Cmds.
- The pattern at `internal/app/model_loading_refresh_test.go:117` captures a returned Cmd from `m.Update(msg)` WITHOUT calling `applyMessages(runBatch(cmd))` — leaving the gateway Cmds "in flight" from the model's perspective.

For the board/search package tests, lift `runBatch` / `applyMessages` into a shared test-helpers location (or duplicate locally — pick whichever fits the package's existing conventions; check internal/mode/board/model_test.go patterns).

## Required tests

### Board (internal/mode/board/model_double_refresh_test.go)

1. **`TestBoardManualReloadIgnoredWhileInFlight`**
   - Setup: NewModel with fake gateway; call `m.Init()` and `applyMessages(runBatch(cmd))` to settle cold-start.
   - Snapshot: `selectedRow`, `focusedColumn`, `pendingResults` (should be 0 after settle).
   - Send first reload key event: capture returned Cmd, do NOT drain it.
   - Assert: `m.pendingResults == 4`, all columns `loading == true`.
   - Send second reload key event WITHOUT draining the first Cmd.
   - **Assert (currently FAILS):** `m.pendingResults` is still 4 (not reset).
   - **Assert (currently FAILS):** `selectedRow` and `focusedColumn` from snapshot are preserved (the second handler must NOT have wiped them via the manual-refresh reset at startReload lines 358-362).
   - **Assert (currently FAILS):** The second Update returned `nil` Cmd (no new gateway batch).
   - Drain the first Cmd, verify counter goes to 0 cleanly and never negative.

2. **`TestBoardPendingResultsNeverGoesNegativeUnderKeySpam`**
   - Setup as above.
   - Fire reload key event, capture Cmd, do NOT drain.
   - Fire reload key event 10 more times without draining.
   - Drain the first Cmd's messages individually.
   - **Assert (currently FAILS):** at every step `m.pendingResults >= 0`.
   - **Assert:** after all messages drain, `m.pendingResults == 0` and no `maybeCompose` call ran with partial data (verified via a deterministic gateway response — assert the composed columns match the gateway's single response, not a mix).

3. **`TestBoardInternalStartReloadGuardedByInflightFlag`** (gated on 5q6t.3 landing)
   - Setup as above.
   - Call `m.startReload(refreshModeManual)` directly, capture Cmd, do NOT drain.
   - Call `m.startReload(refreshModeManual)` a second time directly.
   - **Assert:** returned Cmd is nil; `m.pendingResults` is still 4; partial buffers untouched.

### Search (internal/mode/search/model_double_refresh_test.go)

1. **`TestSearchManualReloadIgnoredWhileInFlight`**
   - Setup with fake gateway returning a deterministic SearchResultPage.
   - Settle initial state (typing into query, applying search).
   - Send first reload key event, capture Cmd, do NOT drain.
   - Assert: `m.loading == true`, gateway has 1 query call recorded.
   - Send second reload key event WITHOUT draining the first Cmd.
   - **Assert (currently FAILS):** gateway still has only 1 query call (not 2).
   - **Assert (currently FAILS):** `m.pendingSelectionAnchor` matches the value set by the FIRST keypress (not overwritten).
   - **Assert (currently FAILS):** second Update returned nil Cmd.

### App-level (internal/app/model_double_refresh_app_test.go)

1. **`TestAppRapidMutationsDoNotEnqueueConcurrentRefreshes`** — lock down the existing guard at app/model.go:1076 against silent regression.
   - Setup: app with board active, fake gateway.
   - Fire mutation messages (e.g., issueClosedMsg) in rapid succession, each triggering `maybeAutoRefreshActiveSurfaceCmd()`.
   - Capture the Cmds at each step.
   - **Assert:** at most one refresh batch is ever in flight; `boardIsLoading()` guard fires when expected.
   - This test should PASS on current code — its purpose is to prevent future regression of the existing guard, not to expose a current bug.
