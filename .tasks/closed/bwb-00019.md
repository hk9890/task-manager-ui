Zero-behavior refactor that introduces `repository.DashboardOptions` with `ClosedLimit int` and `ForceFresh bool` fields, and updates all 16+ `Dashboard(ctx)` call sites to pass `DashboardOptions{}`. No user-visible behavior change.

This epic exists to unblock `beads-workbench-iwvm` (variable closed-fetch limit) and `beads-workbench-fbea` (caching layer correctness + force-fresh reload), which both need the same interface change. Landing it standalone lets the two behavior-change epics ship independently without forcing one into a noisy rebase across all call sites.

## Why a separate epic

Both iwvm and fbea need to add a field to the `Dashboard` signature. If iwvm ships first, fbea must rebase its 16+ call-site edits onto iwvm's. If fbea ships first, iwvm does. Either way, whichever lands second eats a noisy merge across the same files for no behavioral reason. A zero-behavior interface PR is the cleanest split.

## Interface change

```go
// internal/repository/repository.go
type DashboardOptions struct {
    ClosedLimit int  // 0 = use implementation default
    ForceFresh  bool // bypass any cache and force a backing call
}

type Repository interface {
    Dashboard(ctx context.Context, opts DashboardOptions) (DashboardData, error)
    // …other methods unchanged…
}
```

This epic implements ONLY the signature change. The two new fields are accepted but **not honored** by any implementation:
- `beads.Repository`: ignores `opts.ClosedLimit` (keeps hardcoded `defaultLeanClosedLimit`); ignores `opts.ForceFresh` (no cache).
- `memory.Repository`: ignores `opts.ClosedLimit`; ignores `opts.ForceFresh`.
- `caching.Repository`: ignores both. (The fbea epic teaches it to honor `ForceFresh`; iwvm teaches it to honor `ClosedLimit`.)
- `validating.Repository`, `errorinjecting.Repository`: pass through.

Full enumeration is the first subtask.

## Success Criteria

1. `repository.DashboardOptions` struct exists with `ClosedLimit int` and `ForceFresh bool` fields.
2. `Repository.Dashboard` signature updated to take `DashboardOptions`.
3. All call sites in production and test code updated to pass `DashboardOptions{}` (or an explicit value where the caller already has one — e.g. the board model may pass `m.closedLimit()` through here, that's fine).
4. **No behavior change:** full `mise run quality` passes WITHOUT any other changes. Crucially: `TestRepositoryContract` (`internal/repository/contract/parity_integration_test.go`) still passes for both memory and beads impls.
5. Doc updated in `internal/repository/repository.go` interface comment to document the new options.

## Out of scope

- Honoring `opts.ClosedLimit` (tracked in iwvm).
- Honoring `opts.ForceFresh` and the seeds-into-memory fix (tracked in fbea).
- Any other interface changes (e.g. options structs for Issue, Search).

## Risk

Touching all call sites in one PR makes it large but mechanical. The parity contract suite is the safety net — if it passes, the refactor is correct.

## User-facing workaround (until this ships)

Not applicable: this is a zero-behavior refactor. Workarounds for the actual bugs are in iwvm and fbea.
