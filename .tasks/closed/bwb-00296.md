## Where discovered

`internal/mode/board/model.go` during the [[xm6u]] fix (duplicate `component` JSON key).

## Expected

Debug/Warn records from the board model carry `component=board` so log queries can distinguish board-layer events from dashboard-layer events.

## Actual

All board model records carry `component=dashboard` (inherited from the parent logger set in `cmd/bwb/main.go:46`: `services.Logger = opts.logManager.Component("dashboard")`).

The now-removed inline `"component", "board"` overrides at `internal/mode/board/model.go:243` and `:359` were presumably an attempt to correct this at each call site, but slog's JSON handler appended both keys rather than overwriting — producing the [[xm6u]] duplicate-key bug instead of a fix.

## Reproduction

Run bwb, open board, trigger a refresh; `grep '"component":"dashboard"' ~/.local/state/bwb/bwb.log` shows board-layer records labeled `dashboard`.

## Impact

- Log queries filtering by `component` cannot distinguish board model events from dashboard-component events.
- Low operational severity (no functional bug), but confusing during debugging — exactly the confusion that motivated the original (broken) inline override.

## Proposed fix

In `NewModel` (or at construction in `cmd/bwb/main.go`), derive the board model logger with `logger.With("component", "board")` — or pass a freshly-derived `logManager.Component("board")` logger — so board-layer records self-identify correctly.

Also audit other mode models for the same pattern (details, search likely affected).

## Acceptance Criteria

- [ ] Board model log records carry `component=board` (verifiable via `bwb.log` after a refresh).
- [ ] Other mode models audited and similarly corrected if affected.
- [ ] Regression test capturing slog output asserts the expected `component` value per mode.
