## Description
Verifier review of beads-workbench-ekp found that board and search still assemble compact issue rows through separate code paths.

## Evidence
- internal/ui/board/board.go:202-235 still defines renderRowLine(Row, maxWidth)
- internal/ui/search/search.go:161-183 still assembles prefix/meta/title inline inside renderResultsContent
- internal/ui/shared/ contains only renderhelpers/helpers.go; there is no shared row component
- board still uses local uiboard.Row input while search renders domain.IssueSummary directly
- id width logic still diverges: board caps with minInt(12, MaxInt(7, maxWidth/5)) while search uses MaxInt(7, width/5) with no cap

## Why
This fails the epic expectation that board and search share one compact row renderer implementation path, leaves row-level duplication in place, and keeps the input/id-width decisions implicit instead of explicit.

## Acceptance Criteria
- [ ] Board and search call the same shared compact row renderer for selection indicator, type/priority/status/ID token assembly, and width-aware title truncation.
- [ ] Input type choice is explicit and coherent (reuse domain.IssueSummary or document a dedicated shared input type) and redundant board-only row structs are removed or justified.
- [ ] ID width behavior is unified or any intentional divergence is documented as a renderer parameter.
- [ ] Tests prove both call sites use the shared path and preserve deterministic truncation.
