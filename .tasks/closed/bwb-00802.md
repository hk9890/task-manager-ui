## Description
Implement the rich external-editor issue editing workflow for Beads Workbench by selectively adapting donor helpers from Perles. This task should preserve the editor-friendly product model without carrying over legacy architecture.

## Instructions
1. Inspect these donor paths:
   - `/home/hans/dev/github/perles/internal/ui/shared/vimtextarea/external_editor.go`
   - `/home/hans/dev/github/perles/internal/ui/shared/editor/issue_markdown.go`
2. Copy or adapt only the external-editor and issue-document helpers that fit the new standalone architecture.
3. Define the issue document format before implementation, including editable fields, protected/read-only fields, and how round-trip parsing maps back to gateway update capabilities.
4. Implement the temp-file editor flow: load issue details, render editable issue document, launch `$EDITOR`, parse the saved result, compute changes, and apply them through the gateway.
5. Put editor launching behind a replaceable seam so tests never need to spawn a real interactive editor.
6. Remove any assumptions tied to old Perles UI/service wiring.
7. Own the rich create/edit document flow if issue creation uses the same editor-based document model; otherwise document the boundary with the create/update task explicitly.
