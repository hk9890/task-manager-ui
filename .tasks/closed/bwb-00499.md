## Description

Verify that the `CountIssues` method on `BeadsGateway` is in sync with real bd 1.0.4 behavior, the FakeBeadsGateway implementation, and the existing test coverage.

Part of epic `beads-workbench-yspw`.

## Methods in scope

- `BeadsGateway.CountIssues(ctx, query) (IssueCountResult, error)` — `internal/gateway/beads/interface.go:477`

## Argv shapes in scope (from ARGV_CONTRACT.md)

- `bd count --by-status --json`
- `bd count --by-status --json --status <csv>`

## Contract restated (key clauses — see interface.go for full text)

**Preconditions:**
- ctx must be non-nil.

**Postconditions:**
- `result.Total` = sum of all group counts in `result.Groups`.
- `result.Groups` contains ONLY status groups with non-zero count (bd omits zero-count groups).
- ALL statuses (including closed) included by default — NO `--all` flag required (unlike `bd list`).
- When `query.Statuses` set, only matching statuses counted; result contains only those groups.
- `Groups[i].Status` values are raw bd names (open, in_progress, blocked, closed, ...).

**Error semantics:**
- `ErrorCodeCommandFailed` for non-zero bd exit.
- `ErrorCodeDecodeFailed` for unparseable count JSON object.

**bd quirks observed:**
- 2026-05-16: `bd count --by-status --json` includes all statuses (incl. closed) by default — no `--all` needed. Disposition: ACCEPT, documented as postcondition.
