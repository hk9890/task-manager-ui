## Description

The composer logs `warn: backend sort assumption broken ... group=Closed count=N threshold=-1` on real datasets even when bd is returning correctly ordered results. The check at `internal/dashboard/composer.go:121-132` uses `UpdatedAt` as a proxy for `closed_at` because `IssueSummary` does not carry `closed_at`. When many closed issues share an `updated_at` (bulk close, migration, scripted updates), the proxy fires a false positive.

## Expected vs Actual

- **Expected**: The "backend sort assumption broken" warning fires only when bd's `closed_at desc` ordering is actually violated.
- **Actual**: It fires whenever the first two returned closed issues have equal or non-descending `updated_at`, even though `closed_at` ordering is correct.

## Root Cause

`IssueSummary` lacks a `ClosedAt` field, so the composer uses `UpdatedAt` as a proxy. This is documented at composer.go:113-120 but the assumption breaks on real data.

## Suggested Fixes (pick one)

1. **Plumb `ClosedAt` through** — add `ClosedAt time.Time` to `IssueSummary`; map it from the gateway; use it in the proxy. Honest fix.
2. **Remove the defensive check** — it has produced more noise than signal; the gateway test already pins the sort flag we send to bd, and the parity test [[izds]] verifies Done order against bd on real data. Cheap fix.
3. **Tighten the proxy** — only warn if `len(in.Closed) >= 10` AND first 10 items are all equal/non-descending. Still proxy-based but quieter.

Recommendation: option 1 if `closed_at` is cheap to surface, otherwise option 2 (the parity test makes the defensive check redundant).
