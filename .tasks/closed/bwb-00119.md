## Summary

`FakeBeadsGateway.SearchIssues` at `internal/testing/fakes/beads_gateway.go:334` returns `SearchResultPage{Results: nil}` when `SearchResultsByText` is non-nil but the text key is absent from the map. The real gateway always returns a non-nil empty slice via `make([]SearchResult, 0, ...)` in `toSearchResults`. The interface contract postcondition states `result.Results is never nil; it may be empty`.

## Discovered during

`beads-workbench-vhp2` Read-search audit (parent epic: `beads-workbench-yspw`).

## Why the existing invariant test misses this

The `RunReadContract/SearchIssues/ResultsNeverNil` invariant test uses `SearchIssuesQuery{}` (empty text → cache key `""`). The fake test fixture seeds `SearchResultsByText[""]` so the lookup hits and returns the seeded non-nil slice. Any query with a text value not in the seeded map triggers the buggy nil-Results path, which the invariant test does not exercise.

## Fix

`internal/testing/fakes/beads_gateway.go:334` — change `Results: nil` to `Results: []domain.SearchResult{}`.

## Test coverage

Add a `ResultsNeverNil` test case (or extend the existing one) that uses `SearchResultsByText` configured with a key that is NOT queried, then queries with a text value not in the map, and asserts `len(result.Results) == 0 && result.Results != nil`.
