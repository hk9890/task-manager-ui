## Description

The Done column shows "50" or "56+" on a DB with 452 closed issues. Reproduces on this repo (`bd count --status closed --json` returns 452, bwb shows 50/56+) and on `/home/hans/dev/github/ai-config-manager` (622 closed → bwb shows capped value).

## Expected vs Actual

- **Expected**: Done column total displays the real DB count of closed issues (e.g. 452), independent of the page cap. The "+" badge is fine when the rendered list is capped, but the number itself must be the population total.
- **Actual**: Done column total = `len(returned_slice)`, which is min(real_count, ClosedLimit). Display reads "50" or "N+" where N is the cap.

## Root Cause

`internal/dashboard/composer.go:163-174` — `done.Total = len(closedIssues)`. The dashboard never calls `bd count` for closed; only `bd list --status closed --limit N`. The gateway has `CountIssues` (`internal/gateway/beads/read_gateway.go`) but no dashboard caller uses it.

## Minimal Repro

```
cd /home/hans/dev/github/beads-workbench
bd count --status closed --json    # → 452
mise run bwb                        # Done column shows 50 then 56+
```

## Impact

- Shipping with this means every release shows the wrong headline number for the most populated column.
- Users will conclude bwb's data is unreliable and stop trusting any number it shows.

## Suggested Fix

1. Add a 4th gateway call to the dashboard fetch plan: `CountIssues(status=closed)`.
2. Extend `dashboard.Inputs` with `ClosedTotal int`.
3. In `composer.Compose`, set `done.Total = in.ClosedTotal` (or `len(closedIssues)` when `ClosedTotal == 0`/unset, as a fallback). `TotalIsExact` becomes `len(closedIssues) >= ClosedTotal` — false only when the cap actually clips the visible list.
4. Update `TestCompose` to cover the new field.

## Where Discovered

While diagnosing user-reported "Done shows 50 but bd shows 452" during release prep on 2026-05-16. Confirmed by reading `internal/dashboard/composer.go` and `internal/dashboard/builtin_provider.go`.
