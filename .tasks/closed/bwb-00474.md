What I did: read internal/app/model.go key routing (lines 582-802), then verified at runtime with /tmp/bwb against the embedded fixture.

Mechanism: For board (and search) mode, the active mode's Update runs first (producing modeCmd), and THEN the shell action 'switch' (lines 669-798) also runs. Detail mode early-returns when HandleKey reports 'consumed' (line 642-644) and search early-returns on CapturesShellKey (589-591), but BOARD mode has no such consumed/early-return guard. So in board mode a single key is handled by board.Update AND the shell switch.

With default keybindings this is masked: the only board/shell key overlaps are 'l' and 'r', and both shell actions on those keys (launch_shell_command, reload_detail) are mode-guarded with 'if m.active != mode.Detail { return }'. But the shell actions edit_issue/create_issue/update_issue/close_issue/comment_issue/toggle_help are NOT mode-guarded.

Expected: rebinding a shell action onto a key a browse mode already uses should either be rejected by config validation, or only one handler should act.

Actual: config 'keybindings: {shell: {create_issue: [j]}}' passes '--check-config' with 'config OK' (exit 0) — the per-context conflict detector only checks within one context, never across shell-vs-board/search. At runtime in board mode, pressing 'j' once both moves the board selection (board move_down) AND opens the Create Issue modal (shell create_issue). Verified via scripts/capture_bwb_screen.py: a single 'j' keypress opened the Create Issue modal overlay.

Severity: medium. Latent footgun for any keybinding customization; '--check-config' gives a false 'config OK'. Default config is safe only by coincidence (the two overlapping shell actions happen to be mode-guarded).

Repro:
  printf 'keybindings:\n  shell:\n    create_issue: [j]\n' > /tmp/c.yaml
  /tmp/bwb --config /tmp/c.yaml --check-config   # 'config OK'
  # run /tmp/bwb --config /tmp/c.yaml against a fixture, press j in board mode -> create modal opens

Fix options: (a) give board mode the same 'consumed'-style early return so a key handled by the active browse mode is not re-dispatched to shell actions; (b) extend config validation to detect cross-context collisions between shell actions and board/search navigation keys and warn or fail; (c) document the precedence explicitly if double-handling is intentional (it does not appear to be).
