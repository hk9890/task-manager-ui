## Description

`internal/repository/caching/caching.go:482` (`CreateIssue`) eagerly seeds `c.memory` from input-derived guesses: Status="open", Type default, Priority default, zero CreatedAt, no Description hydration in Summary. The backing `bd` may assign different defaults (project policy priority, server-stamped CreatedAt, normalized title). Subsequent `Issue(newID)` cache-hit returns the fabricated detail rather than backing truth — until the user edits the issue (which triggers Forget) the read-only view shows wrong values.

## Instructions

Two acceptable approaches:

A. **Skip the seed entirely**: after `backing.CreateIssue` returns, don't seed memory; let the first `Issue(newID)` call miss the cache and fetch real backing detail. Simplest, no fabrication, but pays one backing round-trip on first detail view.

B. **Refetch and seed authoritatively**: after `backing.CreateIssue`, call `backing.Issue(newID)` and `SeedDetail` with the real result. Adds latency to CreateIssue but eliminates the fabrication.

Pick (A) unless there's evidence that the post-create detail view is frequently fetched and the round-trip is costly. Document the decision in the Create method's godoc.

Also: mark `dashboardDirty=true` (this already happens — verify it).
