## Summary

`TypeCatalog()` returns `ErrorCodeDecodeFailed` in any workspace where `bd config set types.custom <value>` has been used. Real bd 1.0.4 returns `custom_types` as a JSON array of bare strings (`["widget","gadget"]`), but the gateway payload `bdTypeCatalogPayload.CustomTypes []bdCatalogEntryPayload` expects objects with `{name, description}`.

## Discovered during

`beads-workbench-r1m3` Read-catalogs audit (parent epic: `beads-workbench-yspw`).

## Reproduction

```bash
TMPDIR=$(mktemp -d) && cd "$TMPDIR" && BD_NON_INTERACTIVE=1 bd init
bd config set types.custom "widget,gadget"
bd types --json
# Output includes: "custom_types": ["gadget", "widget"]
```

Then decoding the payload via the gateway path fails with:
```
cannot unmarshal string into Go struct field bdTypeCatalogPayload.custom_types of type beads.bdCatalogEntryPayload
```

Fresh/default workspaces are unaffected (the `custom_types` key is omitted entirely when no custom types exist, and Go's nil-slice default handles that case).

## Impact

Any user who runs `bd config set types.custom <...>` on their project will see `TypeCatalog()` fail. This blocks the create/update mutation catalog load path in the UI. Currently no known user has this configured, but the bug would surface immediately for anyone who does.

## Fix options

Option A (preferred): change `CustomTypes` field type to `[]string` and adapt `toTypeOption` to construct `TypeOption{Name: s}` for each string entry (no description available for custom types).

Option B: add a custom `UnmarshalJSON` on `bdTypeCatalogPayload.CustomTypes` that accepts both shapes.

Update `internal/gateway/beads/interface.go:706-708` quirk note to reflect the actual bd 1.0.4 shape.
