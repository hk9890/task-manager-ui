Discovered during verification of beads-workbench-59z. Expected: in detail-mode Dependencies, Blocked by / Blocks / Related rows visibly use the shared selected-row treatment, up/down moves that visible selection, and Enter opens the selected dependency issue. Actual: the live app only derives dependency-pane selection from BrowserItems/Structure rows. On an issue with normal dependency refs but no parent-group structure (for example the embedded fixture issue bwf-2), rows render compact metadata but no dependency row is visibly selected, down does not move a visible selection, and Enter does not open a dependency.

## Root Cause
`BrowserItems` is only populated from `ParentGroupBrowser`. When an issue has BlockedBy/Blocks/Related refs but no parent-group structure, the model leaves the selectable dependency list empty, so the render fix from beads-workbench-59z has nothing meaningful to select.

## Acceptance Criteria
- [ ] When an issue has BlockedBy, Blocks, or Related refs but no parent-group structure, the detail-mode selectable dependency list is populated from those refs.
- [ ] The default dependency selection is initialized predictably for those refs.
- [ ] Up/down in Dependencies visibly moves selection across blocked/blocks/related rows in the real app.
- [ ] Enter on the selected dependency opens the selected issue in the real app.
- [ ] The fix preserves existing parent-group/structure browsing behavior when that context is present.
