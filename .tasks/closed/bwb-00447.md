## Description
`docs/TESTING.md` defines: *"if the test does not fork a real subprocess and costs <100ms, it is a unit test. If it forks bd, it is integration."*

Two test files violate this contract on every `mise run test` / `mise run quality:fast` invocation:

- `internal/launcher/process_runner_test.go:18` — `runner.Run(ctx, "sleep", []string{"0.01"}, ...)` forks `sleep`
- `internal/launcher/process_runner_test.go:37` — forks `sh` to write probe files
- `cmd/bwb/architecture_guardrails_test.go:63` — forks `go list -m`
- `cmd/bwb/architecture_guardrails_test.go:84` — forks `go list -deps -json ./cmd/bwb` (0.13s — the largest single non-edit-flow unit test)

A correctly-tagged peer file exists at `internal/launcher/process_runner_integration_test.go` (`//go:build integration`), so the convention is established.

**Plan-review note (Q7):** `TestArchitectureGuardrails` is the ONLY check pinning the bql-import, direct-SQL, and orchestration-package rules. Moving it to the integration tier would silently drop those guardrails from `quality:fast`. Instead, **refactor `TestArchitectureGuardrails` to use `golang.org/x/tools/go/packages` in-process** so it stays in the unit tier and continues to run on every commit. Do NOT relocate the architecture guardrails to integration. Only the `process_runner_test.go` forking tests are relocated.

## Instructions
1. Read both source test files in full and enumerate every test function that shells out.
2. **For `internal/launcher/process_runner_test.go`** (pure relocation):
   - Move all forking test functions into `internal/launcher/process_runner_integration_test.go` (file already has `//go:build integration`)
   - Add any imports the moved functions need
   - Delete the moved functions from the source file
   - Helpers stay in the source file unless they are used ONLY by forking tests
3. **For `cmd/bwb/architecture_guardrails_test.go`** (refactor, do NOT relocate — plan-review Q7):
   - Replace `exec.Command("go", "list", "-m", ...)` with a programmatic load via `golang.org/x/tools/go/packages` (use `packages.Load` with `packages.NeedModule`). Adding the dep to `go.mod` is acceptable; `go/packages` is the canonical Go-tooling library for this.
   - Replace `exec.Command("go", "list", "-deps", "-json", "./cmd/bwb")` with `packages.Load` using `packages.NeedDeps|packages.NeedImports|packages.NeedName` and walk the resulting `*packages.Package` graph in-process.
   - The three sub-tests (`forbids internal/bql in standalone app`, `forbids direct SQL`, `forbids orchestration package`) keep their existing failure messages and structure.
   - Each individual test should run in <100ms (target — the current 0.13s is from the subprocess overhead, not the work). If the in-process load still exceeds 100ms after refactor, document the new ceiling but the test must still be tagged unit (no `//go:build integration`).
   - Do NOT relocate `cmd/bwb/architecture_guardrails_test.go` to integration. There is no `cmd/bwb/architecture_guardrails_integration_test.go` file to create.
4. Do not change test logic for `process_runner_test.go` — pure relocation only. The architecture-guardrails test changes are a true refactor (different mechanism, same observable contract).
5. Verify the changes:
   ```
   go vet ./...
   mise run test ./internal/launcher/... ./cmd/bwb/...     # should NOT run moved process_runner functions; SHOULD run refactored architecture guardrails
   mise run test:integration ./internal/launcher/...        # should run the moved process_runner forking tests
   mise run quality:fast    # total wall time should not increase materially
   ```
6. Verify `mise run test -v ./cmd/bwb/...` STILL shows the three architecture-guardrails sub-tests passing (they were not moved, only refactored).
