## Description
Deliver the first useful read path for the future UI by implementing typed read operations on top of official `bd` commands only. This establishes the production data-loading seam without SQL or BQL.

## Instructions
1. Implement `ListIssues`, `ReadyIssues`, `BlockedIssues`, `ShowIssue`, and `SearchIssues` on the CLI-backed gateway.
2. Implement catalog reads needed by the UI and write flows, including status, type, and label catalogs.
3. Use the shared command runner and decoding helpers from the prior task.
4. Map JSON output from official commands into the domain models defined for Beads Workbench.
5. Use the official command surface available in this environment, such as `bd list --json`, `bd ready --json`, `bd blocked --json`, `bd show --json`, and any official catalog/search commands exposed by `bd`.
6. Normalize command and decode failures into the gateway error model.
7. Do not introduce any direct SQL reads, schema inspection, or BQL execution.

## Files to Modify
- `internal/gateway/beads/` (implement read operations)
- `internal/domain/` (adjust models only if needed)
- `*_test.go` under gateway packages (create/update tests)

## Acceptance Criteria
- [ ] `ListIssues` returns typed issue summaries
- [ ] `ReadyIssues` returns typed ready-work models
- [ ] `BlockedIssues` returns typed blocked-work models
- [ ] `ShowIssue` returns a typed issue detail model
- [ ] `SearchIssues` returns typed search results through the gateway
- [ ] Status, type, and label catalogs are available through the gateway
