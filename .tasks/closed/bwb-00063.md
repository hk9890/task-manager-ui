## Description
Release reconciliation found that this repository has no `.github/workflows` directory and no GitHub Actions runs, which means releases have no automated CI provenance. That is a process gap: future releases depend entirely on local manual verification and there is no repository-native signal showing that build, vet, and test passed on the candidate commit.

## Evidence
- Release reconciliation task `beads-workbench-j8m.1` confirmed local quality gates passed.
- The same task also confirmed there are no GitHub Actions workflows and no recorded runs.
- `docs/CODING.md` defines the expected quality gates, but the repository does not automate them.
- `go.mod` currently declares Go `1.24.2`, so CI should use that toolchain series unless the repository deliberately chooses a different supported policy.

## Instructions
1. Add a minimal GitHub Actions CI workflow under `.github/workflows/` for this Go repository.
2. Trigger it on at least `push` and `pull_request`.
3. Configure the workflow to use the Go version from `go.mod` (`1.24.2`, or the corresponding `1.24.x` setup-go version syntax if required by the action).
4. Have the workflow run the repository's baseline quality gates:
   - `go build ./cmd/bwb`
   - `go vet ./...`
   - `go test ./...`
5. Name the workflow clearly so future release verification can reference it.
6. Note in the implementation or issue comment that this repository is currently private, so GitHub Actions minute usage should be considered; this is not a blocker to adding CI, but it should not be ignored.
7. If `docs/RELEASING.md` exists by the time this lands, update it to reference the CI workflow as part of release provenance; otherwise note the expected doc follow-up in the issue comment.
8. If workflow setup reveals repository-specific constraints, document them in the issue and create follow-up work rather than silently narrowing scope.

## Files to Modify
- `.github/workflows/ci.yml` (create)
- `docs/RELEASING.md` (update if present; otherwise record follow-up in issue comment)

## Acceptance Criteria
- [ ] A GitHub Actions workflow exists under `.github/workflows/`

- [ ] The workflow uses the Go version policy derived from `go.mod`
- [ ] The workflow executes `go build ./cmd/bwb`, `go vet ./...`, and `go test ./...`
- [ ] The workflow is named/documented clearly enough to use as release provenance
- [ ] Private-repo GitHub Actions usage is acknowledged in the implementation notes or issue comment
- [ ] Any repository constraint discovered during setup is tracked explicitly rather than hidden
