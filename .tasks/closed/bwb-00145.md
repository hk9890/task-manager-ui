## Description
Add a generic `Query` method on `BeadsGateway` that wraps `bd query "<expr>" --json` so callers can express filter logic with the bd query DSL (`status=X`, `AND`, `OR`, `NOT`, relative dates, etc.) instead of forcing the gateway to grow a method per use case. Required by the dashboard rewrite (epic beads-workbench-au54) for the in-progress and closed columns, and reusable beyond it.

## Verified payload shape (do not skip)
`bd query "<expr>" --json` returns the SAME field set as `bd list --json` — confirmed by spot check: keys are `id, title, status, priority, issue_type, owner, created_at, updated_at, closed_at, close_reason, description, dependency_count, dependent_count, comment_count, created_by`. **Critically: `blocked_by` and `dependencies[]` are absent** (this is why the epic uses a separate `ReadyExplain()` for the ready/blocked split — do NOT try to derive dependency state from `Query` results).

## Pre-flight verification (step 0, before coding)
Spot-check that `bd query` actually accepts the argv we plan to assemble. Run from the project root:
- `bd query "status=in_progress" --json --sort updated --reverse -n 5`
- `bd query "status=closed" --json -a --sort closed --reverse -n 5`
If either rejects `--sort`/`--reverse`/`-r` (the assumption was carried over from `bd list`, not verified for `bd query` specifically), update this ticket: either narrow `Query` to filter-only and add a separate follow-up for sort/reverse, or drop the flags.

## Instructions
1. In `internal/gateway/beads/interface.go`, extend `BeadsGateway` with:
   - `Query(ctx context.Context, expr string, opts domain.QueryOptions) ([]domain.IssueSummary, error)`
2. In `internal/domain/queries.go`, add `QueryOptions` with:
   - `Limit int` (0 = uncapped; bd's `-n N`)
   - `Offset int` (clients paginate via the existing `withOffsetWindow` pattern in `read_gateway.go:744-754`; do NOT skip Offset — Query is meant to be reusable for paginated callers, not page-1-only)
   - `IncludeClosed bool` (maps to `-a`)
   - `SortBy domain.SortField` (reuse existing enum; map via `mapListSortField`)
   - `SortOrder domain.SortDirection` (descending → `-r`)
3. In `internal/gateway/beads/read_gateway.go`, implement `Query` against `g.runner`:
   - argv: `["query", expr, "--json"]` + `-a` if `IncludeClosed` + `--sort <field>` if mapped + `-r` if descending + `--limit N` if `withOffsetWindow(Limit, Offset) > 0`.
   - Reuse `decodeIssueArray` and `mapIssueSummaries` (no new payload struct needed; the response shape matches the existing `bdIssuePayload`).
   - Reject empty/whitespace `expr` with `domain.ErrorCodeValidationFailed`.
4. Add a new operation constant `operationQuery = "query issues"` next to the existing ones in `read_gateway.go`.
5. Make the new method behave gracefully on bd CLI errors — reuse the existing `normalizeExecutionError` path; no new error code shapes.
