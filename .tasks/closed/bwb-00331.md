## Description

`internal/repository/memory/forget.go:33` (SeedDetail) and `memory/repository.go` (toDetailLocked) do not store or project the `Creator` field of `domain.IssueDetail`. CachingRepository serves cache hits via `memory.Issue`, so every Issue cache hit returns `Creator=""` — even though the backing `bd` Repository populates it on the first call. The Creator line in the metadata pane disappears on every second view of any issue.

## Instructions

1. Add a `creator string` field to the in-memory `storedIssue` struct.
2. Populate it in SeedDetail from `detail.Creator`.
3. Project it back in toDetailLocked by setting `Creator: si.creator` on the returned IssueDetail.

## Files to Modify

- `internal/repository/memory/forget.go` (SeedDetail and storedIssue if defined here)
- `internal/repository/memory/repository.go` (storedIssue, toDetailLocked)
- `internal/repository/memory/repository_test.go` (new test)

## Test

Add `TestSeedDetail_PreservesCreator`:
1. Seed memory with an IssueDetail having Creator="alice".
2. Call Issue(id).
3. Assert Creator=="alice".

Or, more robustly, extend an existing round-trip detail test to assert Creator equality.
