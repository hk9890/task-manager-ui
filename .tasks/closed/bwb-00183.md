## Description
`v0.1.0` exists as a GitHub release, but the repository itself is private, so the release is not publicly accessible. Before anyone tries to "fix" this, the project owner needs to decide whether public release visibility is actually desired for this repository.

## Why This Needs Discussion
This is not an implementation-only task. The correct outcome depends on product/repository policy:
- the repo may intentionally remain private
- releases may need to move to a different public repository
- the current private release may be acceptable for internal distribution only

An agent should not guess which policy is correct.

## Open Questions
- [ ] Should `beads-workbench` be publicly accessible, or remain private?
- [ ] If public release visibility is desired, should that happen by making this repo public or by publishing from a different public location?
- [ ] If private visibility is intentional, where should that be documented for future release operators?

## Instructions
Blocked pending maintainer decision. Once the policy decision is made:
1. Record the chosen visibility strategy in the issue.
2. If a repository/settings change is required, create a concrete implementation task for that work.
3. If documentation changes are sufficient, update `README.md` and/or `docs/RELEASING.md` to reflect the intended release visibility model.
4. Remove the `needs:discussion` label and reopen the task only when the policy choice is explicit.

## Files to Modify
- `README.md` (possible, depending on decision)
- `docs/RELEASING.md` (possible, depending on decision)
- repository visibility/settings outside the repo (possible, depending on decision)

## Acceptance Criteria
- [ ] A maintainer decision exists for private vs public release visibility
- [ ] The chosen strategy is recorded in the issue
- [ ] Concrete follow-up implementation work is created if settings or repo moves are required
- [ ] Documentation is updated if the intended visibility model needs to be explained to future operators
