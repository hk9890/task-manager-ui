Discovered from live user review after the metadata selection styling polish shipped. Expected: metadata rows should keep their value columns aligned whether `Priority` or `Status` is selected. Actual: on closed issue detail views (example: `beads-workbench-omd.1`), the selected-row prefix still causes the selected metadata row to indent differently from sibling rows, e.g. `Priority: P1` vs `› Status  : closed`, so the value column does not line up cleanly. Repro: open `beads-workbench-omd.1` in detail mode, focus Metadata, compare alignment of `Priority` and `Status` rows as selection moves. Impact: metadata pane feels visually sloppy even though shared selection styling is mostly in place.

## Acceptance Criteria
- [ ] Selected and non-selected editable metadata rows preserve the same label/value column alignment.
- [ ] Moving selection between `Priority` and `Status` does not visually shift indentation or the value column.
- [ ] Real-app review on a closed issue detail confirms clean alignment.
