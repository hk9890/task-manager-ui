## Description
`internal/domain/errors.go` (line 28-42) — `GatewayError.Error()` never includes the underlying `Cause` message in its string output. The struct has fields `Code`, `Operation`, `Message`, and `Cause`. The current `Error()` method has 4 branches based on empty Message/Operation but never appends the Cause in any of them. When a user sees a gateway error, they get the message but never the root cause detail.

## Instructions
1. Open `internal/domain/errors.go`
2. The current `Error()` method (line 28-42) has this logic:
   - Empty Message + empty Operation → returns `string(e.Code)`
   - Empty Message + non-empty Operation → returns `"operation: code"`
   - Non-empty Message + empty Operation → returns `e.Message`
   - Non-empty Message + non-empty Operation → returns `"operation: message"`
3. For each of the 4 branches, append `: cause.Error()` when `e.Cause != nil`. For example, the last branch becomes:
   ```go
   s := fmt.Sprintf("%s: %s", e.Operation, e.Message)
   if e.Cause != nil {
       return fmt.Sprintf("%s: %s", s, e.Cause.Error())
   }
   return s
   ```
4. A cleaner approach: build the base string first, then conditionally append Cause at the end (one append point instead of 4)
5. Update any tests in `internal/domain/` that assert on exact error string format
6. Check downstream tests — error string changes could affect assertions in gateway tests
