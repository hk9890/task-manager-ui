## Description

`internal/repository/beads/lean_reads.go:620` (`leanSearchMetadataFromBackend`):

```
if requestedLimit <= 0 || returnedCount < requestedLimit {
    metadata.Completeness = domain.SearchResultCompletenessPartial
}
```

When the caller passes `query.Limit=0` (no limit, return all), the result is by definition exhaustive — but the code still reports Partial. The UI then shows the "maybe more / load more" affordance for a result set that has nothing more to load. The same pattern likely exists in `searchPageFromRecords` (~line 539-542) — verify and fix in one pass.

## Instructions

1. Change the condition to: when `requestedLimit <= 0`, completeness is `Exact`. When `requestedLimit > 0`, `Partial` only if `returnedCount >= requestedLimit` (i.e., bd may have truncated).
2. Apply the same fix in `searchPageFromRecords` if the same pattern is present.

## Files to Modify

- `internal/repository/beads/lean_reads.go` (~line 620 and possibly ~line 539-542)
- new test in `internal/repository/beads/` covering the helper

## Test

Add `TestLeanSearchMetadata_NoLimit_ReportsExact`:
- requestedLimit=0, returnedCount=5 → Exact
- requestedLimit=10, returnedCount=5 → Exact
- requestedLimit=10, returnedCount=10 → Partial
- requestedLimit=10, returnedCount=15 → Partial (defensive)

Pure helper — no bd subprocess needed.
