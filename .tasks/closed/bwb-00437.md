## Description
The cold-load metric in `scripts/analyze_dashboard_perf.py` misreports sustained-activity bursts as cold-load. Per docs/MONITORING.md, cold-load is defined as wall time until the first >1.5s idle gap. A 53s continuous browse with NO idle gap (session 695efa80: 220 bd calls, max single-call latency only 1095ms, p95 957ms) therefore collapses into a single 53.49s "cold-load" figure — a measurement artifact, not a startup hang.

Split out of d2oj.3 (which now covers only the per-call / dashboard fan-out latency decision).

## Instructions
1. Adjust the cold-load computation in `scripts/analyze_dashboard_perf.py` so it reflects startup latency only — e.g. cap the initial burst by call count or by a bounded wall window, consistent with the >1.5s-gap definition — OR explicitly document the metric as activity-inclusive (and rename/annotate it so it is not read as startup latency).
2. Mirror the gap definition documented in docs/MONITORING.md so the script and the docs agree.
3. Re-run against the session 695efa80 data and confirm the 53s figure no longer presents as a cold-load hang.

## Acceptance Criteria
- [ ] The cold-load metric is either adjusted to reflect startup latency only, or documented/renamed as activity-inclusive
- [ ] The session 695efa80 53s burst no longer reports as a cold-load hang (or is clearly labelled non-startup)
- [ ] Script behaviour matches the cold-load definition in docs/MONITORING.md
