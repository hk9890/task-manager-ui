## Description

The repository layer has thorough parity coverage (`internal/mode/search/parity`, `internal/dashboard/parity`) that guarantees `repo.Search(query)` matches `bd <verb>` output. Controller logic (`internal/mode/search/model.go`) has thorough unit coverage in `internal/mode/search/model_test.go`.

Both layers test fine **in isolation**, yet `beads-workbench-czkq.4` is a real, reproducible bug where the controller silently drops user keystrokes that race with in-flight async commands. The bug is unreachable by either test tier because:

- repository parity calls `repo.Search` directly, no controller, no key handling;
- controller unit tests use `pressAndResolve` (`internal/mode/search/model_test.go:602`) → `testui.ApplyControllerKeySequence`, which **synchronously drains every command before sending the next message**. By the time the next key arrives, `m.loading` is always `false`. The race window the live `tea.Program` exposes simply does not exist in the test harness.

This task adds a small set of tests that exercise the controller against a **deliberately-delayed** repository, so async command overlap is exercised. The goal is to lock in the user-visible contract: "typed query + Enter → result set eventually matches the typed query", regardless of timing.

## Why this matters beyond czkq.4

Three of the open exploration findings live in this same gap (controller behavior under realistic async cadence):

- **czkq.4** (P1) — Search Enter dropped during in-flight load.
- **znri.6** (P2) — Search stale-state labels persist after Enter (likely the same root cause — labels reflect `draft != applied`, which never resolves if Enter was swallowed).
- **czkq.2** (P2) — Detail-view load window shows literal zeros instead of skeleton (different mode, same shape: controller renders fallback values while an async fetch is in flight; no test asserts what the view should render during that window).

A pattern of "controller-async contract tests" — not just for search — is the durable fix.

## Scope (this task: search mode only)

Add a test file `internal/mode/search/model_async_test.go` (or a new `_integration_test.go` if the harness needs it) that:

1. Provides a `DelayedFakeRepository` wrapper around the existing memory/fake repository. The wrapper:
   - delegates all methods to an inner `repository.Repository`,
   - on `Search(...)`, blocks for a configurable duration (or until an explicit `Release()` is called) before returning.
   - allows the test to observe how many `Search` calls are in flight at any moment.
2. Drives the controller via the real `tea.Program` (using `teatest`) or via a manual cmd-runner that mimics `tea.Program`'s "schedule then deliver later" semantics — **not** the synchronous drain helper.
3. Asserts the following contract scenarios (each in a `t.Run` subtest):

### Required test scenarios

- **TypeAndEnterDuringInitialLoad_EventuallySubmitsTypedQuery**
  Open search → wait one tick so `Init()` has scheduled but not yet returned → type `task` → press Enter. After Init resolves AND the typed-query load resolves, assert `appliedQuery == "task"`, the result set matches `repo.Search({Text:"task", Limit:cap})`, and the `draft` badge is no longer present.

- **TypeAndEnterDuringPriorSearch_EventuallySubmitsLatestDraft**
  After Init has resolved, type `foo` + Enter (let it complete), then type `bar` + Enter while the `foo` search is still loading. After both resolve, assert the visible page matches `repo.Search({Text:"bar"})` — the latest draft wins.

- **EnterIsNotSilentlyDropped**
  Inject the delayed repo, type `task` + Enter while Init is loading. Even if the implementation queues vs. immediately fires, after all commands drain, assert that `triggerSearchWithAnchor("task", _)` was invoked at least once (use a recording wrapper around `loadSearchCmd`, or assert end-state instead).

- **HasDraftChangesResolves**
  After the sequence "open + type `task` + Enter + drain all", assert `state.Query == state.AppliedQuery` (so `hasDraftChanges` is false and the "Results below are stale" banner is not rendered).

- **EmptyAutoInitDoesNotLeakClosedRowsUnderTypedDraft**
  Open search, type `task`, press Enter, drain. Assert no closed issues appear in the result set. (This is the bd `--status all` leak that bites users on real repos.)

### Out of scope for this task

- Detail-mode async (czkq.2) — file a follow-up if this pattern proves useful.
- Modifying the controller to fix czkq.4. **This task only adds the failing tests.** They should be marked `t.Skip` (with a referenced bug ID) or left failing as a regression pin, per repo convention.
