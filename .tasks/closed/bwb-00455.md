What I did:
  - Built bwb from main (commit ccdf0c4); set up the embedded fixture via internal/testing/e2e/embeddedfixture/setup.sh.
  - Captured the live UI with scripts/capture_bwb_screen.py at 180x50, started with --repo beads + BD_NON_INTERACTIVE=1.
  - From the board, selected bwf-2 and pressed Enter, then waited for the header text 'Detail:' to appear before taking the screenshot.

Expected (source: docs/CODING.md UI boundaries; intuition for a triage tool where numeric counts must be trustworthy):
  - During the brief window between 'Detail:' appearing and the issue payload completing, the UI should communicate a loading state for ALL data-bearing fields uniformly, OR not render the wrong values at all.
  - The user must never see numerically wrong counts shown as if they were the truth (e.g. 'Comments: 0' rendered identically to 'Comments: 2').

Actual:
  - With a 1.5s startup wait and a step that only waits for the literal text 'Detail:' (a structural marker), the captured screen for bwf-2 shows:
    - Dependencies panel: 'Blocked by (0) (none)', 'Blocks (0) (none)', 'Related (0) (none)'
    - Metadata Counts: 'Comments: 0', 'Blocked by: 0', 'Blocks: 0', 'Related: 0'
    - Description column: rendered as a clearly-placeholder pattern of 'X XX XXX ░░░░░░░░' style blocks (this IS visibly a skeleton).
  - Re-capturing with an additional 'wait-for-text:Used to validate' step (the real description string) loads the full real data: Blocked by (1)→bwf-1, Comments=2, etc., all matching bd show / bd comments exactly.
  - So the underlying data fetch is correct; the issue is the loading-state TREATMENT: the description column uses an obvious placeholder pattern, but the counts and dep lists render the *placeholder/default* values (zero/none) styled identically to real data.

Severity: P3
  - Cosmetic / UX, not a data corruption. Once the data finishes loading (sub-second on a tiny fixture; longer on bigger repos) the counts are correct.
  - But on a slower repo or a slow bd invocation, a user pressing Enter and immediately reading the metadata could be misled into thinking the issue has zero deps / zero comments. The header still says 'blocked' so the discrepancy ('blocked but Blocked by: 0') is internally inconsistent and looks like a bug to the user.

Repro:
  1. go build -o /tmp/bwb ./cmd/bwb
  2. repoPath=$(mktemp -d) && sh internal/testing/e2e/embeddedfixture/setup.sh "$repoPath" internal/testing/e2e/embeddedfixture/seed.json
  3. python3 scripts/capture_bwb_screen.py --cwd "$repoPath" --width 180 --height 50 --startup-wait 1.5 \
       --step 'wait-for-text:Ready:5000' \
       --step 'send-key:ENTER' \
       --step 'wait-for-text:Detail::3000' \
       --step 'checkpoint:detail-open' \
       --step 'send-key:CTRL+Q' \
       -- -- env BD_NON_INTERACTIVE=1 /tmp/bwb --repo beads
  4. Inspect the captured screen for bwf-2: dep lists are 0/(none) and Metadata Counts are all 0, while header reads 'Detail: bwf-2 · blocked'.

Suggested direction:
  - Use the same placeholder/skeleton treatment for the counts and dep lists as the description currently does, so the loading state is uniformly recognisable.
  - Or hold rendering of the Counts and Dependencies panels until the payload arrives (show a spinner / 'Loading...' in those frames).
