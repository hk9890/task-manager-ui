## Problem

Manual refresh ('r' key) in both board and search modes bypasses the in-flight guard that AutoRefresh respects. Pressing 'r' while a refresh is already running enqueues a second concurrent refresh, which corrupts the board's pendingResults counter and partial-result buffers and leaves the UI in a broken state.

## Observed symptoms

- After two rapid 'r' presses the board can render stale or incomplete columns.
- Selection / focus is wiped twice and restored against partial data, producing out-of-bounds rows.
- Subsequent refreshes inherit a poisoned (negative) pendingResults counter — they can fire maybeCompose early with only a fraction of the columns updated.
- Each extra refresh spawns 4× additional bd subprocesses.

## Root cause

`internal/mode/board/model.go:233-234` — the BoardActionReload keyboard handler calls `m.startReload(refreshModeManual)` with no guard. Compare `AutoRefresh()` at line 323-328 which guards with `if m.IsLoading() { return nil }`.

`internal/mode/search/model.go:243-244` — the SearchActionReload keyboard handler calls `m.triggerSearchPreservingSelection()` with no guard. Compare `Reload()` (line 380-385) and `AutoRefresh()` (line 388-396) which both check `if m.loading`.

`startReload()` itself (line 333-372) is destructive — it resets pendingResults=4 and clobbers the partial buffers unconditionally. There is no defensive guard inside the function, so any future caller can re-trigger the bug.

`maybeCompose()` (line 376-439) only guards with `if m.pendingResults > 0`, so negative counter values trigger compose on every result arrival.

## Goal

Refresh must run one-at-a-time per surface. A manual refresh request while a refresh is in flight must be a no-op (logged at debug level so the suppression is visible in `--debug` traces).

## Acceptance

- Pressing 'r' rapidly in board mode never causes pendingResults to be reset while >0.
- Pressing 'r' rapidly in search mode never spawns concurrent gateway queries.
- pendingResults never goes negative under any keypress sequence (regression test).
- Visible user behavior: the refresh that is already running completes normally; the user's extra keypresses are no-ops (no flicker, no selection reset, no compose with partial data).
- Suppressed refreshes are logged at debug level with the trigger source (board-manual, search-manual) so the suppression is visible in `bwb --debug` traces and the persistent JSON Lines log.
- Auto-refresh paths in app/model.go continue to work unchanged.

## Out of scope

- Cancelling an in-flight refresh.
- Queueing multiple pending refreshes (one-pending-after-current is a possible follow-up if UX demands it).
- Surfacing a "refresh ignored" toast — defer unless explicitly requested.

## Notes / design

- Simplest fix is to add `if m.IsLoading() { return nil }` at the keyboard-handler call site for board, and `if m.loading { return nil }` for search.
- Defense-in-depth (task .3): introduce a dedicated `inflight bool` set by `startReload` and cleared by `maybeCompose`. Use it inside `startReload`'s own guard so future callers cannot bypass it. The dedicated bool removes the Init/cold-start ambiguity that `IsLoading()` (column-loading driven) would create.
- Bubble Tea's single-threaded Update means a mutex is not needed — boolean state is sufficient.
- Tests are model-level: drive the Update loop with two BoardActionReload key events back-to-back and assert no negative counter, no double column-reset. The existing `applyMessages` / `runBatch` pattern in `internal/app/model_test.go` plus capturing returned Cmds without invoking them (pattern used at `internal/app/model_loading_refresh_test.go:117`) is sufficient — no fake-gateway changes needed.
