## Description

After the restructure, `docs/TESTING.md` should describe the new three-tier model clearly: unit / gateway-integration-read / gateway-integration-mutate, plus the fake/real contract suite that bridges them.

## Why

Future contributors (human and agent) need to know where new tests belong without re-deriving the architecture from the file structure.

## Instructions

1. Read the current `docs/TESTING.md`.
2. Restructure it to cover, in this order:
   - **Tiers**: unit (fast, no external deps), gateway-integration (real `bd`, behind `//go:build integration`)
   - **Where new tests go**:
     - Asserts app behavior given any gateway state? → unit, use `fakes.FakeBeadsGateway`
     - Asserts the bd CLI adapter's contract? → `internal/gateway/beads/contract/` read contract
     - Asserts a multi-step bd write workflow? → add to an existing scenario or create a new one (cap: ~3 scenarios total)
   - **Fixture rules**:
     - Reads → call `embeddedfixture.SharedFixtureRepoPath(t)`. Never mutate the shared snapshot.
     - Mutations → call `embeddedfixture.Seed(t, tb.TempDir())` for a fresh fixture.
   - **The contract test** — why it exists, what it guarantees (fake/real parity), what to do when it catches drift (fix the fake, don't weaken the assertion).
   - **Commands** — already covered, but update if any task names changed.
3. Cross-check `docs/CODING.md` and `AGENTS.md` for any stale references to integration-test patterns; update if needed.
4. Verify links from other docs still resolve.

## Files to Modify

- `docs/TESTING.md`
- (possibly) `docs/CODING.md`, `AGENTS.md` — only if they reference test patterns that changed
