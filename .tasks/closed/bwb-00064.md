## Description
The repository has no `docs/RELEASING.md`, so the release process currently lives only in operator memory and ad-hoc issue comments. Add a concrete releasing guide that captures the commands, checks, and expectations already used in this repository so future releases are repeatable.

## Instructions
1. Create `docs/RELEASING.md`.
2. Document the release prerequisites:
   - clean working tree
   - branch synced with remote
   - required quality gates from `docs/CODING.md` (`go build ./cmd/bwb`, `go vet ./...`, `go test ./...`)
3. Document when runtime verification is required for user-facing changes, linking to `docs/TESTING.md` and `docs/RUNTIME_UI_VERIFICATION.md`.
4. Document the release steps for this repository, including:
   - choosing the version/tag
   - confirming whether release notes/changelog updates are needed
   - creating/pushing the tag
   - creating or editing the GitHub release with `gh release`
   - post-release verification
5. Explicitly note the current CI reality discovered during release reconciliation: there is no `.github/workflows` automation yet, so releases currently rely on manual local verification until CI is added.
6. Explicitly note that release visibility/public-vs-private policy is still unresolved and tracked separately in `beads-workbench-2rj`; the guide should avoid inventing a policy and instead mark that decision point clearly.
7. Update `README.md` to link to `docs/RELEASING.md`.
8. Keep the document command-oriented and specific to this repo rather than generic release advice.

## Files to Modify
- `docs/RELEASING.md` (create)
- `README.md` (update: add releasing guide link)
