## Description

The "two column-top borders stacked" bug shipped because no test exercises the cold-start → WindowSizeMsg → re-render sequence as a single `View()` capture. Existing goldens (`TestRenderBoardColumnsGolden`, `TestBoardModeDashboardLayoutGoldensAcrossWidths`) snapshot at one resolved width — they cannot catch frame-stacking.

## Instructions

1. Create `internal/mode/board/render_regression_test.go`.
2. Use `teatest` to drive the full app model (not just board mode in isolation):
   - **Scenario A — cold start**: build model, do NOT send WindowSizeMsg, call `View()`, assert exactly 4 occurrences of `╭─` (column-top borders) in the output.
   - **Scenario B — resize**: build model, send `WindowSizeMsg{80,30}`, feed all 3 column results, call `View()`, assert exactly 4 `╭─`. Then send `WindowSizeMsg{200,60}` and feed fresh results; call `View()` again, assert exactly 4 `╭─`. Never 8.
   - **Scenario C — pre-size-then-data**: send `WindowSizeMsg` before any data, then feed data, then resize, then more data. Assert column-top count remains 4 in EVERY `View()` capture.
3. Generalize as a helper `assertColumnTopCount(t, view string, want int)` that counts box-drawing top-left/top-right corners (`╭` and `╮`), accepting both with and without horizontal extension.
4. Add a parallel `internal/mode/search/render_regression_test.go` covering the search view's panes (query box + results box + preview pane) with the same approach.
5. Wire one of these into `mise run quality:fast` so it runs pre-commit.

## Acceptance Criteria

- [ ] All scenarios pass on current main when the suspected doubled-header bug is reproduced (test demonstrates failure mode)
- [ ] After [[bug-doubled-headers]] is fixed, all scenarios pass
- [ ] Helper rejects views with >4 column-top borders with a clear message: `expected 4 column-top corners, got 8 — frame stacking detected`
- [ ] Search-view variant similarly counts pane borders and rejects duplicates
- [ ] Runs in <2s as part of `mise run test`
