## Description
Several slow integration packages assert their cost without a named per-test risk justification in TESTING.md: `cmd/bwb-smoke` (62.8s), `internal/dashboard/parity` (51.8s), `internal/mode/search/parity` (27.3s), `internal/mode/search` integration (18.4s), and the `datasets`/`embeddedfixture`/`loadgen` (~20s each) packages. Q3 of the review requires each slow test to name one bug a fast test would miss. The `bwb-smoke` 62s release smoke check is the headline challenge: what regression does a real-binary run catch that the in-process embedded-fixture teatest suite does not?

## Instructions
1. For each slow integration package above, add a one-line risk justification to TESTING.md naming a concrete regression a fast test would miss.

3. Explicitly resolve the `cmd/bwb-smoke` challenge: justify the 62s or recommend reducing it.
