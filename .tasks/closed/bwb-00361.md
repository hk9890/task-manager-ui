## Description

`internal/config/load.go:209-234` (`validateResolved`) calls `ResolveKeyBindings` as a side-effect validation step. Then `internal/app/model.go:170` (`NewModelWithOptions`) re-resolves them and **panics** on error. If anything mutates `cfg.KeyBindings` between load and model construction, or if a test/caller builds a `Model` directly with a custom `cfg`, the panic path triggers at runtime instead of a clean startup error.

Constructor panic is the wrong failure mode — startup errors should surface via the existing exit-code contract (CODING.md exit codes 1/2).

Major finding BS4 from the 2026-05-16 audit.

## Files to Modify

- `internal/config/load.go`
- `internal/app/model.go`
- `internal/app/services.go` (maybe)

## Instructions

1. Have `config.Load` return the already-resolved `KeyBindings` (or attach them to the `Config` struct) so the model can read them without re-resolving.
2. `NewModelWithOptions` should consume the pre-resolved bindings. If they are missing, return an error rather than panic.
3. Adjust callers (`cmd/bwb/main.go`) to handle the error explicitly and exit with the documented code.
4. Add a test: construct a `Config` with an invalid keybinding and assert `NewModelWithOptions` returns a typed error (not a panic).

## Acceptance Criteria

- [ ] Keybinding resolution happens exactly once at startup
- [ ] `NewModelWithOptions` returns errors instead of panicking on invalid bindings
- [ ] Test asserts the error path
