## Description
Complete the first end-to-end backend seam by implementing the core write operations through official `bd` commands. This makes the new gateway useful for real issue mutation flows before the UI is built.

## Instructions
1. Implement gateway support for create issue, update issue, close issue, and add comment operations.
2. Use the shared command runner and normalized error model.
3. Use this command baseline:
   - `bd create`
   - `bd update`
   - `bd close`
   - `bd comments add`
4. Map gateway input models to official command arguments without introducing unofficial backend access.
5. Keep error handling consistent with the read-path behavior.
6. Do not add SQL, BQL, or orchestration dependencies.

## Files to Modify
- `internal/gateway/beads/` (implement write operations)
- `internal/domain/` (adjust mutation models only if needed)
- `*_test.go` under gateway packages (create/update tests)

## Acceptance Criteria
- [ ] Create issue is available through the gateway interface
- [ ] Update issue is available through the gateway interface
- [ ] Close issue is available through the gateway interface
- [ ] Add comment is available through the gateway interface
- [ ] Write-operation failures surface through the same normalized gateway error model used by reads
- [ ] No unofficial backend access is introduced
