## Description
The fatal error screen path in `internal/app` has no test coverage. A future refactor of the health check gate or `Update()` dispatch order could silently break the fatal error screen without any test failing.

Discovered during acceptance review of beads-workbench-5j04.

## Instructions
1. In `internal/app/model_test.go` or a new `scenario_healthcheck_test.go`, add tests covering:
   - `startupHealthCheckMsg` with `ErrorCodeCommandUnavailable` → `fatalErr` is set, `View()` returns fatal error screen
   - `startupHealthCheckMsg` with `ErrorCodeCommandUnavailable` → `Update()` only handles `q`/`ctrl+c` (returns `tea.Quit`) and window resize; all other messages are ignored
   - `startupHealthCheckMsg` with nil error → normal board initialization proceeds (no `fatalErr`)
   - `startupHealthCheckMsg` with a non-`GatewayError` → falls through to normal operation (no fatalErr set)
2. Add at least one test for `internal/ui/fatalerror/fatalerror.go`:
   - `View(80, 24)` output contains "beads is not available", "bd", and "q" hint
   - `View(0, 0)` does not panic and still contains expected content

## Files to Modify
- `internal/app/model_test.go` (or new scenario file) — health check tests
- `internal/ui/fatalerror/fatalerror_test.go` — new, basic View tests

## Acceptance Criteria
- [ ] Tests cover all four startupHealthCheckMsg scenarios above
- [ ] At least one fatalerror.View test exists
- [ ] `go test ./internal/app/... ./internal/ui/fatalerror/...` passes
