Per user direction, the currently-viewed issue must never appear in the detail browser side panel. It only ever sneaks in via the parent-group sibling list (parentChildSiblings returns ALL of the parent's children, including self), so when you open a child it was listed among its own siblings.

Fix: exclude detail.Summary.ID in browserItemsFromDependencies (internal/mode/details/model.go) — the single chokepoint downstream of all repository backends (beads/memory/caching), which keeps the panel groups, nav flat-list, and DependencyRefLineIndex walk in lockstep. Since the viewed issue can then never be in the panel, the entire 'subject marker' mechanism added by 44rr.2 is now dead and was removed in one pass: SubjectIssueID State field + plumbing, the isSubject render branch, and the unused DependencyCursorStyle/DependencyCursorBgColor styles. The browser panel now has exactly one marker — the movable '> ' cursor.

Verified against real parent-child data (bd create --parent fixture, 160-col TUI): opening child pc-1.2 shows Structure = [pc-1 parent, pc-1.1, pc-1.3] with pc-1.2 absent. The board-selected issue is the anchor (SelectionID); Enter on a sibling previews it (9dn stable-browser design) with the cursor on the previewed row.
