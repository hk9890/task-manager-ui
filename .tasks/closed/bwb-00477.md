What I did: built /tmp/bwb, ran 'bwb --config <cfg> --check-config' with three crafted keybinding configs.

Expected (source: docs/CONFIGURATION.md lines ~105-108): 'Unknown actions, invalid key names, missing required actions, and conflicting keys in the same context fail startup.'

Actual:
1. Invalid key name — config 'keybindings.board.move_down: [dwn]' -> output 'config OK', exit 0, NO warning. move_down is silently bound to the literal string 'dwn', which Bubble Tea never emits, so j/down stop moving down on the board with zero diagnostic. Root cause: isValidKeyName() in internal/config/keybindings.go accepts any non-empty, non-space string, so it cannot detect a typo'd key name.
2. Unknown action — config 'keybindings.board.fly_away: [z]' -> only a WARNING ('unknown config key "keybindings.board.fly_away" ignored'), still 'config OK', exit 0. The strict-YAML decoder treats it as an unknown key before keybinding resolution; the allowed-action check in buildContextBindings never runs for it. So a typo'd ACTION name also silently does nothing.
3. Conflicting key — config 'move_down: [j, h]' -> correctly fails with exit 1 ('key "h" conflicts between actions...'). This one matches the doc.

Severity: medium. Operator customizing keybindings with a small typo is told 'config OK' and silently loses a key binding. No machine-visible signal for case 1 at all.

Repro:
  go build -o /tmp/bwb ./cmd/bwb
  printf 'keybindings:\n  board:\n    move_down: [dwn]\n' > /tmp/c.yaml
  /tmp/bwb --config /tmp/c.yaml --check-config   # prints 'config OK', exit 0

Fix options: either (a) make isValidKeyName validate against a real key-name set so case 1 fails or at least warns, and surface unknown keybinding actions as a hard error rather than a generic unknown-key warning; or (b) correct CONFIGURATION.md to describe the actual behavior (key names are not validated; unknown actions are warned-and-ignored, not fatal).
