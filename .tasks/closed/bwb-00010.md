## Description
User-provided screenshots comparing Perles (`pearles.png`) with the current Beads Workbench dashboard (`our-tool.png`) confirm that the current UI is still not close to a proper triage-quality board. The latest screenshots make the remaining problems clearer than earlier terminal captures: Perles achieves much higher information density, clearer lane semantics, and better scanability, while Beads Workbench still feels sparse, over-expanded, and structurally less useful for day-to-day triage.

## Screenshot-Specific Findings
1. **Rows are still too tall and too sparse**
   - Perles fits significantly more actionable items per column.
   - Beads Workbench uses two-line rows everywhere, which dramatically reduces visible issue count.

2. **The detail pane is consuming too much value relative to its cost**
   - In Beads Workbench, the selected-issue pane occupies a large area but mostly shows long wrapped issue body text.
   - In Perles, the board itself remains the star of the screen and supports fast triage.

3. **Queue/group semantics are weaker than Perles**
   - Perles uses triage-meaningful groupings (`Not Ready`, `Ready`, `In Progress`, `Done`) that map well to decision-making.
   - Beads Workbench's current `Ready / In Progress / Blocked / Recent Work` layout is less immediately useful and less balanced visually.

4. **Metadata presentation is less glanceable**
   - Perles rows encode issue type, priority, readiness state, and ID in a compact, scannable prefix.
   - Beads Workbench still requires more visual effort to parse row meaning.

5. **Overall information architecture still feels like a shell around data, not a polished dashboard**
   - The current screen is improved over earlier iterations but still far from the clarity and confidence of the Perles dashboard.

## Instructions
1. Redesign the default board/dashboard so it is optimized for triage density and glanceability, using Perles as a UX benchmark only.
2. Revisit the default lane/group strategy; do not assume the current grouping is correct.
3. Rework row presentation so key status/type/priority/ID/title signals are readable in a compact single-row or otherwise denser form.
4. Rebalance or reduce the persistent detail pane if it harms board density; the board view itself must become the primary high-value surface.
5. Validate changes with updated golden tests plus real PTY/runtime review against realistic repo data and screenshot comparison.

## Files To Modify
- `internal/dashboard/`
- `internal/ui/board/`
- `internal/app/model.go`
- related board/app tests and goldens

## Acceptance Criteria
- [ ] The default dashboard is materially closer to Perles-level triage usability
- [ ] The board shows significantly more actionable work at once than the current two-line-row layout
- [ ] Row metadata is compact and glanceable enough for fast scanning
- [ ] The board/detail balance improves overall triage usefulness rather than consuming space on low-value detail text
- [ ] PTY/runtime review on realistic repo data shows a polished, reviewable dashboard experience
- [ ] Updated golden/render tests lock in the improved dashboard information architecture and presentation
