## Description
`internal/app/edit_flow_teatest_test.go` has three test cases that wait for a condition with `testui.WaitForConditionWithTimeout` and then `time.Sleep(200 * time.Millisecond)` to let the BubbleTea runtime dequeue `editIssueResultMsg` and run the Update handler that sets the toast before sending `tea.Quit`.

The comment is honest: there is no observable condition the test can poll. The 200ms is a flake budget under CI load. The three sleeps are also the dominant cost in the slowest unit tests:

- `internal/app/edit_flow_teatest_test.go:161` (TestEditFlowSuccessPathTeatest — 0.30s)
- `internal/app/edit_flow_teatest_test.go:247` (TestEditFlowNoChangeTeatest — 0.28s)
- `internal/app/edit_flow_teatest_test.go:332` (TestEditFlowEditorErrorTeatest — 0.28s)

## Instructions
1. Read `internal/app/edit_flow_teatest_test.go` in full, focusing on the three sleep sites and what assertion follows each. Note the existing pattern at lines 142, 237, 323 — these already use `testui.WaitForOutputContainsAllWithTimeout` polling the rendered output. The post-condition sleep exists because after the gated condition resolves, the toast-set message has not yet flowed through Update.
2. Read the BubbleTea model in `internal/app/model.go` (look for the `Update` handler for `editIssueResultMsg` and the `toast` field). The toast is rendered via `m.toast.Overlay(view, ...)` only when `m.toast.Visible()` is true (see `model.go` around line 867).
3. Find an observable signal that the toast has actually been set. Preference order (revised after plan-review):
   - **Preferred**: extend the existing `testui.WaitForOutputContainsAllWithTimeout` pattern in this file to poll for the toast text in the rendered View output. The View already includes the toast overlay when `m.toast.Visible()`, so output-text polling is the most natural primary signal.
   - **WARNING**: the in-file doc-comments at lines 92-96 and 278-283 note that "post-tea.Exec View() frames do not reliably reach the output pipe under CI load." If a stress test (step 5) confirms output polling is flaky here for the post-`tea.Exec` frames, you must fall back to the signal-channel approach below.
   - **Fallback (test-only signal)**: add a test-only `chan struct{}` or counter on the model that the Update handler signals when it processes `editIssueResultMsg`. Wire it via a constructor option used only from the test, then poll the channel/counter with `WaitForConditionWithTimeout`. This is invasive but reliable. Only take this path if output polling demonstrably flakes under 10× -race.
   - **DO NOT** attempt to poll the live model state via `WaitForConditionWithTimeout` reading a model field — teatest owns the model during the run; only `FinalModel()` exposes it after Quit.
4. Replace each `time.Sleep(200 * time.Millisecond)` with `testui.WaitForConditionWithTimeout(t, 1*time.Second, ...)` (or the WaitForOutputContains variant) polling the chosen signal. Fail with a clear error if the timeout fires.
5. Stress-test for flake:
   ```
   mise run test -count=10 -race ./internal/app/...
   ```
   All three teatest cases must pass on all 10 iterations. If output polling fails any iteration, switch to the signal-channel fallback and re-run.
6. Confirm individual test wall times dropped:
   ```
   mise run test -v ./internal/app/... 2>&1 | grep -E "TestEditFlow.*Teatest"
   ```
