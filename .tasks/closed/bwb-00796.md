## Description
Implement the built-in dashboard definition provider and map dashboard sections to supported gateway queries. This preserves the product rule that dashboards are built-in and query-backed, not BQL-driven.

## Instructions
1. Implement dashboard definition types and a `DashboardDefinitionProvider` interface under `internal/dashboard/`.
2. Implement the built-in provider under that interface.
3. Define built-in views for ready, in-progress, blocked, recent work, and assigned-to-current-user if supported cleanly by official surfaces. If implementing the assigned-to-current-user view, determine the current user identity from the beads environment (for example `BEADS_ACTOR` or an equivalent supported source) and pass it as a filter through the gateway.
4. Map each board section to supported gateway query types rather than arbitrary query strings. In-progress and recent-work views may map to filtered/sorted `ListIssues` queries rather than dedicated gateway methods.
5. Keep dashboard rendering separate from dashboard definition loading.

## Files to Modify
- `internal/dashboard/` (create provider interface and built-in definitions)
- `internal/domain/` (update dashboard models if needed)
- `internal/mode/board/` (create/update board query integration if needed)

## Acceptance Criteria
- [ ] A `DashboardDefinitionProvider` interface exists
- [ ] A built-in dashboard provider exists behind that interface
- [ ] Ready, in-progress, blocked, and recent views are defined
- [ ] Assigned-to-current-user is either implemented or explicitly documented as deferred with a reason
- [ ] Dashboard definitions map to supported gateway query types only
- [ ] No BQL parser or arbitrary query execution is introduced
