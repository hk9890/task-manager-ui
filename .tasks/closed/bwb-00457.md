## Status: Closed — not reproducible (PTY harness artifact)

This bug was filed during a 2026-05-24 PTY-driven exploration and **does not reflect a real defect**. Kept as a historical record so the same false-positive isn't refiled.

## Original Observation

`scripts/capture_bwb_screen.py` ran `bwb --repo caching` against a fresh embedded fixture (no prior `~/.cache/bwb/...` for the project) and observed the board rendering with all four columns at `0` and a status of `Idle`. The persistent log (`~/.local/state/bwb/bwb-<session>.log`) showed only `bd vc status` and `bd ping --json`; no `bd query`, `bd ready`, or `bd count` calls were recorded. `--repo beads` against the same fixture rendered correct counts immediately.

The original filing hypothesized a manifest-hash invalidation bug: empty cache + manifest written with current bd hash → subsequent invalidation check sees "manifest matches" → no read fan-out fires.

## Investigation

Subsequent reproduction with the contract test suite and the kb8e cold-start verification harness (`scripts/verify_kb8e_manual_test.go` F1) demonstrates the caching backend correctly fetches Dashboard data on cold start and writes a populated JSONL. The original PTY capture race was the cause:

- `capture_bwb_screen.py` had a silent fall-through on unknown key names (now fixed in znri.8) and timing-based waits that did not synchronize with the cache's async load.
- The 2.5–5 s wait was occasionally too short for a cold cache + first Dashboard fetch on this hardware.
- Once the harness was fixed and tests were added at the package level, the bug could not be reproduced.

## Resolution

Closed `not-reproducible`. The real fix that landed was indirect: znri.8 hardened the PTY harness so the same false-positive cannot recur, and the kb8e epic later added structural tests around the caching layer that would have caught a genuine cold-start regression.
