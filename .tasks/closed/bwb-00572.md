## Discovered during

[[8qw9.1]] (gateway contract audit, 2026-05-16).

## Expected vs actual

SearchIssues routes text queries through `bd search --json --status <csv>`. `bd list` accepts comma-separated status values, but **`bd search --status` accepts ONLY a single token**. `bd search task --status open,closed` returns `[]` without error, silently discarding all results.

## Verification required

Must verify in the running app (`mise run bwb`, Search mode) whether the UI currently exposes a path that sends text + multiple status filters to the bd search route. If no UI path triggers this (the search view may not support multi-status filtering today), the bug is INERT and should be closed as 'not a bug' with rationale. If it does, fix.

Orchestrator preliminary check: read internal/mode/search/model.go and internal/ui/search/search.go to confirm whether multi-status + text is reachable. If not currently reachable, defer or close as not-a-bug.

## Suggested fix layer (if confirmed reachable)

Gateway (SearchIssues in `read_gateway.go`): when routing through `bd search` and multiple statuses are requested, either (a) issue one `bd search` call per status and merge in-memory, or (b) use `--status all` and post-filter in-memory.
