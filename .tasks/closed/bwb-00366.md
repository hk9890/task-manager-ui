## Description

Two related interpolation hygiene problems in `internal/launcher/service.go`:

1. **Env entries without `=`** — `service.go:156-159` + `process_runner.go:20-22` pass interpolated Env strings to `cmd.Env` without validating each contains `=`. A YAML override like `Env: ["BWB_TITLE_{{issue.title}}"]` produces malformed env entries that depend on libc behavior to drop. Minor finding m4.
2. **Control chars in interpolation values** — `service.go:92-106` does not strip `\x00`, `\x1b` (ANSI), or newlines from interpolated values. `\x00` causes `exec.Command` to reject argv (probably fine), but newlines in Env values produce multi-line env entries; ANSI escapes in argv reach child shells. Minor finding m5.

## Files to Modify

- `internal/launcher/service.go`
- new test: extend `internal/launcher/service_test.go` (or the new security file from beads-workbench-db0z.1)

## Instructions

1. After interpolating an Env entry, validate it matches `^[A-Za-z_][A-Za-z0-9_]*=.*` (the standard env-name shape). Reject the launch with a clear toast error if invalid.
2. Strip C0 control chars (`\x00`-`\x1f` except `\t`) from interpolation values before substitution. Apply to argv, command, env values, and workdir.
3. Add tests:
   - `Env: ["NO_EQ"]` after interpolation → launch fails with validation error
   - `Title: "line1\nline2"` interpolated into argv → child receives the literal `\n`, not actual newline
   - `Title: "\x1bdanger"` → ANSI escape stripped before reaching child

## Acceptance Criteria

- [ ] Env entries missing `=` are rejected with a clear error
- [ ] Control chars stripped from interpolation values
- [ ] Tests cover the three scenarios above
