## Description

The dashboard and search rely entirely on `beads.BeadsGateway`. Today its contract is 33 lines of method signatures with almost no documented invariants. The `RunReadContract` test suite pins specific fixture IDs but does not assert generic semantics (sort order, completeness, ID non-emptiness, count-vs-list-length consistency). When real bd or the fake silently drifts from what consumers expect, the bug surfaces in the UI, not in tests.

This epic makes the contract the single source of truth: documented invariants in `interface.go`, machine-checked in `RunReadContract` against BOTH the fake AND real bd on the same fixture, enforced at runtime with `slog.Warn` tripwires when bd violates expectations in production. The fake becomes a contract-conforming peer of the real gateway, not a casual mock.

User intent: "we will not build 100 integration tests; we must invest into the test layer; it must mimic the beads system." This epic is that investment.

## Discovered-during-audit bug discipline (applies to all child tasks)

The audit (8qw9.1) and the fake-fidelity work (8qw9.3) will almost certainly reveal cases where bwb assumes something the gateway/bd doesn't actually guarantee — places where the contract bwb relies on is broken in reality. When this happens:

1. **File a separate bug** (`bd create --type=bug`) — do NOT just note it in the audit; surface it as a real defect with its own tracking. Title format: "Gateway contract violation: <consumer> assumes <X>, real bd does <Y>".
2. **In the bug body, require app-level verification**: state explicitly "Must verify in the running app (mise run bwb) that this assumption-vs-reality gap causes user-visible incorrect behavior — not just a theoretical contract violation." Some assumption mismatches are inert (the wrong behavior never reaches the user); those get closed as "not a bug" with rationale. Real ones get fixed.
3. **Fix the bug at the right layer**:
   - Consumer side (board model, search model, app model): fix the consumer's expectation OR add a normalization step in the gateway.
   - Gateway side (decoder, argv builder): fix the gateway to honor the documented contract.
   - Fake side: update the fake to match real bd's observed behavior (so future tests catch the issue).
4. **Test the fix with the fake gateway, not just integration tests**. The fake should reproduce the original violating behavior; after the fake is updated to mimic real bd, the fixed consumer should pass against the updated fake. This is the loop the user wants: invest in the test layer so it mimics reality, then bugs are caught at fake-level test time, not in production.
5. **Link the bug** to the audit task (8qw9.1) or fake task (8qw9.3) that surfaced it via `bd dep add` so the discovery context is traceable.
