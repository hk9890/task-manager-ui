Reported during review of the Parent-only work: opening an epic, drilling into a child, then trying to jump back to the epic via the child's Parent row did not work. Root cause: drilling (OpenRelatedIssueIntent) set TargetID but not SelectionID, so detailLoadedMsg routed the load to ApplyPreviewDetail — Content/Metadata updated but the Dependencies rail stayed ANCHORED to the issue you came from. So a drilled child showed the EPIC's Children group, never its own Parent group. This is the 'Dependencies completely updated on selection' behavior requested originally. Fix: drilling is now a full navigation — set SelectionID=TargetID and seed an optimistic placeholder via ApplyLoadedDetail so the rail follows the target while the header renders immediately. Also fixed a fake-vs-real divergence: the in-memory repository did not populate detail.Children at all (real bd always does), so it now derives Children via reverse parentID lookup.
