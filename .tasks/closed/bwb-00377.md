## Description
Research how Beads Workbench should represent and handle result completeness for capped `bd` queries, especially in search flows, and determine both the search-interface design and the data-access contract changes required to support correct semantics.

## Instructions
1. Inspect the current flow from `bd` command execution through gateway decoding, domain result types, and search-mode consumption.
2. Document where result completeness is currently lost or misrepresented.
3. Evaluate search-interface design options for communicating complete vs capped/partial results, query state, and operator next actions.
4. Evaluate design options for expressing completeness/partial-result state in the gateway/domain contract, including whether new metadata or result abstractions are required.
5. Assess constraints from official `bd` command surfaces, including what is and is not available in JSON output versus stderr.
6. Propose a recommended architecture direction with UI/UX tradeoffs, data-contract tradeoffs, risks, and follow-up implementation slices.
7. Create follow-up task proposals needed to execute the recommended direction.

## Acceptance Criteria
- [ ] Current completeness/truncation semantics are mapped from runner → gateway → domain → UI.
- [ ] The research explains how the search UI should communicate result completeness and operator options.
- [ ] The research explains whether interface changes are required and why.
- [ ] At least one recommended architecture direction is documented with tradeoffs.
- [ ] Follow-up implementation tasks are identified concretely enough to create tracker issues.
