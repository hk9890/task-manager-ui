## Description
Research complete for a compact wide-screen left related-issues rail distinct from the donor tree view. This task defines the recommended layout and interaction model so implementation tasks can be created without ambiguity.

## Recommendation
### Layout
- Enable a 3-column detail layout only on wide terminals at about 140 columns or wider.
- Allocate the left rail at roughly 20 percent of available width, clamped to 22-28 columns.
- Keep the existing right metadata rail at 34 columns.
- Keep the main content column as the dominant pane. If width falls below the breakpoint, fall back to the current 2-column or single-column detail layout and keep related work inline as today.

### Related groups
- v1 groups: Blocked by, Blocks, Related.
- Do not show a tree, nesting, connectors, or expand/collapse affordances.
- Do not show Children in v1 until gateway/domain parsing can distinguish parent-child dependents from ordinary blocking dependents.
- When the data model gap is addressed later, insert Children as its own group between Blocks and Related.

### Interaction model
- Dedicated detail mode only: the left rail should be keyboard navigable.
- Left/right changes focus across left rail, content, and metadata panes.
- Up/down moves within the active rail list; Enter opens the selected issue in detail mode.
- Search preview and compact detail views remain read-only summaries rather than interactive rail surfaces.
- Mouse support can remain limited to scroll for v1; no donor-style tree interactions.

### Row content
- Use a single-line compact row derived from shared issuerow/renderhelpers.
- Show type, priority, compact status, compact issue id, and a truncated title.
- Prefer one dense line per issue over two-line rows or metadata-heavy badges.
- Keep groups always expanded in v1 to avoid extra state and visual noise.

### Donor boundaries
- Reuse only the donor ideas of grouped dependency lists and pane focus.
- Do not port the donor tree widget, dependency tree mode toggle, or children/deps tree presentation.

## Current data model constraints
- `internal/domain.IssueDetail` currently exposes `BlockedBy`, `Blocks`, and `Related` only.
- `internal/gateway/beads/read_gateway.go` maps all `bd show` dependents into `Blocks` and does not preserve `dependency_type`.
- `bd show --json` already contains `dependency_type=parent-child` for child issues, so a future task can parse and expose `Children` separately.

## Follow-up implementation tasks after approval
1. Add relationship-aware detail data mapping and tests if Children will be shown.
2. Implement a 3-column detail renderer with wide-screen left-rail fallback behavior.
3. Add detail-mode pane focus and navigation behavior for rail/content/metadata.
4. Extend shared compact issue row rendering for the narrower left-rail width and add golden coverage.
