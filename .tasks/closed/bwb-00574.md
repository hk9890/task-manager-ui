## Description

The search view has the same data-consistency risk as the dashboard: result count and order must match what `bd search` returns for the same query. Existing tests cover rendering/state but not parity with bd. We need explicit assertions across real datasets.

## Instructions

1. Create `internal/mode/search/parity/search_parity_integration_test.go` (build tag `integration`).
2. Define a small fixed query matrix (literal strings to keep tests deterministic):
   - empty query (default list)
   - common word query (e.g. `"test"`, `"fix"`, `"render"`)
   - filtered query (status=closed plus text)
   - work-state query (ready/blocked)
3. For each (dataset, query) pair:
   - Run the search through the gateway as the search mode does (`SearchIssues` with the same args).
   - Run `bd search <text> [flags] --json --limit N` from the dataset's directory.
   - Assert: total count match, first-page IDs match in order, `IsExact` flag matches whether bd hit its limit.
4. Add a sub-test asserting that the search "result count" header in `internal/ui/search/search.go` reports the same number that appears in the rendered rows — catches a count-vs-rows mismatch class.

## Acceptance Criteria

- [ ] Test passes on fixture for all queries
- [ ] Test passes on `this_repo` and `ai_config_manager` (env-gated) for the fixed query matrix
- [ ] When bwb's search ever diverges from bd (text matching, ordering, completeness), test fails with the specific query and the first divergent ID
- [ ] Header count vs rendered rows mismatch is detected as a separate failure mode
