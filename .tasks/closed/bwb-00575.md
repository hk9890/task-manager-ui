## Description

The composer applies its own sort to active columns and trusts backend order for Done. We need explicit assertions that the rendered order matches what `bd` returns when invoked with the same sort flags â€” across real datasets where many issues share priority/timestamps and exercise tie-breaking.

## Instructions

1. Create `internal/dashboard/parity/sort_parity_integration_test.go` (build tag `integration`).
2. For each available dataset and each column:
   - **Ready**: assert `Columns.Ready.Issues` IDs match `bd ready --json` IDs in order. (Composer re-sorts; assert the re-sort matches bd's contract of `priority asc, updated_at desc, id asc`.)
   - **Not Ready (blocked)**: assert IDs match `bd blocked --json` IDs in order after composer's re-sort.
   - **In Progress**: assert IDs match `bd list --status in_progress --json` IDs in order after composer's re-sort.
   - **Done**: assert IDs match `bd list --status closed --sort-by closed_at --sort-order desc --json` IDs in order, WITHOUT composer re-sort (composer must preserve backend order for Done).
3. Where a tie exists, log both sides' tie-broken tail; do not flake.
4. Add a focused sub-test `t.Run("ClosedAtDescPreservedOnRealData", ...)` that catches the spurious "backend sort assumption broken" warning when many closed issues share `updated_at`. Currently the composer fires a false positive â€” assert that on real data the warning's *cause* (broken ordering) does NOT actually hold. Expected to expose [[bug-sort-proxy-false-positive]].
