What I did:
  - Built bwb (commit ccdf0c4); set up the embedded fixture.
  - Opened TUI with --repo beads + BD_NON_INTERACTIVE=1 against the fixture.
  - Pressed ctrl+space to enter search; typed 'task'; pressed Enter; waited up to 15 s.
  - Repeated with explicit '/' to focus query, with three Enter presses, with query 'bwf', and with single-character queries.

Expected (source: docs/user-guide/key-bindings.md § Search Mode):
  > Typing in the query field edits a draft query; press Enter to run the search.
  > `enter` (query field focused) — submit the draft query and run the search

  So for query 'task' the Results pane should show 3 results matching `bd search task` (bwf-1, bwf-5, bwf-4) and the Results header should converge to '3 results' (or 3 exact), with the stale marker cleared.

Actual:
  - Input shows 'task' tagged 'draft'. Top header says 'Search: 5 results · Selected: bwf-2 (blocked) · Idle'. Results pane shows 5 rows: bwf-2, bwf-1, bwf-5, bwf-4, bwf-3 — that is, every issue in the fixture INCLUDING the closed bwf-3.
  - The overlay banner 'Results below are stale. Press Enter to search ...' is shown above the results.
  - Pressing Enter again (and again and again) does not change the result count, does not clear the stale banner, and does not narrow the results to 'task' matches.
  - Waiting 15 s after Enter doesn't help.

  Same with query 'bwf':
  - bd search 'bwf' returns 4 (closed bwf-3 excluded). UI shows 5 (bwf-3 included).

  So the displayed result set on a typed-then-Enter search is consistently the entire fixture filtered with --status all and *ignoring the text query*. That matches `bd search 't' --status all` = 5; the result set never advances past the live-preview triggered by the first keystroke.

Severity: P1
  - Search is one of two primary navigation modes (per docs/OVERVIEW.md). When the user types a query they cannot trust the rows the UI shows; the count and members of the result set silently fail to reflect the typed query.
  - The 'press Enter to search' overlay text directly invites the action that this bug renders useless — the user is told to do exactly the thing that does not work.
  - Closed issues leak into all searches (the surfaced 5th row was bwf-3, which has status=closed); bd default excludes these. Operators rely on the default-exclude-closed convention.
  - Distinct from znri.6 (which describes the *labels* being stuck while results were correct for the test query). The 'bwf' query znri.6 used would have masked this bug because 'bwf' is a substring of all 5 issue IDs and the count 'happened' to align — but only if the closed bwf-3 was also returned, which means znri.6's observation was actually also wrong (just compatible-with-correct in that one case).

Where to look:
  - internal/mode/search/model.go — the Enter handler (when query field is focused) should submit the draft. Verify it does, and that the submit-handler call site actually reaches the search method, given the focus state observable on the capture.
  - internal/mode/search/parity passes `TestSearchParityFixture/fixture/text=task` etc., meaning the repository-level search method correctly returns 3 for 'task'. So the bug is above the repository — in the controller wiring or the auto-search vs explicit-submit logic.
  - There may also be a defaulting bug where the live preview uses --status all (returning the closed bwf-3) while bd's default excludes closed. Worth confirming alignment regardless of the Enter bug.

Acceptance Criteria:
  - After typing 'task' + Enter on the embedded fixture, the Results header reads '3 results' (or equivalent), the rows shown are exactly bwf-1, bwf-5, bwf-4 in bd's order, the 'stale' overlay is cleared, and the input loses the 'draft' tag.
  - The closed bwf-3 does NOT appear in any default-status search.
  - A teatest- or PTY-based regression covers 'type + Enter changes the result set to match bd search'.
