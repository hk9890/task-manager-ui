## Description

`internal/repository/beads/lean_payloads.go:447` (`leanDependentsFromPayload`) classifies every non-`related` dependent into the Blocks bucket. The forward-direction helper `leanDepsFromPayload` (~line 422-432) explicitly handles `parent-child` by routing it to `hasParent`/parent-id resolution; the reverse-direction `leanDependentsFromPayload` has no such case. Result: when opening an epic P that has children c1, c2 (bd dependents with `dependency_type=parent-child`), the children appear in `Issue.Blocks` and the UI's "Blocked by" / "Blocks" panel shows the children as if the epic blocked them.

## Instructions

1. In `leanDependentsFromPayload`, add a case for `dependency_type == "parent-child"`. Children of the current issue should populate a separate Children list (parallel to how leanDepsFromPayload handles parent). Or, if the dependents endpoint can also report parent-of-X (X being depended-on by parent), handle both directions.
2. Verify against actual bd output for an epic with children — inspect a real `bd dependents <epic-id>` payload during development to confirm the `dependency_type` and direction semantics.
3. Make sure callers of leanDependentsFromPayload populate `Issue.ChildrenIDs` (or whatever the projected list is) instead of `Issue.Blocks`.

## Files to Modify

- `internal/repository/beads/lean_payloads.go` (~line 447)
- `internal/repository/beads/lean_reads.go` (call sites that consume the helper's output)
