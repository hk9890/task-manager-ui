## Description

Build a new, lean `beads.Repository` that implements `repository.Repository` (the 9-method interface) directly on `*bdrunner.CommandRunner`. No `BeadsGateway` interface, no intermediate types, no copy-paste from `read_gateway.go`. Coexists with the legacy gateway-backed code so the build stays green; the wiring switch happens in E4.3.

## Principles

- **Reimplement, do not copy.** Read `read_gateway.go` as specification (which bd argv, which JSON fields actually feed the public domain types). Write the new code fresh.
- **Parse only what `repository.Repository` returns.** If bd ships 20 fields and the return type uses 7, the JSON struct has 7. No "complete" DTOs.
- **No public types beyond `Repository`.** Helper types and JSON DTOs are package-private.
- **No runtime contract checks inlined.** Parse errors are the only validation in this task. The validating decorator is a separate concern (E4.2).
- **Dashboard fan-out lives privately.** No public `ReadyIssues`/`BlockedIssues`/`Catalogs` methods — Dashboard composes them internally.
- **Parent-sibling cache stays.** Move it from the gateway to a private field on `*Repository`.

## Instructions

1. Create new file(s) under `internal/repository/beads/` for the lean implementation (e.g. `lean.go` + `reads.go` + `writes.go` + `dashboard.go` — pick the split that reads cleanly). The new constructor signature is `func New(runner *bdrunner.CommandRunner) *Repository`.
2. For each of the 9 `repository.Repository` methods (`Dashboard`, `Issue`, `Search`, `CreateIssue`, `UpdateIssue`, `CloseIssue`, `AddComment`, `HealthCheck`, `Catalogs`), implement: argv build → `runner.Run(ctx, argv)` → minimal JSON parse → domain type → return. Each method may use private helpers.
3. Move the parent-sibling cache from the legacy `Gateway` struct to a private field on the new `*Repository`.
4. Leave the legacy gateway-backed `Repository` (`gw BeadsGateway` field) and constructor in place under a different name (e.g. `newLegacy`) so callers in `cmd/*` and tests still compile. Wiring switch happens in E4.3.
5. Update `internal/repository/contract/parity_integration_test.go` so the memory↔beads parity suite can be run against the **new** lean Repository as well. Both Repositories pass identical assertions.
6. Add bd-integration smoke coverage: at minimum, the new Repository's `Dashboard`, `Issue`, and `Search` paths exercised against real bd with a known fixture.

## Acceptance Criteria

- [ ] `go build ./...` green with both lean and legacy Repository constructors compiling
- [ ] New lean Repository passes parity tests against `memory.Repository`
- [ ] bd-integration smoke green against real bd
- [ ] `internal/repository/beads/` has no new exported types beyond `Repository` and its constructor
- [ ] The new lean code does not import any DTOs or helpers from `read_payloads.go` or `read_gateway.go`
