## Description

The dashboard's Done column shows the size of a capped slice as the total (e.g. "50" or "56+") when the real DB count is 452. No automated test catches this because the contract test runs against a 3-issue fixture and `internal/dashboard/composer.go:163` encodes `Total = len(Closed)` as correct. A doubled-column-header visual bug also shipped because no test exercises a cold-start → WindowSizeMsg → re-render sequence as a single `View()` output.

This epic builds the test infrastructure to catch this class of bug — counts, sorting, render correctness — across multiple real-scale datasets, plus a manual smoke tool to run before every release. Datasets exercised: embedded fixture (3 issues), this repo's `.beads` (452 closed), and `/home/hans/dev/github/ai-config-manager`'s `.beads` (622 closed, 2 open). External DBs are read-only (`bd --readonly`); we never write to them.

## Success Criteria

- [ ] Automated parity test asserts dashboard column totals match `bd count --by-status --json` on every supported DB
- [ ] Automated parity test asserts dashboard row order matches `bd list/ready/blocked` order with identical sort flags
- [ ] Automated parity test asserts search result count and order match `bd search`
- [ ] Manual smoke binary `bwb-smoke` runs against fixture / this repo / external repo (read-only) and prints pass/fail comparison report
- [ ] Render regression harness fails when two consecutive `View()` outputs stack column headers (catches doubled-header class)
- [ ] Docs cover the smoke workflow for release verification
- [ ] Acceptance review task complete

## Out of Scope

- Fixing the 3 already-identified defects (filed as separate bugs and linked)
- Search ranking quality (only result count + order parity, not relevance)
- Performance benchmarking
