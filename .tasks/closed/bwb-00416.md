## Description
Add the runtime config loader for BWB and include editor/launcher overrides in the same task so the app can read `bwb/config.yaml`, merge it with defaults, and start with a fully resolved runtime config.

## Instructions
1. Add config loading code under `internal/config` that resolves the config path with `os.UserConfigDir()` and reads `bwb/config.yaml` when it exists.
2. Keep `config.Default()` as the source of built-in defaults, then merge file-backed overrides on top of those defaults.
3. Use `gopkg.in/yaml.v3` for YAML decoding.
4. Treat explicit config values as higher precedence than environment fallbacks; for example `editor.command` from YAML wins, while `$EDITOR` only fills the default when the config omits that field.
5. Merge launcher definitions by `action` so built-in actions (`editor`, `nvim`, `opencode`, `shell-command`) remain available unless explicitly overridden.
6. Ignore unknown YAML keys, but collect and surface clear warnings during startup (for example via stderr/log output from `cmd/bwb/main.go`).
7. Return descriptive errors for unreadable existing config files, invalid YAML, invalid config values, and any validation failures produced by the resolved config.
8. Update `cmd/bwb/main.go` to load runtime config through the new loader instead of calling `config.Default()` directly, and print any startup warnings before launching the UI.
9. Add unit tests for missing config, valid config, invalid YAML, unreadable existing config, warning generation for unknown keys, precedence behavior, and launcher merge behavior.
10. Update docs for config location, precedence, warning behavior, and launcher/editor examples.

## Files to Modify
- `cmd/bwb/main.go` — load runtime config via the new loader and surface warnings
- `internal/config/model.go` — keep defaults and merge-friendly model definitions
- `internal/config/doc.go` — document package role if needed
- `internal/config/*.go` — add loader, parsing, warning collection, merge, validation, and tests
- `docs/CODING.md` — document config path, precedence, warnings, and load behavior
- `README.md` — add user-facing config examples if appropriate
- `go.mod` / `go.sum` — add YAML dependency

## Acceptance Criteria
- [ ] BWB resolves its config path via `os.UserConfigDir()` and looks for `bwb/config.yaml`.
- [ ] Missing config file keeps current defaults without error.
- [ ] `editor.command` from YAML overrides `$EDITOR`, while omitted values still fall back to the environment/defaults.
- [ ] Built-in launcher actions remain available by default, and config overrides merge by launcher `action`.
- [ ] Unknown YAML keys produce visible warnings without aborting startup.
- [ ] Invalid YAML or unreadable existing config returns a clear startup error.
- [ ] Automated tests cover load path, missing file, parse failure, unreadable file, warnings, precedence, and merge behavior.
