## Argv shapes in scope (from ARGV_CONTRACT.md — 4 routing paths)

1. Empty text + no WorkState filter → `bd list --json --all [--limit N]`
2. Empty text + status filter → `bd list --json --status <csv> [--limit N]`
3. WorkState=Ready → `bd ready --json` + in-memory filter
4. WorkState=Blocked → `bd blocked --json` + in-memory filter
5. Non-empty text (or text + WorkState=Any) → `bd search <text> --json --status <token> [--type <csv>] [--priority-min N] [--priority-max N] [--assignee <name>] [--label <l>]... [--limit N]`

## Contract restated (key clauses — see interface.go for full text)

**Routing strategy:** determined by `query.Text` emptiness and `query.WorkState` value (see argv shapes above).

**Preconditions:**
- ctx must be non-nil.
- If `PriorityMin` AND `PriorityMax` both set, `Min <= Max` → otherwise `ErrorCodeValidationFailed`.

**Postconditions:**
- `result.Results` never nil; may be empty.
- `result.Metadata.Source` indicates which bd command was used.
- `result.Metadata.Completeness`:
  - `MaybeMore` when `bd search` is used with limit (bd search default cap = 50 unless `--limit` passed)
  - `Partial` when returned count < requested limit or no limit requested
- `result.Metadata.Notice` set to no-text-filter notice when WorkState=Ready/Blocked and no text.

**Error semantics:**
- `ErrorCodeValidationFailed` for `PriorityMin > PriorityMax`.
- `ErrorCodeCommandFailed` for non-zero bd exit.
- `ErrorCodeDecodeFailed` for unparseable stdout.

**bd quirks observed:**
- 2026-05-16: `bd search` requires non-empty query arg (exits non-zero with `{"error":"search query is required"}`). Gateway CARVES OUT via list-routing for empty text.
- 2026-05-16: `bd search --status` accepts SINGLE token only; multi-value comma-separated returns empty. Disposition: INERT (4fgn 2026-05-16 closed not-a-bug; UI doesn't expose multi-status+text). Gateway passes `--status all` when Statuses empty and routes through search.
- 2026-05-16: `bd search --json` without `--limit` defaults to 50-result cap (backend default, not gateway-applied). Disposition: ACCEPT — reflected via `Completeness=MaybeMore`.
