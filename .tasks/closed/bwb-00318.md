**Discovered while landing vtvb.9.** `bd` 1.0.4 (the version pinned in `.mise.toml`) does not support the `--offset` flag. The integration test in `parity_integration_test.go::PaginatedClosedFetch` works around this by intercepting via `repobeads.WithCommandHook` and emulating server-side pagination — but the production `beads.Repository.queryClosedPage` (commit 95aa728) emits `bd query --status=closed --json -a --sort closed --limit N --offset M` directly. Against a real `bd` 1.0.4 binary, this argv is rejected and the load-more flow planned in vtvb.6 will fail at runtime.

This is structurally identical to the existing bd 1.0.4 CloseIssue idempotency workaround (CHANGELOG v0.5.1: "Repository emulates `CloseIssue` idempotency over a `bd` 1.0.4 close-lookup bug").

## Fix

In `internal/repository/beads/lean_reads.go::queryClosedPage`, when `offset > 0`:
- Emit `bd query --status=closed --json -a --sort closed --limit (offset+limit)` — i.e., over-fetch the full window from the start.
- In-memory, slice the returned summaries `[offset:offset+limit]`.
- Do NOT emit `--offset M` to bd.

When `offset == 0`, the argv stays exactly as it is today (no change).

The existing `lean_reads_argv_test.go::TestDashboardClosedOffsetArgv` needs to update the `offset=35` case: expected argv becomes `["query", "status=closed", "--json", "-a", "--sort", "closed", "--limit", "85"]` (no --offset), and the test should assert the in-memory slice math separately (caller still receives 50 items starting at the 36th).

Update `internal/bd/ARGV_CONTRACT.md` to reflect the over-fetch workaround and reference the bd 1.0.4 limitation.

The integration test workaround in `parity_integration_test.go::PaginatedClosedFetch` (vtvb.9) can stay — it tests the production path indirectly. Add a comment noting the relationship: the prod impl now does over-fetch; the hook in the test does the same thing externally.

## Acceptance criteria

- [ ] `queryClosedPage` over-fetches and slices when `offset > 0`; argv contains only `--limit (offset+limit)`, no `--offset`.
- [ ] `TestDashboardClosedOffsetArgv` updated for the new argv.
- [ ] A new test asserts the returned slice has `len == limit` and starts at the `(offset+1)`-th issue in ClosedAt DESC order.
- [ ] `mise run quality:fast` green.
- [ ] `mise run test:integration` green (the PaginatedClosedFetch scenario still passes; the hook can be simplified or removed if redundant — leave it as-is if simpler).
- [ ] ARGV_CONTRACT.md updated.

## When bd ships --offset

Add a `// TODO(bd-upstream)` note in queryClosedPage. When bd's upstream adds `--offset`, the over-fetch can be removed and the argv switched back. The argv-pinning test will fail and force the update — this is the desired tripwire.
