## Description

`internal/launcher/process_runner_integration_test.go::TestExecProcessRunnerReapsChildren` calls `time.Sleep(2 * time.Second)` to wait for the reaper goroutine to call `Wait()` on each child. On a normally-loaded machine the children exit in <10ms. The 2s margin was chosen for CI safety. There's a better way: synchronize on the reaper actually completing.

## Why

Small win (2s) but it's a known pattern smell and easy to fix. Worth doing while we're cleaning up the test suite.

## Instructions

1. Inspect `internal/launcher/process_runner.go` to find the reaper pattern (likely `go func() { _ = cmd.Wait() }()`).
2. Add a test-only hook so the reaper can signal completion. Options (pick simplest):
   - Expose a package-private `reaperWG *sync.WaitGroup` initialized only in test builds, or
   - Refactor the reaper to write to a buffered channel that production code ignores but the test can drain.
3. Update `TestExecProcessRunnerReapsChildren` to wait on the sync primitive with a generous timeout (e.g. 5s as a safety net) instead of `time.Sleep(2*time.Second)`.
4. Confirm the test runs in well under 100ms on a quiet machine.

## Files to Modify

- `internal/launcher/process_runner.go` — add minimal sync hook
- `internal/launcher/process_runner_integration_test.go` — replace sleep
