## Description
Replace ambiguous search result-count semantics with an explicit contract that can represent exact versus possibly capped results across `bd search`, empty-text list fallback, and ready/blocked fallback paths.

## Instructions
1. Update the search domain result model so it can represent returned-count, requested-limit, completeness, source, and any operator-facing notice without overloading `Total` as an exact total count.
2. Adjust gateway search implementation to populate the new metadata consistently for `bd search`, empty-text `bd list`, and the ready/blocked in-memory filtering paths.
3. Preserve honest semantics: when the backend cannot prove completeness, expose `maybe_more`/partial semantics instead of pretending the count is exact; when the gateway fully filters an in-memory set, mark the result exact.
4. Update fake gateway support and unit tests to cover the new contract and the path-specific completeness behavior.

## Acceptance Criteria
- [ ] Search result data no longer overloads `Total` as an exact total count.
- [ ] The gateway distinguishes exact versus maybe-more/capped semantics for search/list/ready/blocked paths.
- [ ] Fake gateway support and targeted tests compile and cover the new contract.
- [ ] No UI layer is forced to infer completeness from `len(results)` alone.
