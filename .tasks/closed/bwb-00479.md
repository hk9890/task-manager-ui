What I did: Verified the default launcher config (bwb --print-config) and the launcher interpolation code against the shell-launcher security rule in docs/CODING.md (Core Architectural Rule 6).

Expected (source: docs/CODING.md rule 6): Issue fields (title, assignee, labels) are 'operator-untrusted input'; they must not be embedded where they can be re-parsed as code. The repo already enforces this for the sh launcher (positional args $0-$3, fixed in beads-workbench-db0z.1).

Actual: The DEFAULT 'nvim' launcher definition (internal/config/model.go defaultLauncherDefinitions) ships this arg:
  +call append(0, ["Issue: {{issue.id}}", "Title: {{issue.title}}", "Assignee: {{issue.assignee}}", "Labels: {{issue.labels}}"])
This whole string is passed to nvim as a '+{command}' argument, which nvim executes as an Ex command (vimscript). {{issue.title}} (and assignee/labels) are interpolated UNESCAPED: internal/launcher/service.go:113-125 Interpolate() does strings.ReplaceAll with only stripC0() applied — stripC0 removes C0 control chars but does NOT escape double-quotes, brackets, or pipes.

Consequence: an issue title such as
  x"]) | call writefile(['pwned'],'/tmp/pwned') | "
closes the vimscript string literal and the append() call, then chains an arbitrary Ex command. Pressing 'n' on that issue in detail mode runs attacker-chosen code. This is the same code-execution class as the P0 shell-injection bug db0z.1 — that fix correctly hardened the 'sh -lc' launcher but the nvim '+call' vector was not covered, because rule 6's wording is scoped specifically to 'sh -c'/'sh -lc'.

Threat path: attacker controls an issue title in a beads DB the operator browses (beads trackers are often shared) -> operator opens the issue -> operator presses 'n' (nvim is the default launcher tool). Per the project's own stated threat model ('issue fields ... are operator-untrusted input') this is in scope.

Severity: medium. Arbitrary code execution, but multi-step and dependent on nvim being the configured launcher (it is the default) and the operator pressing 'n'. Not exploited live during exploration — confirmed by code inspection only (unescaped interpolation + default template).

Repro (inspection): bwb --print-config -> see launcher.definitions[action: nvim].args; internal/launcher/service.go:113 Interpolate() -> only stripC0, no metacharacter escaping.

Suggested fix options: (a) generalize rule 6 beyond sh — forbid interpolating untrusted fields into ANY argument that a launched tool re-parses as code (nvim Ex commands, awk, etc.); (b) change the default nvim launcher to write issue context to a temp file and open it, or pass fields via env vars / -c with a non-executing mechanism; (c) add a vimscript-safe escaper for the nvim template. Also update docs/CODING.md rule 6 to state the principle (untrusted fields never embedded in re-parsed code) rather than only the sh-specific instance.
