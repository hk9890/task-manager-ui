Add a measurement harness that exercises the bwb data layer against a generated workload and emits per-operation timing statistics.

## Operations to measure (at minimum)

- `Repository.Dashboard` cold + warm
- `Repository.SearchIssues` across the matrix in `internal/mode/search/parity`
- `Repository.GetIssueDetail` for representative issues
- Caching backend: cold hydrate, hot read, force-fresh

## Output

- JSON report: per operation, sample size, p50 / p95 / p99 / max in ms, plus repo characteristics (counts by status, etc.).
- Human-readable summary table emitted to stdout when the mise task runs.

## Done when

- Each measured operation is run N times (configurable, with a sensible default).
- Statistics are computed across the samples and emitted both as JSON and as a human table.
- Output is deterministic in shape (field names, ordering) so a future epic can diff reports for regression detection.

## Out of scope

- Pass/fail thresholds on the numbers (a follow-up epic could add CI gating).
- Bubble Tea / TUI rendering benchmarks.
