## What

Programmatic runtime verification of the refresh-concurrency fix using `scripts/capture_bwb_screen.py` + pyte to drive the built `bwb` binary against the embedded fixture. Per user decision (2026-05-17), do NOT defer to manual operator verification — drive the TUI programmatically and capture evidence.

## Why

Per docs/TESTING.md "Full-App Verification" section, layout/navigation/runtime UI changes require built-binary proof. The bug being fixed manifests as visible UI breakage, so the proof of fix is visible UI stability under load. Programmatic capture via pyte gives reproducible CI-grade evidence; pure manual operator verification leaves no artifact in the PR.

## Approach

1. **Inspect existing harness.** Read `scripts/capture_bwb_screen.py` to learn its PTY/pyte capture API, readiness signals, timeout policy, and cleanup behavior (per docs/TESTING.md process-level capture policy lines 142-159).

2. **Author a new capture scenario** (likely `scripts/verify_refresh_concurrency.py` or extend the existing script with a CLI subcommand) that:
   - Builds `bwb` (or uses pre-built binary path).
   - Spawns it via PTY against an embedded-fixture temp repo, with `--debug` flag and `BD_NON_INTERACTIVE=1`, redirecting stderr/debug log to a known path.
   - Waits for a readiness signal (board rendered, e.g., screen contains "Ready" column header).
   - Sends 20 'r' keypresses rapidly (e.g., 50ms apart).
   - Waits ~500ms for any in-flight refreshes to settle.
   - Captures final pyte screen state.
   - Reads the captured debug log.
   - Sends Ctrl+C / quit to terminate cleanly.

3. **Assertions** (script exits non-zero if any fail):
   - Final screen contains all 4 column headers (UI not corrupted).
   - Final screen contains at least one issue from the fixture (data still rendered).
   - Debug log contains AT LEAST ONE "manual board refresh suppressed" entry (proves the guard fired).
   - Debug log contains NO panic stack traces.
   - Debug log shows pendingResults never went negative (look for any error log mentioning pendingResults).

4. **Repeat for search mode** (Ctrl+Space to switch, type a query, Enter, then hammer 'r' 20×, capture, assert "manual search refresh suppressed" entries present).

5. **Hard timeout**: entire scenario must complete within 10 seconds or fail explicitly with "harness timeout" (per docs/TESTING.md line 154).

6. **Cleanup**: guaranteed child-process termination on success, timeout, AND failure (per docs/TESTING.md line 155).

## Deliverables

- `scripts/verify_refresh_concurrency.py` (or equivalent integration of new logic into existing script — pick whichever fits the existing harness shape better).
- A `mise` task entry: `mise run verify:refresh` (add to .mise.toml or equivalent).
- Evidence captured into bd comments:
  - Final pyte screen capture (board + search).
  - Debug log excerpt showing suppressed-refresh entries.
  - Exit code 0 from the harness.
