## Description
Add automated guardrails that enforce the standalone architecture and quality gates during implementation. The current plan repeatedly says "no SQL, no BQL, no orchestration," but those constraints need build-time enforcement so donor transplants cannot silently reintroduce forbidden dependencies.

## Instructions
1. Add automated checks that fail the build or test run when code imports forbidden legacy paths or forbidden dependency classes.
2. Enforce these architectural boundaries at minimum:
   - no direct SQL in the active product path
   - no `internal/bql` dependency in the standalone app
   - no orchestration/control-plane subsystem in the active product path
3. Add standard quality gates for `go build ./cmd/bwb` and `go vet ./...` so acceptance reviews do not rely on implied checks.
4. Document the enforced architectural boundaries and quality gates in `docs/CODING.md`.
5. Keep the enforcement lightweight and automated so agents can run it in CI-like local verification.

## Files to Modify
- project-local build/test verification scripts or Go-based checks (create)
- `docs/CODING.md` (update)
- `*_test.go` or verification helpers as needed

## Acceptance Criteria
- [ ] Automated checks fail if forbidden SQL/BQL/orchestration dependencies are introduced
- [ ] `go build ./cmd/bwb` is part of the standard quality gate
- [ ] `go vet ./...` is part of the standard quality gate
- [ ] `docs/CODING.md` documents the enforced architecture guardrails and quality gates
