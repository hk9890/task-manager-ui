## Description

Two changes in one task because they share the details renderer and would conflict in parallel:

1. **Rewrite `renderColdStartSkeleton`** to render the full 3-pane layout (Dependencies | Content | Metadata) from frame 1 via the existing `renderResponsiveLayout` / `renderThreePane` paths, fed by a synthetic skeleton `domain.IssueDetail`. Eliminates the visible layout jump on data arrival.

2. **Rework `detailLoadingStub` in `internal/ui/search/search.go`** to render a skeleton inside the search detail-preview pane instead of stuffing `loading.View()` text through the markdown renderer. Delete `loading.View()` and the `⏳` hourglass once this caller is gone.

## Why

**For (1):** Today `renderColdStartSkeleton` returns a narrow strip: `targetID + blank + "Description" + 6 skeleton rows`. When data lands the renderer flips to the 3-pane layout — frame and pane boundaries shift. That's exactly the layout instability skeletons are meant to prevent.

**For (2):** `detailLoadingStub` injects styled ANSI bytes via `IssueDetail.Description` → `markdown.Renderer.RenderReadOnly` (`internal/ui/details/details.go:514`). At best the markdown renderer re-wraps and strips the styling; at worst it corrupts the placeholder glyph widths. T1 cannot fix this without entangling search-detail and details-pane rework, so it's grouped here.

## Instructions

### Part A: Details cold-start full-layout

1. In `internal/ui/details/details.go`:
   - Add helper `skeletonDetail(targetID string) domain.IssueDetail`:
     - `Summary.ID = targetID`, `Summary.Title = ""`, `Summary.Status = ""`, `Summary.Priority = -1`, `Summary.Type = ""`.
     - `Description = ""`.
     - Empty `BlockedBy`, `Blocks`, `Related`, `Comments`, `ParentGroupBrowser`.
   - Rewrite body of `renderColdStartSkeleton(targetID, width int) string` (preserve signature):
     - Build a `State` with `Loading: true`, `Detail: skeletonDetail(targetID)`, `Width: width`, `Height: defaultDetailHeight` (or rework signature to accept height — check call site in `Render()` to pick the cleaner path).
     - Dispatch to `renderResponsiveLayout` or `renderThreePane` per `usesResponsiveDetailLayout(width)`.

2. Branch inside `renderContentPaneLines`:
   - Add helper `isSkeletonState(detail domain.IssueDetail) bool`: returns true when `Summary.Title == "" && Description == ""` (the skeletonDetail shape).
   - When `isSkeletonState(detail)`, return N rows of `issuerow.RenderCompactSkeleton({Width: width, Seed: i, Styled: true})` in place of the description body. Dependencies pane and Metadata pane keep their existing `(none)` / placeholder-dash fallbacks — they render correct empty frames naturally.

### Part B: Search detail-preview skeleton

3. In `internal/ui/search/search.go`:
   - Rewrite `detailLoadingStub(summary domain.IssueSummary) domain.IssueDetail` to return a detail with `Summary: summary` and `Description: ""` (no `loading.View()` injection).
   - The `selectedDetailForRender` path then feeds this into the details renderer (via `uidetails.RenderContentPane` at `search.go:116,157`). Because the skeleton-state branch added in Part A step 2 also fires when `Summary.Title == "" && Description == ""`, the search detail preview will render the same Content-pane skeleton automatically.
   - **Caveat:** the search detail preview has a non-empty `summary` (id/title/status/type/priority known from the result list). So `isSkeletonState` keyed on `Summary.Title == ""` won't match — the title is populated. Use a different seam: add a `Skeleton bool` field to `details.State` (NOT to `IssueDetail`), set by both `renderColdStartSkeleton` (it builds the State) and the search renderer (`search.go:116,157` calls take a State-equivalent — pass `Skeleton: state.DetailLoading || state.SelectedDetail.Summary.ID != state.SelectedID`). Branch `renderContentPaneLines` on `State.Skeleton` (signature change).

4. Delete `loading.View()` from `internal/ui/loading/loading.go`. Keep `Summary`, `Glyph`, `NextFrame`, `SpinnerTickCmd`, `TickMsg`, `SpinnerFrames`, `Scope`, `State`.

5. Update `internal/ui/loading/loading_test.go`: delete the `View` tests; keep tests for `Summary`, `Glyph`, etc.
