## Description

Fill in the per-method assertion bodies inside `RunReadContract` so the suite actually exercises every read method on `BeadsGateway`. Both the fake and real factories must produce identical observable behavior.

## Why

The contract is what makes the fake load-bearing. Until the assertions are concrete, the suite is structure-only.

## Instructions

For each method, write assertions that hold deterministically against the fixture (`seed.json`: 3 issues + 1 dependency, prefix `bwf`). Replace the `t.Skip` stubs in `RunReadContract`.

Per method:

1. **HealthCheck**: returns nil error.
2. **ListIssues** (no filters): returns all 3 fixture issues, sorted by ID. Each has expected Title/Status/Type/Priority/Assignee/Labels.
3. **ListIssues** (status=open): returns only the open ones.
4. **ReadyIssues**: returns the issues with no blockers.
5. **BlockedIssues**: returns the issue with the seeded blocker, with the correct blocker ID populated.
6. **ReadyExplain**: returns a `ReadyExplainResult` whose `Ready` slice matches `ReadyIssues` output.
7. **ShowIssue** (`bwf-1`): returns the full `IssueDetail` with description, status, etc.
8. **ShowIssue** (nonexistent ID): returns an error (both impls must error consistently — assert the typed error code if present).
9. **SearchIssues** (empty query): returns all fixture issues.
10. **SearchIssues** (term from fixture title): returns the matching subset.
11. **Query** (simple expression like `status:open`): returns the open issues.
12. **CountIssues**: returns total=3 (or matches actual seed count).
13. **StatusCatalog / TypeCatalog / LabelCatalog**: each returns a non-empty slice containing at least one expected value (e.g. `open` status; the labels seeded in `seed.json`).

Use a table-driven structure where each method is one `t.Run` with the factory call inside. Assertions should compare structured fields, not string-formatted output — that's where fake/real drift hides.

If a real-vs-fake mismatch is discovered while writing assertions: **file a bug** with `bd create --type=bug` for the fake to be reconciled, then write the assertion against real bd behavior. Do NOT weaken assertions to make the fake pass — the fake is the thing under test.
