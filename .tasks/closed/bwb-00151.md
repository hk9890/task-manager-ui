## Description
Initial board load feels slow. The board already sets `m.loading = true` and shows "⏳ Loading issues from gateway…" (`internal/ui/loading/loading.go:View`), but users see no indication of how many sections are loading or how far along the load is. This task profiles the load path, identifies where time is spent, and improves the loading feedback to show real progress.

## Instructions

### 1. Profile the startup load path
- Trace the flow: `boardmode.Model.Init()` → `loadDashboardsCmd()` → `dashboardsLoadedMsg` → N × `loadSectionCmd()` (parallel via `tea.Batch`) → `sectionLoadedMsg`
- Add temporary timing instrumentation to `loadSectionCmd` and `loadDashboardsCmd` to measure wall-clock time per `bd` command
- File: `internal/mode/board/model.go` lines 454–488, `internal/gateway/beads/runner.go`
- Key question: are sections loading in parallel or is there serialization? (They should be parallel via `tea.Batch`)
- Key question: what fraction of load time is `bd` CLI process startup overhead vs actual query time?

### 2. Document findings
- Record: typical time from app start to board fully rendered, time per `bd` CLI call, number of parallel calls
- If the bottleneck is per-call `bd` startup overhead, document that as a follow-up optimization candidate

### 3. Add progress-aware loading feedback
- Replace the generic "⏳ Loading issues from gateway…" with a counter: e.g., `⏳ Loading board (2 / 3 sections)…`
- Track `pendingLoads` and total sections in `internal/mode/board/model.go` (fields already exist: `pendingLoads`, `sections`)
- Update `loading.View()` in `internal/ui/loading/loading.go` to accept an optional progress hint, or render the counter directly in the board's `View()` during loading
- The counter should decrement as each `sectionLoadedMsg` arrives (lines 150–173 in `board/model.go`)

## Files to Modify
- `internal/mode/board/model.go` — loading state rendering, `View()` during load
- `internal/ui/loading/loading.go` — optionally extend `State` with progress fields
- Timing instrumentation can be temporary (removed before merge) or kept as debug-level log lines

## Acceptance Criteria
- [ ] Load timing for a typical board is documented in the PR description (or a code comment)
- [ ] During initial load, the board shows "Loading (N / M sections)…" or equivalent progress counter
- [ ] Counter decrements as sections arrive — user can see progress
- [ ] When all sections are loaded, the loading state disappears and the board renders normally
- [ ] No regression to existing loading behavior on refresh or manual reload
