## Description
Expose the normalized parent-group context needed by the issue-details left browser panel so the UI can show the parent issue and sibling child issues without mutating the panel contents while browsing inside the same group.

## Instructions
1. Extend the issue detail read model to carry the parent-group browser context needed by the redesigned issue details view.
2. Use real `bd show --json` parent-child relationship data to derive:
   - the current issue's parent issue when one exists
   - the sibling child issues that share that parent
3. Keep arbitrary related/blocking relationships out of this browser context for v1; the left panel is for parent-group browsing, not general relationship browsing.
4. If the current issue has no parent issue, return an empty browser-group context so the UI can hide or omit the left panel for now.
5. Add regression tests that lock down the observed parent-child mapping behavior and the no-parent fallback behavior.

## Files to Modify
- internal/domain/issues.go
- internal/gateway/beads/read_gateway.go
- internal/gateway/beads/read_gateway_test.go
- gateway fixtures/testdata as needed

## Acceptance Criteria
- [ ] Issue detail exposes parent-group browser context separate from generic related/blocking references
- [ ] Gateway parsing derives parent and sibling issues from real parent-child relationship data
- [ ] No-parent issues produce an empty browser-group context
- [ ] General related/blocking references remain separate from the browser-group model
- [ ] Tests pass: go test ./internal/gateway/beads
