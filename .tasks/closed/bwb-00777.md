## Description
Make the gateway foundation safe enough for UI work to begin by adding focused automated coverage around command building, decoding, failure behavior, and any remaining cross-cutting gateway verification not already required by the implementation tasks.

## Instructions
1. Add focused automated tests for command construction across representative read and write flows.
2. Add decoding tests for representative payloads returned by official `bd` commands.
3. Add failure-case tests for process failures, invalid output, missing fields, and command stderr handling.
4. Use fixtures or test seams where helpful so coverage is deterministic and not tied to fragile shell state.
5. Define the initial test vocabulary for this repo (`unit`, `integration`, `end-to-end`/`smoke`) so later UI work uses consistent terminology.
6. Document any known limitations in the official command surface that affect the gateway design.
7. Create or update `docs/TESTING.md` with the initial gateway test strategy, commands, and test vocabulary.
8. Ensure tests run with standard Go tooling.

## Files to Modify
- `internal/gateway/beads/*_test.go` (create/update)
- `internal/gateway/beads/testdata/` (create fixtures if needed)
- Repo docs or code comments for known gateway limitations (update where appropriate)
- `docs/TESTING.md` (create or update)

## Acceptance Criteria
- [ ] Tests cover representative command building for read and write flows
- [ ] Tests cover representative decoding cases
- [ ] Tests cover representative failure scenarios and error normalization
- [ ] The repo’s initial unit/integration/end-to-end test vocabulary is defined
- [ ] Any known command-surface limitations are documented
- [ ] Gateway-related test harness/fixture guidance is documented in `docs/TESTING.md`
- [ ] `go test ./...` passes for the gateway foundation
- [ ] No new SQL, BQL, or orchestration dependency is introduced by the test strategy
