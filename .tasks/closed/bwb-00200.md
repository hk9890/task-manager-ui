runner.go lines 92-94: CommandRunner.Run holds runMu for the entire duration of executor.Run(). This means the TUI cannot run bd list, bd ready, and bd blocked in parallel — every concurrent gateway call queues behind a single mutex. The runner_test.go TestCommandRunnerRunSerializesConcurrentExecutorCalls test explicitly asserts this behavior as desired, but this is architecturally wrong for a TUI that needs to populate multiple dashboard sections concurrently. If serialization was added to work around a bd CLI limitation (e.g. Dolt lock contention), that should be documented with a comment. Otherwise this mutex should be removed — subprocess isolation already provides process-level safety. Suggested action: either (a) remove the mutex and the concurrency test, or (b) add a code comment explaining the specific bd/Dolt limitation that requires serialization.
