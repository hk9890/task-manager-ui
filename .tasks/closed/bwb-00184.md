## Description
The repository has no durable changelog, so the only record of `v0.1.0` is the GitHub release body. Add a repository-level changelog so future releases have a canonical in-repo history and contributors can see shipped scope without relying on GitHub release pages alone.

## Instructions
1. Create `CHANGELOG.md` at the repository root.
2. Use a simple Keep a Changelog style structure with a top-level title and semver-style release sections.
3. Backfill a `v0.1.0` entry using the already-published GitHub release notes as the source of truth; do not invent features that are not present in the published release body.
4. Group the `v0.1.0` notes into clear sections such as `Added`, `Changed`, and `Fixed` only when supported by the published release content.
5. Update `README.md` to link to `CHANGELOG.md` in the project guidance/docs area so release history is discoverable.
6. Keep the change documentation-only; do not mix unrelated product changes into this task.

## Files to Modify
- `CHANGELOG.md` (create)
- `README.md` (update: add changelog link)

## Acceptance Criteria
- [ ] `CHANGELOG.md` exists at the repository root

- [ ] The entry is structured and readable rather than a raw pasted blob
