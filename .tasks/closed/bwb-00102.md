## Outcome

A first-class load-testing surface for bwb that any contributor (human or agent) can run on demand, with configurable workload shapes, measured response times, and reproducible reports.

## Why

Today the only large-corpus exercise we have is the scale fixture (~590 issues, 75 closed, hardcoded mix) gated behind `BWB_SCALE_FIXTURE=1`. It's used by two parity tests and the bwb-smoke scale test. There is:

- No way to vary the workload (counts per status, dependency density, comment depth, etc.) without editing `scale-seed.json` by hand.
- No instrumentation: the tests assert correctness but never measure how long Dashboard/Search/Detail take.
- No agent-runnable recipe: 'load test bwb on a 10k-issue repo and tell me where the latency cliff is' is not a thing today.
- No baseline / regression detection for performance.

## Outcome shape

1. **Dedicated mise task** — `mise run test:load` (or similar) that runs the load-test suite end-to-end without requiring the operator to remember env vars or flags.
2. **Workload parameters** — caller can configure:
   - Number of issues per status (open / in_progress / blocked / closed)
   - Optional dependency density (avg edges per issue)
   - Optional comment depth
   - Optional priority distribution
   - Random seed (for reproducibility across runs)
3. **Measured operations** — at minimum:
   - `Repository.Dashboard` cold + warm
   - `Repository.SearchIssues` across the matrix in `internal/mode/search/parity`
   - `Repository.GetIssueDetail` for representative issues
   - Caching backend: cold hydrate, hot read, force-fresh
4. **Output** — machine-readable JSON + human-readable summary table with p50/p95/p99 per operation, plus repo characteristics (issue count by status, etc.).
5. **Agent-runnable recipe** — a doc section (likely in docs/TESTING.md or a new docs/LOAD_TESTING.md) that an agent or contributor can follow standalone: 'to load-test bwb on N issues with shape X, run … and read the report at …'.
6. **No silent skip** — the task runs to completion and produces a report. If `bd` isn't on PATH or some other precondition fails, it fails loudly with actionable output, not a skip.

## Success Criteria

- [ ] `mise run test:load` exists and runs end-to-end on a fresh checkout (with `bd` available).
- [ ] Workload parameters are configurable via env vars or task args, not by editing seed JSON.
- [ ] Per-operation timing measurements are captured and reported with at least p50 and p95.
- [ ] An agent can follow the documented recipe and produce a load-test report without prior context from this session.
- [ ] Output is reproducible given the same seed + parameters.
- [ ] Existing scale-fixture tests are either subsumed by this task or co-exist cleanly (no test runs twice).

## Out of scope

- CI integration / performance regression gating (would be a follow-up epic).
- Benchmarking the TUI itself (Bubble Tea render loop, key latency). This is data-layer load testing.
- Real bd CLI benchmarking (we measure the bwb path, not bd internals).

## Notes for future readers

The existing `scale-seed.json` (~590 issues, 75 closed) was hand-crafted to satisfy specific edge-case test requirements (cap engagement, etc.). It is not a load-test corpus. This epic introduces a separate generator path so the existing fixture stays stable for the parity tests that depend on it.
