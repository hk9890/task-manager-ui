## Description

`internal/repository/caching/caching.go:391` (Issue cache-hit path) routes through `memory.Issue` which calls `toDetailLocked`. For Related/Blocks/ParentGroupBrowser cross-references, `toDetailLocked` looks up referenced issues in `r.issues`. If a referenced issue was never seeded (e.g. the user opened A which references B, but B was never opened), the resulting `IssueReference` carries only `{ID: depID}` with empty Title/Status/Type/Priority. The detail view "Blocked by" row then shows a bare ID instead of the human-readable title — appears on every cache hit after the first.

## Instructions

Two acceptable approaches:

A. **Store the projected references at SeedDetail time** instead of reconstructing them from the memory map. SeedDetail receives the full IssueReference list from backing; persist that list verbatim on the storedIssue and return it from toDetailLocked without re-resolving. Cost: extra memory per issue.

B. **Fall back to backing for unresolved cross-refs**: when toDetailLocked finds an unseeded cross-ref ID, return the bare ID but mark the result as partial; caching layer detects partial and refetches from backing. Cost: extra round-trip.

Pick (A) — simpler, lower latency, and matches what bd already returns. The slight extra memory cost is bounded by the number of cross-refs.
