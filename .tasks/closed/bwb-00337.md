## Description

`internal/repository/caching/caching.go:263` (`Hydrate`) releases `c.mu.RLock` before the multi-second `vcStatusFunc` call and `loaded.Dashboard` precompute, then reacquires `c.mu.Lock` to swap state. Any `CreateIssue`/`UpdateIssue`/`CloseIssue`/`AddComment` mutation that happens during this window is overwritten when `c.memory = loaded` executes — the freshly-created issue's cache entry vanishes; next `Issue(newID)` misses and pays a backing round-trip.

## Instructions

Pick one of:

A. **Forbid concurrent writes during Hydrate**: add a `hydrating` bool flag set under c.mu.Lock at Hydrate entry. Write methods check the flag and either wait via a `*sync.Cond` or fail-fast. Document this in the Repository contract for callers.

B. **Merge loaded into existing memory rather than swapping**: only seed entries that don't already exist in `c.memory`; preserve any in-flight mutations. This requires a `memory.MergeFromSeed` helper.

C. **Document a hard precondition**: "Hydrate MUST be called before Start and before any user-facing write." Add a runtime guard: if Hydrate is called after Start (started=true), return an error.

Option C is the simplest correct fix because there's no legitimate use case for late Hydrate. The current code already states this informally in a comment; just enforce it.
