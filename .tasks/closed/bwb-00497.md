## Description

Verify that the write methods on `BeadsGateway` are in sync with real bd 1.0.4 behavior, the FakeBeadsGateway implementation, and the existing test coverage. Writes are special: they hold the `runMu` exclusive lock and have idempotency / side-effect semantics that reads do not.

Part of epic `beads-workbench-yspw`.

## Argv shapes in scope (from ARGV_CONTRACT.md)

- `bd create --json --title <title> [--description <desc>] [--type <type>] [--priority <N>] [--assignee <name>] [--labels <csv>]`
- `bd update <issueID> [--title <t>] [--description <d>] [--status <s>] [--type <t>] [--priority <N>] [--assignee <name>] [--set-labels <csv>]`

- `bd close <issueID> [--reason <reason>]`
- `bd comments add <issueID> <body>`

## Contract restated (key clauses — see interface.go for full text)

### CreateIssue
- App layer pre-validates Title (gateway does NOT).
- Type must be valid bd type or empty.
- Labels elements MUST NOT contain commas (gateway joins with comma).
- On success: `IssueID` non-empty, trimmed.
- NOT idempotent.
- created_by = git user.name (gateway never passes `--actor`).
- 2026-05-17: labels passed via `--labels` NOT in create response JSON. Disposition: ACCEPT — use ShowIssue.

### UpdateIssue
- IssueID must be non-empty → otherwise `ErrorCodeValidationFailed` (no bd call).
- Pointer-to-empty-string clears that field.
- All-nil-fields + ClearLabels=false → bd called with no flags, exits 0 silently with "No updates specified" on stdout, gateway returns nil.
- Idempotent (last call wins).
- 2026-05-17 BUG WORKAROUND: `--set-labels ""` silently fails in bd 1.0.4. Gateway works around by pre-fetching labels via ShowIssue then emitting `bd update --remove-labels <csv>` for each existing label. If no labels exist, bd update is skipped entirely. See [[ubav]].

### CloseIssue
- IssueID must be non-empty → otherwise `ErrorCodeValidationFailed` (no bd call).
- IDEMPOTENT (gateway emulates this over a bd 1.0.4 bug).
- 2026-05-17 BUG WORKAROUND: bd 1.0.4 (upstream `gastownhall/beads#4025`) emits "issue not found" on re-close within same wall-clock second (RowsAffected==0 because DATETIME has second resolution and target state is already present). Gateway catches the close-specific "issue not found" stderr, runs ShowIssue, returns nil iff issue exists with status=closed. Truly missing issues still surface error.
- When no `--reason` supplied, bd stores close_reason="Closed" (literal string).
- BD_NON_INTERACTIVE=1 always set via resolveEnv → no interactive prompts expected.

### AddComment
- IssueID must be non-empty → otherwise `ErrorCodeValidationFailed` (no bd call).
- App layer pre-validates Body (gateway does NOT).
- NOT idempotent (each call appends).
- author = git user.name (gateway never passes `--actor`).
- Comments work on closed issues (bd does not guard).
- Markdown content stored verbatim.
- updated_at NOT advanced by comment addition (separate record).

**Shared:**
- All writes hold `runMu` exclusive (IsWrite=true).
- BD_ACTOR env var is stripped by filterEnvToAllowlist; all writes attributed to git user.name.
- Writes are immediately consistent with subsequent reads (embedded dolt auto-commit).
- Stdout is discarded for Update/Close/AddComment (no `--json`); only Create uses `--json` to extract ID.
