## Problem

Per-surface loading placeholders (board cold-start, search results cold-start, search detail preview, details cold-start) are visually ugly and inconsistent:

1. Placeholder glyph `░` reads as gritty noise rather than a placeholder bar.
2. Rows are static — only the global header braille spinner animates, so the eye sees "dead grey wall + tiny tick in the corner" and can't tell loading from stuck.
3. Slot layout is lumpy — hard-coded 8-char "id" block + giant slab, repeated 6 times identically, looks like a barcode.
4. Three visual languages across surfaces: board/search use 2-slot skeleton rows; details cold-start uses 3-slot rows; search detail preview uses `⏳ Loading details for X…` text+emoji.
5. Details cold-start renders only a narrow strip (targetID + Description + 6 rows). When data lands the renderer flips to the full 3-pane layout, causing a visible layout jump.

## Goal

Per-surface placeholders that:

- Look like real issue rows (same slot recipe via shared internal helpers).
- Pulse in sync with the header spinner so loading state is visibly alive.
- Render in the surface's final layout from the start — no layout shift when data arrives.
- Share one implementation: a new sibling function `RenderCompactSkeleton` in the `issuerow` package, used by every loading surface.

## Design notes (resolved during plan-review pass)

- **Skeleton API shape:** sibling `RenderCompactSkeleton(opts SkeletonOpts) string`, NOT a `Skeleton bool` on `RenderConfig`. Keeps `RenderCompact` data-only; skeleton evolves independently.
- **Slot parity:** real issue rows have variable-width segments (state = 2 or 3 chars, id varies up to `CompactIDWidth(width)`). Skeleton acceptance is "5 distinct ▓ blocks separated by single spaces" + `lipgloss.Width == width`, NOT "identical slot boundaries with real rows".
- **`detailLoadingStub` in search:** moved from T1 to T2 because its current implementation stuffs styled bytes through the markdown renderer; correct fix couples with details cold-start rework.
- **`PlaceholderDetail` in `internal/mode/details/model.go`:** 5th caller of the old skeleton primitive; covered in T1.
- **Animation plumbing:** signature change on mode-level `View()` methods, with phase derived from `loading.SkeletonPhase(m.spinnerFrame)`. No package-level atomic.
- **Light theme:** 3-shade `AdaptiveColor` palette with hand-picked Light + Dark hex values; dark-theme dynamic range will be wider by design. T4 Check 6 verifies on both themes.
