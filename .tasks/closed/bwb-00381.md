## Description
Discovered from an external user report while validating `beads-workbench-2shr` acceptance. Opening the epic detail viewer in a normal real terminal can still render badly even when the screen is not narrow. Comment-heavy history, especially pasted test output/log-like text, can blow out the content pane so the overall detail layout becomes unreadable.

## Expected Behavior
The detail viewer should keep each pane bounded to its allotted width. Long comment bodies and pasted command output should wrap or otherwise stay contained inside the content pane without corrupting neighboring panes.

## Actual Behavior
In a real terminal, the `beads-workbench-2shr` detail view can still look broken at normal widths. The comments section horizontally overflows with long lines from historical comments, making the content pane look absurdly wide and leaving the overall detail layout looking broken even though the previous narrow-width responsive fallback shipped.

## Reproduction
1. Build the latest app: `go build -o /tmp/bwb ./cmd/bwb`.
2. Run the app against this repository: `env BD_NON_INTERACTIVE=1 /tmp/bwb`.
3. Open epic `beads-workbench-2shr` in detail mode.
4. Scroll into the comments section with the historical test-output comment.
5. Observe that the content pane can overflow horizontally and the detail view looks bad in a normal terminal, not just at narrow widths.

## Impact
This is still a user-visible detail-view rendering defect in the epic tracked by `beads-workbench-2shr`. The previously closed narrow-width fix was insufficient; acceptance review should stay blocked until comment-heavy detail rendering is contained and readable in real terminals.

## Instructions
1. Reproduce against the real epic detail view and inspect the comment-rendering path used by `internal/ui/details/details.go`.
2. Investigate why long comment text/log output is escaping the intended pane width even after markdown/plain-text rendering and truncation.
3. Ensure content/comment rendering stays bounded inside the content pane at normal terminal widths.
4. Add or update focused regression coverage for comment-heavy detail rendering so this specific overflow pattern is caught.
5. Re-verify with the built binary against `beads-workbench-2shr` in a real terminal.
