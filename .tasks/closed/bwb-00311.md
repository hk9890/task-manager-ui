catalog.go: StatusOption, TypeOption, and LabelOption are all {Name string, Description string}. Three identical structs for the same shape. This adds zero type safety because the fields are both plain strings — there is no compiler help from having separate types. The gateway StatusCatalog/TypeCatalog/LabelCatalog methods and their consumers would be simpler with a single CatalogOption type. Counter-argument: if these types are expected to diverge (e.g. TypeOption gains an Icon field), separate types are forward-looking. But today they are dead weight. Suggested action: evaluate whether these are expected to diverge. If not, unify into a single CatalogOption type.
