## Description

`internal/mode/search/model.go:150-152` returns nil on `tea.KeyEsc` from `handleKey` unconditionally. `CapturesShellKey` allows `ShellActionEscape` pass-through ONLY when `focus == FocusQuery`, so Esc on Results / Content / Metadata focus is silently swallowed by the search mode and never reaches the shell-level escape handler.

This is a discoverability/affordance bug — operators expect Esc to always be a way out. Major finding U3 from the 2026-05-16 audit.

## Files to Modify

- `internal/mode/search/model.go`
- `internal/app/model_test.go` or new teatest under `internal/app`

## Instructions

1. In `handleKey`, only swallow `tea.KeyEsc` when `focus == FocusQuery` (where Esc has a specific meaning — clear input or unfocus). For all other focus states, return nil from handleKey AND ensure `CapturesShellKey` returns false so the shell-level escape action runs.
2. Verify `CapturesShellKey` actually drives the precedence the way the search mode expects. If the shell calls `CapturesShellKey` before delegating, the fix is just CapturesShellKey-side; if shell delegates first, fix handleKey instead. Confirm by reading `internal/app/model.go` key routing.
3. Add a teatest scenario at app level: enter search mode, move focus to Results, press Esc, assert shell escape behavior fires (typically returns to board or clears mode).

- [ ] Esc with focus on Query has its existing behavior preserved
- [ ] Esc with focus on Results / Content / Metadata reaches shell escape
