## Description

E4.1 created the lean `beads.Repository`. Scenario 10 of the parity-contract suite (`PartialDashboardFailure`) is currently **skipped** for the `beads_lean` variant because there is no way to inject a partial error into one of the Dashboard fan-out branches without a `BeadsGateway` wrapper. Before E4.4 deletes the legacy gateway code, the lean Repository must expose an injection seam so the scenario runs (and passes) against it. Otherwise we lose the test coverage that proves Dashboard gracefully degrades when one of its underlying bd calls fails.

## Instructions

1. Add an optional fault-injection hook to `*beads.Repository` (e.g. a `CommandExecutor` override field, an `options.WithExecutor(...)` constructor option, or a similar testable seam). Production wiring keeps using the default real-runner; tests can swap in an executor that errors for specific argvs.
2. Update the lean implementation's bd-call sites to go through this seam so the override actually takes effect.
3. Re-enable scenario 10 (`PartialDashboardFailure`) in `internal/repository/contract/parity_integration_test.go` for `beads_lean`; remove the skip; assert identical behaviour to the legacy variant.
4. Confirm: scenario 10 PASSES for both `beads_legacy`/`beads_lean` (whatever the names) and for `memory`.

## Acceptance Criteria

- [ ] `*beads.Repository` exposes a test-only seam for executor / command-result override (one constructor option or one settable field — pick the simplest shape)
- [ ] Scenario 10 (`PartialDashboardFailure`) runs against `beads_lean` (no skip) and passes
- [ ] `mise run test:all` green
- [ ] No production code change is required to use the seam — default behaviour is unchanged

## Files to Modify

- `internal/repository/beads/lean.go` (or wherever the runner-call seam lives)
- `internal/repository/beads/lean_reads.go` / `lean_writes.go` if needed
- `internal/repository/contract/parity_integration_test.go` (remove skip, re-enable)
- Possibly a new `internal/repository/beads/options.go`
