## Description

Hands-on audit of real `bd` CLI write behavior (CreateIssue, UpdateIssue, CloseIssue, AddComment), with findings recorded directly in `internal/gateway/beads/interface.go` as per-method contract sections plus an "Observed bd write quirks" appendix. The output of this task IS the canonical write contract — no separate report. Mirrors the 8qw9.1 read-side audit pattern.

## Instructions

1. Set up a writable temp DB for controlled experiments:
   ```bash
   tmp=$(mktemp -d) && cd "$tmp" && BD_NON_INTERACTIVE=1 bd init
   ```
   NEVER touch external DBs (the operator may not have authorized writes there).

2. For each WRITE method on `BeadsGateway`, exercise real bd. Maintain an **audit log file** (`/tmp/9x70-audit.log` or similar) recording each real bd invocation made during the audit — this is the evidence floor for the acceptance review.
   - **CreateIssue**: minimum fields, all fields, missing required fields, duplicate IDs, very long titles, special chars in title/description, with/without parent, with/without labels, with/without assignee.
   - **UpdateIssue**: update each field independently; clear-to-empty; invalid status name; non-existent issue ID; concurrent update (run two writes back-to-back); update parent to non-existent ID.
   - **CloseIssue**: close-then-close (idempotent?); close with/without resolution; close non-existent.
   - **AddComment**: empty body; very long body; non-existent issue ID; comment with special chars/markdown.

3. Document discoveries directly into `interface.go` per the same format as read methods (Preconditions / Postconditions / Side effects / Idempotency / Error semantics / bd quirks observed with disposition). New sections to cover:
   - **Side effects**: what changes in the DB after the call; what becomes visible to subsequent reads (ShowIssue, ListIssues).
   - **Idempotency / collision**: behavior on repeated calls; behavior on ID collisions.
   - **Confirmation prompts**: bd has interactive confirmations for some destructive operations; document which ones bwb's `BD_NON_INTERACTIVE=1` bypasses and which (if any) still prompt.
   - **Actor / audit trail**: bd attributes writes to an actor (env var or git user.name); document how bwb threads this and what bd records.
   - **Dolt consistency**: bd's dolt backend has auto-commit policy (off/on/batch); writes may not be visible to subsequent reads in some modes. Document the consistency model.

4. Read existing gateway write code (`internal/gateway/beads/writes.go`) to reconcile actual implementation against documented contract.

5. Extend the top-of-file audit appendix in `interface.go` with a new "Observed bd write quirks (audit YYYY-MM-DD)" section.

## When the audit reveals a real assumption-vs-reality mismatch

Apply the parent epic's "Discovered-during-audit bug discipline" (same as 8qw9.1). For each discovered mismatch:
1. File a separate bug `bd create --type=bug --title="Gateway write contract violation: <consumer> assumes <X>, real bd does <Y>"`.
2. Bug body MUST require app-level verification.
3. Link back to this task via `bd dep add <bug-id> beads-workbench-9x70.1`.
4. Don't fix in this task — bug fixes happen in their own ticket.

As a subagent, return proposed bug filings to the orchestrator as text.
