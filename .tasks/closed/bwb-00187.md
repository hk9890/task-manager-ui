In `internal/mode/details/model.go`, the `Model` struct exports every field:
- SelectionID, TargetID, Detail, Loading, Error, Keys, ScrollOffset

These are all directly manipulated by the app shell (`internal/app/model.go`) rather than going through methods. This means:
1. Any consumer can put the model into an inconsistent state (e.g. set Loading=true without setting TargetID)
2. The model doesn't own its own state transitions — the app shell does
3. Unlike board and search which are behind the `mode.Controller` interface, details is a raw struct with exported fields

Contrast with board/search: they own their state transitions and emit messages. The details model is essentially a ViewModel with no ownership boundary.

Suggested action: Either add setter methods that enforce consistent state transitions (e.g. `SetTarget(id string)` that handles Loading/Error/TargetID together), or if the current approach is intentional (detail as passive view-model), document that explicitly in the doc comment.
