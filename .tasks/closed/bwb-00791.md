## Description
Implement the visible application shell: layout, top-level mode switching, selected-issue state, loading/error feedback, and shared navigation conventions. This task creates the UI structure that later browsing and editing features will plug into.

## Instructions
1. Implement the shell layout for the main app surface.
2. Add top-level mode switching for the future board and search workflows.
3. Define whether the issue details experience in v1 is a persistent detail pane, a dedicated detail mode, or a hybrid pattern, and implement the shell accordingly.
4. Add selected-issue state and wire board/search selection into the shell’s detail presentation model.
5. Define and implement baseline keybinding/navigation conventions for mode switching, selection, and action entry points.
6. Integrate the shared modal/toast/loading/error primitives from the app-shell epic.
7. Keep the shell independent from legacy orchestration or BQL assumptions.

## Files to Modify
- `internal/app/` (update root app behavior)
- `internal/mode/` (create or update mode controllers/contracts)
- `internal/ui/layout/` (create if needed)
- `internal/ui/` (update shared shell rendering helpers if needed)
- `docs/CODING.md` or in-code architecture comments if documenting the chosen detail presentation model is needed

## Acceptance Criteria
- [ ] The shell renders a stable main layout
- [ ] Users can switch between top-level modes
- [ ] Selected-issue state exists and is visible in the shell
- [ ] The v1 detail presentation model is explicit, implemented consistently, and documented for downstream tasks
- [ ] Baseline keybinding/navigation conventions are implemented and documented in code or docs
- [ ] Loading and error feedback is surfaced through shared UI primitives
