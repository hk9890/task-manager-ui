What I did: launched bwb against the bwb project's own beads DB at a 40-row terminal (the runbook's small-terminal case).

Expected (source: docs/RUNTIME_UI_VERIFICATION.md §5 'Done-column closed-limit: resize-then-refresh', step 4): 'Confirm N is 50 (the floor). If N is not 50, the floor is broken.'

Actual: Done header reads '35 of 736'. N=35, below the documented floor of 50. M=736 matches bd count --status closed and is correct.

Root cause: bug afwe ('Remove min-50 floor in board.closedLimit() — client asks for what the viewport needs', P1, closed) removed the 50-row floor. internal/mode/board/model.go:250 sectionItemCapacity() now returns height-3 with no floor; the function header comment also no longer mentions a floor.

The iwvm runbook predates afwe and was not updated when the floor was removed. Its fail signal 'N ≠ 50' is now the expected behavior, so the documented procedure cannot pass.

Severity: medium — operator-facing runbook gives a false-positive failure signal. Anyone following the procedure today would file a regression that does not exist.

Repro:
1. Build bwb: go build -o /tmp/bwb ./cmd/bwb
2. Open a terminal with fewer than 70 rows (the runbook's small-terminal case).
3. BD_NON_INTERACTIVE=1 /tmp/bwb (against a repo with >50 closed issues).
4. Read the Done column header. Per the runbook, expect '50 of M'. Observe e.g. '35 of M' or '37 of M' depending on exact height.

Fix sketch: update docs/RUNTIME_UI_VERIFICATION.md §5 to reflect that closedLimit() now scales linearly with height (sectionItemCapacity = height - 3, no floor). Rewrite the runbook's pass/fail table accordingly — the resize-up + r flow still demonstrates the variable limit; only the small-terminal floor assertion needs to change.
