## Description
Write down the intended UI component boundaries so future rendering work does not keep reinventing rows, lists, and detail surfaces independently.

## Instructions
1. Review the current rendering packages under `internal/ui/` and summarize the current responsibilities of board, search, details, shared helpers, and `styles.FormSection`.
2. Update the appropriate project-facing doc surface with a concise architecture section that defines these responsibilities:
   - shared compact issue row component
   - optional shared issue list component
   - generic panel/section container
   - dedicated issue detail component
3. Document the decision that `internal/ui/details` is the issue detail component and should stay separate from compact list rendering.
4. Document the rule that a list abstraction should only be introduced when it removes proven duplication above the row level.
5. Include a short dependency diagram or package tree snippet showing how board/search/details should compose shared UI primitives.

## Files to Modify
- `docs/CODING.md` — add or update UI component architecture guidance.
- optionally `project-plan/ARCHITECTURE.md` if a deeper architecture note is needed.

## Acceptance Criteria
- [ ] Documentation explicitly identifies row, list, panel, and detail responsibilities.
- [ ] Documentation states that `internal/ui/details` is the dedicated issue detail component.
- [ ] Documentation states that list abstraction is conditional on proven duplication.
- [ ] The guidance is specific enough that a tasker can implement row/list extraction without guessing boundaries.
