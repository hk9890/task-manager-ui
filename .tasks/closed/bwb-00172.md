Discovered from user review after epic beads-workbench-90w was closed. Expected: content scrolling in detail mode should move predictably with each key press and stop immediately when key input stops. Actual: user reports that when content exceeds the page and they press Down, scrolling feels slow and appears to continue after they stop pressing.

## Investigation First
Code inspection suggests detail scrolling is currently discrete integer-offset scrolling with no animation, so this may be terminal key-repeat event queuing rather than a renderer bug. Verify that hypothesis before changing code.

## Acceptance Criteria
- [ ] Reproduce with controlled single-tap vs key-hold behavior and determine whether extra motion is caused by queued repeat key events, render lag, or a detail-mode state bug.
- [ ] If single discrete taps already behave correctly, capture the finding and decide whether this should be documented or handled differently.
- [ ] If a real detail-mode bug exists, file or convert this issue into a concrete implementation fix with exact reproduction and expected behavior.
