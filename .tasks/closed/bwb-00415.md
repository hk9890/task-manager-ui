## Description
Move editor and launcher runtime settings onto the new file-backed config surface so users can choose which tools BWB launches without changing code.

## Instructions
1. Extend the config model so `editor.command` and launcher definitions can be read from YAML and merged with defaults.
2. Preserve the existing built-in launcher actions (`editor`, `nvim`, `opencode`, `shell-command`) as defaults while allowing config to override their command, args, env, and workdir fields.
3. Ensure the app services layer continues to construct launcher and editor services from the resolved merged config.
4. Add tests that prove launcher and editor settings can be overridden from config while unspecified built-ins remain available.
5. Update user-facing docs with example YAML for editor and launcher customization.

## Files to Modify
- `internal/config/model.go` — runtime config structures and defaults
- `internal/config/*.go` — YAML decoding and launcher merge behavior
- `internal/app/services.go` — consume resolved editor and launcher config
- `internal/app/services_test.go` — verify merged launcher/editor config usage
- `internal/config/model_test.go` and related tests — validate defaults and overrides
- `docs/CODING.md` and/or `README.md` — document editor and launcher config examples

## Acceptance Criteria
- [ ] `editor.command` can be overridden from YAML.
- [ ] Built-in launcher actions remain available by default.
- [ ] A config file can override built-in launcher action definitions without redefining every action.
- [ ] Services use the merged launcher/editor config when constructing runtime dependencies.
- [ ] Tests cover launcher and editor override behavior.
