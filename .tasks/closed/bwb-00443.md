What I did: opened bwb against the seeded fixture, pressed Enter on bwf-2 to view detail.

Expected (source: bd show --json on bwf-2 returns separate fields 'owner: hans.kohlreiter@dynatrace.com' and 'created_by: Hans Kohlreiter'): the Metadata pane field labeled 'Creator' should show the human-readable created_by value (e.g. 'Hans Kohlreiter'), not the owner email.

Actual: Metadata Ownership group shows 'Creator : hans.kohlreiter@d...' — this is the truncated 'owner' email from bd, not 'created_by'.

Root cause: internal/repository/beads/lean_reads.go:168 sets domain.IssueDetail.Creator from primary.Owner (the bd 'owner' field). bd's 'created_by' field is never read into the domain model. internal/ui/details/metadata.go:131-132 then renders that as 'Creator'.

Severity: low/medium — UX/data consistency. A user reading 'Creator' naturally expects the display name. The current behavior conflates 'owner' (account/email) with 'creator' (display name). Tests in internal/ui/details/metadata_test.go fix this assumption with Creator: 'hans' which masks the email-vs-name distinction.

Repro:
1. Build bwb: go build -o /tmp/bwb ./cmd/bwb
2. repoPath=$(mktemp -d) && sh internal/testing/e2e/embeddedfixture/setup.sh "$repoPath" internal/testing/e2e/embeddedfixture/seed.json
3. cd "$repoPath" && BD_NON_INTERACTIVE=1 /tmp/bwb
4. Press Enter on bwf-2. Inspect 'Creator' field. Compare to bd show bwf-2 --json | jq '{owner, created_by}'.

Fix sketch: either (a) rename UI label from 'Creator' to 'Owner' to match bd semantics, or (b) thread bd's 'created_by' into a new domain.IssueDetail.CreatedBy field and render that under 'Creator'.
