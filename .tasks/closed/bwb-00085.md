## Description
Runtime verification found that during startup the board can render visible issue rows while the shell still shows `Selected: none` and without a meaningful selected-issue detail/context preview. This creates a confusing browsing state where content appears selectable before the shell's shared selection context is visibly established.

## Evidence
- Real PTY/runtime verification on startup
- Verifier observed visible issue rows while footer selection context still lagged behind content
- This weakens focus/selection clarity in the main browsing screen

## Instructions
1. Ensure startup board loading establishes shell selection context as soon as visible content becomes selectable.
2. Ensure selected-issue footer/detail-preview state stays synchronized with the first visible/active board selection during startup.
3. Add or update app-level tests that cover startup selection synchronization.
4. Re-check the startup runtime in a PTY after the fix.

## Files to Modify
- `internal/app/model.go`
- `internal/app/model_test.go`
- related board/app integration tests if needed

## Acceptance Criteria
- [ ] Once visible board content is rendered on startup, shell selected-issue context is synchronized with the active board selection
- [ ] The startup browsing screen does not show visible issues while still reporting `Selected: none`
- [ ] Automated tests cover startup selection synchronization
- [ ] Real runtime/PTTY re-check confirms startup selection clarity
