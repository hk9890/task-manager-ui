## Description

`mode/details/Model` intentionally differs from its `mode/board` / `mode/search`
siblings: no constructor (struct-literal init at `app/model.go:250`), no
`Init`/`Update`, communicates via `HandleKey(...) (bool, *OpenRelatedIssueIntent)`
instead of `Update` returning `tea.Cmd`, and `View(maxWidth, viewportHeight int,
compact bool, skeletonPhase int)` vs the siblings' `View(skeletonPhase int)`.

The divergence is only noted in a comment inside `model_teatest.go`, while
`docs/OVERVIEW.md` describes `internal/mode/*` uniformly as feature-local state
emitting shell contracts. This is documentation-only work — NO FUNCTIONAL
CHANGE, NO CODE CHANGE. The divergence stays; it just gets documented as a
deliberate exception so the `mode/*` contract is not misleading.

## Instructions

1. In `docs/OVERVIEW.md` (the "UI component boundaries" / mode section) add a
   short paragraph: `mode/details` is a shell-driven detail panel, NOT a full
   `tea.Model` — it has no `Init`/`Update` lifecycle and is driven by the app
   shell via `HandleKey`, returning an `OpenRelatedIssueIntent`. `mode/board`
   and `mode/search` are the independent-mode shape.
2. Optionally cross-reference this from `docs/CODING.md` rule 8 (app shell owns
   mode lifecycle) if it clarifies the contract.
3. Do not change any Go code.
