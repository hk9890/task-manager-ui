## Where discovered

Verifier pass on [[5wyt]] (9qpm.1) ÔÇö RenderConfig Dim+Phase foundation.

## Expected

`TestRenderCompactDimAppliesSkeletonShadesForeground` in `internal/ui/shared/issuerow/row_test.go` should assert that for each phase, the rendered output contains the *expected* ANSI sequence corresponding to `styles.SkeletonShades[phase]` ÔÇö i.e. constructs the expected hex foreground escape and asserts `strings.Contains(rendered, expected)`.

## Actual

The test currently iterates all 3 phases and asserts only:
1. ANSI escape is present (`strings.Contains(dimmed, "\x1b[")`).
2. Plain text contains the title.
3. Dimmed output != undimmed output.

It does NOT distinguish "correct shade" from "any shade." A regression that wired `Phase` to the wrong shade table (or got the modulo math wrong) would not be caught by this test.

## Reproduction

Read `internal/ui/shared/issuerow/row_test.go` lines 280-332. No assertion of the form `strings.Contains(dimmed, expectedAnsiForPhase)`.

## Impact

- Low ÔÇö the implementation is correct.
- Test quality gap: future bugs in shade selection or phase math will not be caught here. Would have to be caught downstream (e.g. by 9qpm.2 wiring tests or runtime UI verification).

## Proposed fix

For each phase, build the expected lipgloss-rendered escape from `styles.SkeletonShades[phase]` and assert it appears in the dimmed output. Mirror the pattern from `TestRenderCompactSkeletonPhaseCyclesColor` if that test already does this for the cold-start case.

## Acceptance Criteria

- [ ] Per-phase loop in `TestRenderCompactDimAppliesSkeletonShadesForeground` includes an assertion that the rendered output contains the expected `SkeletonShades[phase]` ANSI sequence (not just any ANSI sequence).
- [ ] Mutating `RenderCompact` to use a different shade table fails this test.
- [ ] `mise run check` and `mise run test` pass.
