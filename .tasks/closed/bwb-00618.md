## Description

Add one multi-step end-to-end test that exercises every write method on the gateway against real `bd`, using a dedicated (non-shared) fixture.

## Why

Mutations can't share a fixture safely. But we don't need many tests ‚Äî one well-chosen lifecycle covers all 4 write methods plus catches ordering bugs (e.g. "update right after create returns stale data") that isolated tests would miss.

## Instructions

1. Create `internal/gateway/beads/contract/lifecycle_scenario_integration_test.go` with `//go:build integration`.
2. `TestRealGatewayIssueLifecycleScenario` should:
   - Build a fresh fixture (use the existing `embeddedfixture.Seed` on a `tb.TempDir()` path ‚Äî NOT the shared snapshot, since we mutate).
   - Construct the real `BeadsGateway`.
   - Execute these steps in order, asserting after each:
     1. **CreateIssue** with known input ‚Üí assert returned `CreateIssueResult` has an ID and matches input fields when read back via `ShowIssue`.
     2. **UpdateIssue** on the just-created issue (change title and priority) ‚Üí `ShowIssue` confirms the change.
     3. **AddComment** on the same issue ‚Üí `ShowIssue` confirms the comment appears.
     4. **CloseIssue** with a reason ‚Üí `ShowIssue` confirms status=closed and the reason is stored.
     5. **ListIssues** after close ‚Üí confirm the closed issue is excluded from the default open-only listing (sanity for cross-method consistency).
3. Assertions should be structured (field comparisons), not string-matched, for the same reason as the read contract.
4. Use `t.Setenv("BEADS_ACTOR", "fixture-user")` for determinism.

## Files to Modify

- `internal/gateway/beads/contract/lifecycle_scenario_integration_test.go` ‚Äî new

## Acceptance Criteria

- [ ] All 4 write methods are exercised (CreateIssue, UpdateIssue, AddComment, CloseIssue)
