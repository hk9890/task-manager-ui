## Outcome

bwb has a clean 9-method `Repository` interface that abstracts issue storage, with two production-quality implementations (bd-backed and in-memory) plus JSONL file persistence. The interface is designed for our actual app surface, not bd's CLI shape.

## Motivation

The current `BeadsGateway` interface (17 methods in `internal/gateway/beads/interface.go`) reflects bd's CLI surface, not bwb's needs. Practical app surface is only 9 methods (verified by grep): 3 reads (Dashboard, Issue, Search), 4 writes, 2 setup. The dashboard fan-out (5 underlying bd calls) is an implementation detail that belongs inside the implementation, not in the interface.

A repository abstraction with a clean contract makes:
- Per-method caching trivial (Issue keyed by ID, Dashboard by constant, Search by query)
- The bd-show false-positive cache-bust problem disappears (per-ID invalidation)
- Tests fast (in-memory backend, no subprocess)
- Future composition possible (Epic 3: caching decorator)
- Persistence possible (fast startup from disk)

## Scope -- 4 implementation subtasks + acceptance review

- 1.1 -- New `internal/repository/` package: interface, domain types, no implementations yet
- 1.2 -- `internal/repository/beads/` -- bd-backed implementation as a translator over existing CLIGateway (fan out Dashboard internally, 1:1 for mutations and Issue/Search)
- 1.3 -- `internal/repository/memory/` -- standalone in-memory implementation with full semantics (dep resolution, sort, filter)
- 1.4 -- `internal/repository/contract/` parity test suite + JSONL file save/load with manifest (schema_version, bd_commit_hash, synced_at)
- 1.5 -- Acceptance review

## Out of scope (this epic)

- Wiring into the app (Epic 2)
- Deleting the old gateway package (Epic 2)
- CachingRepository decorator composing in-memory + bd (Epic 3)
- Removing the argv-level runner cache (Epic 2 decision)

## Acceptance criteria

- [ ] `internal/repository/` package builds; interface and domain types defined
- [ ] Both implementations pass the shared parity test suite identically
- [ ] In-memory implementation models dep resolution, sort direction, filter semantics correctly (no hand-stub fakery)
- [ ] File save/load round-trips with schema_version + bd_commit_hash + synced_at manifest
- [ ] Schema-version mismatch on load = drop cache (no migration code)
- [ ] `mise run quality` green
- [ ] No app code or other-test code touched (this epic is library-only)
