## Description

Ship the shared skeleton-row renderer used by the per-surface non-blocking loading tasks (Board, Search, Detail).

This task is intentionally narrow and additive. The header braille spinner + ticker is split out into `beads-workbench-0x36.6` so the riskier ticker work does not block `.2/.3/.4`.

## Files to Modify

- `internal/ui/loading/loading.go` (or new `internal/ui/skeleton/skeleton.go` if colocation is awkward) — add `SkeletonRow(width int, slots int) string` helper. Use `░` (U+2591) as the skeleton glyph; pin this exactly so `0x36.5` PTY assertions can match.
- `internal/ui/loading/loading_test.go` — unit tests covering: width-respecting output, slot count, deterministic glyph identity.

## Instructions

1. Add `SkeletonRow(width int, slots int) string` returning a styled row built from `░` blocks with subtle dim styling (use the same lipgloss style helpers the rest of the loading package uses).
2. Slot widths should approximate a typical issue row (id slot ~8 chars + title slot fills remaining width). Slots are visually separated by space gaps so the row looks like distinct fields, not one solid bar.
3. Do NOT modify board/search/detail `View()` methods. Do NOT add any ticker / header spinner. Those land in `0x36.6`.
4. Add a `SkeletonGlyph` exported constant (value `"░"`) so `0x36.5` PTY assertions can reference the canonical glyph rather than hard-coding the literal.

## Acceptance Criteria

- [ ] `SkeletonRow(width, slots)` exists, exported from `internal/ui/loading` (or `internal/ui/skeleton`), and returns a row whose visible width equals `width`
- [ ] `SkeletonGlyph` exported constant exists with value `"░"` (U+2591)
- [ ] Unit test covers width-respect, slot count, and glyph identity
- [ ] `go test ./...` passes
- [ ] `go vet ./...` passes
- [ ] No behaviour change to any existing screen (board/search/detail still use their current loading takeover)
