## Description (REVISED after plan review)
`internal/repository/contract` is the single largest cost in the suite (~202s per the source review). The goal is to cut its wall time without losing fake/real parity coverage.

Premise correction: the original "driven by a fresh `bd init` per case" framing is wrong. Measured on-host, `bd init` is ~1.18s while each `bd create` is ~0.8s; init is only ~15% of the ~7.8s/scenario cost. The driver is the per-scenario `bd create`/mutation subprocess calls (corroborated by TESTING.md ~line 90: "~90s due to sequential `bd create` subprocess calls"). Also note `TestRepositoryContract` runs `t.Parallel()` at both impl and scenario level, so the additive "7.8s x scenario" model overstates serial cost.

Architectural note: "share one initialized repo across read-only scenarios" as originally written is blocked — each scenario calls `impl.build(t, seed)` with a DISTINCT seed (`scenarioSeed{}`, `seed`, `seed2`, `scenarioSeed{issues: issues}`), so there is no single shared fixture to reuse. The real lever is reducing the per-scenario `bd create` subprocess count.

## Instructions
1. Map which scenarios are read-only vs. mutating, and what seed each needs.
2. For read-only scenarios, consolidate onto ONE shared seeded-superset fixture built once (shared `bd init` AND shared `bd create` calls), rather than re-seeding per scenario. Keep isolated build only where a scenario mutates state.
3. Confirm parity coverage is unchanged — no scenario dropped, both factories still exercised.
4. Re-measure and record before/after wall time as a comment on this task.

## Acceptance Criteria
- [ ] All contract scenarios still pass under `-race -tags integration`
- [ ] fake/real parity coverage unchanged (no scenario dropped, both factories exercised)
- [ ] Contract package wall time reduced by a concrete, recorded margin (target: cut sequential `bd create` count for read-only scenarios; before/after recorded)
