writes.go lines 18-24: commandRunner() checks g == nil || g.runner == nil and returns a GatewayError. But NewCLIGateway (read_gateway.go line 30-36) already ensures runner is never nil by defaulting to NewCommandRunner(RunnerConfig{}). The only way commandRunner() nil-guard triggers is if someone constructs a Gateway{} literal bypassing the constructor — which is an internal package so callers are controlled. Meanwhile, the read methods (ListIssues, ShowIssue, etc.) call g.runner directly without this nil-guard, so the protection is inconsistent. Suggested action: either add the same nil-guard everywhere (read and write paths) or remove it from writes.go since the constructor prevents it. The inconsistency is worse than either choice.
