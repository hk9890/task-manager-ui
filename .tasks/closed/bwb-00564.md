## Outcome (placeholder)

A `CachingRepository` decorator composes `memory.Repository` (fast-path) with `beads.Repository` (source of truth). Reads hit memory first, fall back to bd on miss, populate memory, return. Writes go through bd, update memory on success. On startup the in-memory state is hydrated from the persisted JSONL file (from Epic 1 step 4) so the dashboard renders instantly; background refresh syncs from bd asynchronously.

This is what realises the original stated goal: "serve from in-memory, delegate to bd, persist to disk for fast restart."

## Prerequisite

Depends on Epic [[hhho]] (E1) and Epic [[8pxi]] (E2). Cannot start until both are complete.

## Open design questions (to settle at epic kickoff, not now)

- **Cache invalidation signal (the structural fix for the bd-show false-positive).** Per-ID invalidation in `CachingRepository.UpdateIssue(X)` solves stale-`Issue(X)` -- but `Dashboard()` is a single composite cache entry built from many issues, so bd-show advancing `last-touched` busts Dashboard exactly as today unless we choose a different signal. Options:
  - **(a)** Periodic `bd vc status` Dolt-commit-hash poll on the refresh tick; treat unchanged hash as "no external write"; ignore mtime advances caused by our own bd-show calls
  - **(b)** `bd query "updated>X"` polling -- fetch only changed issues and merge into store; granular but more expensive
  - **(c)** Track our own bd-show invocations and suppress the mtime bump in the cache token check (heuristic; race-prone with concurrent external writes)
  - **(d)** Drop external-write detection entirely; only invalidate on bwb-initiated writes and manual refresh; loses correctness for multi-agent scenarios
- Write semantics: optimistic (mutate memory first, delegate to bd, revert on failure) vs synchronous (bd first, memory on success)
- Background refresh trigger: startup-only, periodic tick, true SWR per-call, or hybrid
- Multi-process safety: per-session cache file vs shared with file-lock
- Catalog refresh strategy if accepted as stale in [[hhho.1]] -- still stale here, or refresh on Catalogs() invalidation?

## Acceptance criteria (high-level, to be refined)

- [ ] `CachingRepository` implements `Repository` and wraps both other implementations
- [ ] Cold-start time on a project with a warm persisted cache is sub-second to first paint
- [ ] External bd writes detected within configurable interval (target: 60s default) using the invalidation signal chosen above
- [ ] Per-ID invalidation: write to issue X does not bust unrelated cached issues
- [ ] `bd show` side-effect on `last-touched` mtime no longer causes cache-wide invalidation (via the chosen mechanism, NOT structural by-default)
