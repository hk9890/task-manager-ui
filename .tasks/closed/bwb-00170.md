Discovered during verification of beads-workbench-2l4 in the built binary against the embedded fixture repo. Expected: when an issue has blocked/blocks/related refs but no parent-group structure, the Dependencies pane shows Blocked by / Blocks / Related rows only, with selection moving across those rows. Actual: after browser-item population falls back to dependency refs, the detail renderer also shows a Structure section containing the same refs even though no parent-group context exists.

## Root Cause
`dependencyGroups()` appends a Structure section unconditionally from `browserItems`. After the model populates `browserItems` from dependency fallback rather than parent-group context, those same refs appear twice.

## Acceptance Criteria
- [ ] When BrowserItems are populated from dependency refs because no parent-group exists, the Structure section does not appear in Dependencies.
- [ ] When real parent-group structure exists, the Structure section still appears as intended.
- [ ] Real-app review confirms dependency-only detail view no longer duplicates dependency rows.
