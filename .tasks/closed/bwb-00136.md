## Description
Discovered during acceptance review of epic beads-workbench-au54.

When both `$XDG_STATE_HOME`/config-dir paths are unreadable (e.g. `env -i BD_NON_INTERACTIVE=1 HOME=/root /tmp/bwb`), bwb panics in the deferred `logManager.Close()` at `cmd/bwb/main.go:115` → `internal/logging/logging.go:124` → `lumberjack.Logger.Close` (nil pointer dereference).

The `Manager.Close()` nil-check (`m == nil || m.closer == nil`) does not prevent the crash because `m.closer` is non-nil but partially initialized — the lumberjack `Logger` internal state is corrupted by a failed mid-initialization.

Severity: crash on unusual but plausible environment (CI containers, sandboxed environments, root with no HOME).

## Reproduce
```bash
go build -o /tmp/bwb ./cmd/bwb
env -i BD_NON_INTERACTIVE=1 HOME=/root /tmp/bwb
# observe panic
```

## Instructions
1. Inspect `internal/logging/logging.go` initialization — check what state can be partially constructed before the function returns.
2. Either: (a) guarantee `closer` is fully usable when stored, or (b) wrap the deferred Close in a defensive nil-check on the lumberjack logger's internal state.
3. Add a regression test that simulates partial initialization (mock lumberjack constructor returning a half-init logger) and verifies Close does not panic.

## Files to Modify
- `internal/logging/logging.go`
- `internal/logging/logging_test.go` (new test)
