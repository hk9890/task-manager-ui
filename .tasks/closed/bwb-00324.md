## Description

`internal/repository/caching/caching.go:286-296` (`Hydrate`) handles hash-mismatch by setting `dashboardDirty=true` and swapping `c.memory = loaded`. But it gates `c.lastHash = seedHash` on `if seedHash != ""` — and `seedHash` is only set when the hash MATCHED. On mismatch, `lastHash` stays "". The next `tickLoop` call to `RefreshIfChanged` sees `lastHash == ""`, treats the current hash as baseline (first-observation branch lines 157-160), and skips `memory.Reset()`. The stale loaded-from-disk per-ID entries persist indefinitely; only Dashboard is protected by `dashboardDirty`.

User impact: opening an externally-modified issue serves session-A's stale title/status/labels for the remainder of session B.

## Instructions

1. In Hydrate, always set `c.lastHash = currentHash` after the state swap — both when the hash matches (dirty=false, lastHash=loaded) and when it mismatches (dirty=true, lastHash=current). The point is to anchor a baseline so RefreshIfChanged can detect FUTURE changes, even if it can't detect the past mismatch.
2. Actually the better fix on mismatch is: do NOT swap `c.memory = loaded` at all — start cold-memory and refetch lazily. But that changes warm-startup behavior. Pick the minimal correct fix:
   - **Preferred**: if mismatch, do not load loaded into c.memory (start cold-memory). Document the choice in the doc comment.
   - **Alternative**: if mismatch, swap loaded but also call `c.memory.Reset()` immediately and set `c.lastHash = currentHash`. Doc-comment why.

Choose the preferred approach unless there's an existing reason to ship loaded-but-stale (there isn't — Dashboard refetches anyway).
