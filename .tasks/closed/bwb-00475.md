What I did: ran '/tmp/bwb --config <cfg> --check-config' with config 'keybindings: {board: {move_down: [j, j]}}'.

Expected: an error that tells the operator they listed the same key twice for one action.

Actual: exit 1 with 'key "j" conflicts between actions "move_down" and "move_down" in board context'. The message names the same action on both sides of the conflict, so it reads as an action conflicting with itself and does not point the operator at the real mistake (a duplicate entry in one key list).

Severity: low (cosmetic / diagnostics quality; rejection itself is correct).

Root cause: buildContextBindings in internal/config/keybindings.go uses a single ctx.index map keyed by canonical key name; when the same action lists a key twice, the second pass hits ctx.index[key] already set to the same action and emits the generic between-actions conflict error.

Repro:
  printf 'keybindings:\n  board:\n    move_down: [j, j]\n' > /tmp/c.yaml
  /tmp/bwb --config /tmp/c.yaml --check-config

Fix: detect the within-action duplicate case (existing == action) and emit a distinct message, e.g. 'key "j" listed more than once for action "move_down" in board context'.

Other merge edge cases checked in the same pass behaved correctly: empty key list -> 'must define at least one key' (exit 1); post-merge cross-action conflict -> clear conflict error (exit 1); freeing both sides of a conflict -> config OK.
