## Description
99% of logged gateway calls were byte-identical repeats (session 8c657fd1: 1063 calls, 7 distinct argv) -- the board re-runs the same queries every 60s even when the DB is unchanged. Add an in-process read cache so unchanged-DB reads don't spawn bd.

## Instructions
1. Add a read-result cache to the runner (or a thin same-package wrapper) keyed on resolved argv. Guard all cache state with its own mutex -- it is accessed concurrently by parallel RLock-held reads.
2. Change-token: stat mtime of .beads/last-touched (bd maintains this file; verified it advances on writes -- embeddeddolt dir mtime does NOT, so do not use it). Resolve the .beads dir from the runner's bound WorkDir.
3. On a read request: compute the current token; if a cache entry exists with a matching token, return cached stdout without spawning bd; otherwise exec, then store {token, stdout}.
4. Invalidate on writes: any IsWrite=true request clears/stamps the cache so the next read re-execs. External writes are caught by the token advancing.
5. Add per-argv singleflight (golang.org/x/sync/singleflight, already in go.mod) so identical concurrent argv collapse to one exec. The board's 4 refresh commands are distinct argv -- singleflight does not merge them; that burst is bounded by wcgy.1.
6. Only cache successful reads (exit 0). Do not cache failures or write requests.
7. Update docs/MONITORING.md if the diagnostics/contract surface changes.

## Acceptance Criteria
- [ ] A repeat read with an unchanged .beads/last-touched token returns from cache without spawning bd (counting-executor test)
- [ ] A write request, or an advanced token, invalidates the cache so the next read re-execs
- [ ] Identical concurrent argv collapse to a single exec via singleflight (test)
- [ ] Only exit-0 reads are cached; failures and writes are not
- [ ] Cache access is data-race free (go test -race passes)
- [ ] go test ./internal/gateway/beads/... passes; quality:fast gate passes
