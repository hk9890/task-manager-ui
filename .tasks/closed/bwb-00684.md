## Description
Define the core code contracts that the rest of Beads Workbench will build on. This task creates source-specific domain models, query/mutation inputs, normalized gateway errors, and the `BeadsGateway` interface without leaking SQL-era assumptions into the new architecture.

## Instructions
1. Create domain models for issue summaries, issue details, blocked issue views, search results, and any supporting metadata required by the gateway.
2. Define query input models for list, ready, blocked, show, and search flows.
3. Define mutation input models for create, update, close, and comment flows.
4. Define normalized error types suitable for future TUI presentation.
5. Define a source-specific `BeadsGateway` interface under `internal/gateway/beads/`.
6. Include catalog/query methods needed by the UI and write flows, including status, type, and label catalogs.
7. Keep the interface bound to a single beads source/project. Do not design federation into the interface itself.
8. Do not introduce `database/sql`, BQL contracts, or orchestration dependencies.

## Files to Modify
- `internal/domain/` (create models)
- `internal/gateway/beads/` (create interface and error contracts)

## Acceptance Criteria
- [ ] Source-specific issue summary/detail/search models exist
- [ ] Query models exist for list, ready, blocked, show, and search flows
- [ ] Mutation models exist for create, update, close, and comment flows
- [ ] Normalized gateway error types exist for UI use
- [ ] `BeadsGateway` interface exists and is source-specific
- [ ] The interface includes catalog methods for status, type, and labels needed by the UI
- [ ] No SQL-specific, BQL-specific, or orchestration-specific types leak into the contracts
