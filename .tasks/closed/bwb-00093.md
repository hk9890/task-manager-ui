## Outcome
Pressing Enter on a dependency drills into that related issue (unchanged). After the
drilled issue loads, focus stays on the **Dependencies rail when that issue has
dependencies**, and only drops to the **Content pane when the rail is empty**.

## Root cause
When drilling in, the shell paints an optimistic placeholder detail that has no
dependencies yet (`internal/app/model.go:698-703`). `ApplyLoadedDetail -> syncBrowserPanel`
sees an empty rail and calls `clearBrowserPanel()` (`internal/mode/details/model.go:523-530`),
which flips focus Dependencies -> Content during the in-flight window. When the real
detail arrives with its dependencies, nothing restores focus, so it sticks on Content
even for issues that do have a rail.

## Success criteria
- [ ] Drilling (Enter) into an issue WITH dependencies keeps focus on the Dependencies rail.
- [ ] Drilling (Enter) into a leaf issue (empty rail) moves focus to the Content pane.
- [ ] The transient placeholder phase no longer flips focus prematurely.
- [ ] Acceptance review passes.
