## Description

When the user scrolls deep into issue A in detail mode and then selects issue B from board or search, only the legacy `m.detail.ScrollOffset = 0` is zeroed at `internal/app/model.go:759`. The active offsets `ContentScrollOffset`, `MetadataScrollOffset`, `DependenciesScrollOffset` are NOT reset. `mode/details.ApplyLoadedDetail` (`internal/mode/details/model.go:48-52`) does not touch them either. The bridge `syncLegacyScrollAlias()` (`model.go:578-583`) only copies `ScrollOffset` into `ContentScrollOffset` when the latter is 0 — so when a user has scrolled, the alias-reset is effectively a no-op.

Result: switching from issue A to issue B opens B already mid-scroll (clamped to B's end), which looks like a layout glitch.

Major finding U1 from the 2026-05-16 audit.

## Files to Modify

- `internal/mode/details/model.go` (`ApplyLoadedDetail`)
- `internal/app/model.go` (around line 759 — remove legacy reset once primary fix lands)
- new test: `internal/mode/details/model_scroll_test.go` or add to existing model_test.go

## Instructions

1. In `mode/details.ApplyLoadedDetail`, compare the new `issueID` to the previously held one. If different (or if previous was empty), zero `ContentScrollOffset`, `MetadataScrollOffset`, `DependenciesScrollOffset` BEFORE `ClampScroll` runs.
2. Remove the legacy `m.detail.ScrollOffset = 0` line in `app/model.go:759` once the primary reset is in place.
3. Consider deleting the `ScrollOffset` alias entirely if no consumer reads it — verify with a grep. If consumers exist, leave a deprecation comment.
4. Add a test: load issue A, scroll content to offset N>0, switch to issue B, assert all three offsets are 0 after `ApplyLoadedDetail`. Also assert scroll offsets are RETAINED if the same issue is re-loaded (e.g., from a refresh).
