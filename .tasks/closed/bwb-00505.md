## Description

Add a sibling function `RenderCompactSkeleton(width int, opts SkeletonOpts) string` (and `RenderReferenceCompactSkeleton(...)`) next to `RenderCompact` in `internal/ui/shared/issuerow/row.go`. The new function reuses the same internal slot arithmetic (`CompactIDWidth`, prefix layout) but emits `▓`-filled spans per segment so placeholder rows visually mirror real issue rows. Route every loading surface — *except* the search detail preview, see T2 — through the new primitive.

## Why

Today three skeleton surfaces use `skeleton.SkeletonRow(width, slots)` with hardcoded slot counts (2 for board/search-results, 3 for details). The glyph (`░`) reads as grit; slot widths don't match real rows; the surfaces speak inconsistent visual languages. We want a single primitive shaped like a real `RenderCompact` row.

**Why sibling and not a `Skeleton bool` on `RenderConfig`:** keeps `RenderCompact` data-only (no junk `Skeleton: false` on every real call), no per-test branch discrimination, and the skeleton path can evolve (per-phase coloring in T3, future variants) without touching the data path. Shared "primitive" is the internal slot arithmetic, not the public entry point.
