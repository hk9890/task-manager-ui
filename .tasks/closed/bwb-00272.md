In editor/service.go:57-59, there is a nil-receiver guard:
```go
func (e *IssueEditor) EditIssue(ctx context.Context, issueID string) (Result, error) {
    if e == nil {
        return Result{}, fmt.Errorf("issue editor is not configured")
    }
```

This is unreachable in normal Go code. `NewIssueEditor` either returns a valid pointer or an error — callers cannot get a nil `*IssueEditor` through the constructor. And the `Service` interface is the external contract, so callers work through the interface, not the concrete pointer.

If someone stores a nil `*IssueEditor` in a `Service` interface variable, the nil check won't help — the interface is non-nil (typed nil), so calling `EditIssue` will enter the method with `e == nil`, but this is a misuse pattern that should be caught at construction time, not at every call site.

This guard adds false safety and confuses readers into thinking nil receivers are expected. Remove it.
