## Description
If `bd` is not installed or not in PATH, the current behavior is: the app starts, the board mode loads, a `bd` command fails with `ErrorCodeCommandUnavailable` (see `internal/gateway/beads/runner.go:222`), and the board renders "Unable to load board dashboards. Error: bd command is unavailable" inline in the board area. This is a bad UX — there is no indication that the app itself is broken, no actionable advice, and the rest of the shell chrome (header, footer, etc.) renders as if everything is fine.

Instead, the app should detect a missing `bd` at startup and immediately show a dedicated full-screen error screen that explains the problem and what to do.

## Instructions

### 1. Add a startup health check command
- In `cmd/bwb/main.go` or `internal/app/model.go`, before or immediately on `Init()`, run a lightweight check: try to execute `bd version` (or similar) via the existing `CommandRunner`
- If the check returns `ErrorCodeCommandUnavailable`, the app should short-circuit to a fatal error state
- The check should be fast (not a heavy bd query) — `bd version` or a simple executable lookup is sufficient

### 2. Add a fatal error state to the root app model
- In `internal/app/model.go`, add a `fatalError` field (string or typed error) to the root `Model`
- On `Init()`, dispatch the health check command as the first `tea.Cmd`
- When the health check result arrives and it is `ErrorCodeCommandUnavailable`, set `fatalError` and return no further commands (skip all other initialization)
- In `View()`, check `fatalError` first: if set, render the fatal error screen instead of the normal board/search/detail layout

### 3. Create a fatal error screen component
- Create `internal/ui/fatalerror/fatalerror.go` with a `View(title, body string, width, height int) string` function
- The screen should be a centered, full-terminal view with:
  - Clear title: "beads is not available"  
  - Explanation: "The `bd` CLI tool was not found in your PATH."
  - Actionable hint: "Install beads to use this app. See https://github.com/hk9890/beads-workbench for setup instructions." (use the repo URL from go.mod or CLAUDE.md)
  - Press `q` or `ctrl+c` to quit hint
- Use existing `internal/ui/styles` colors for styling

### 4. Wire key handling for the fatal error screen
- In the root model's `Update()`, if `fatalError` is set, only handle `tea.KeyMsg` for quit (`q`, `ctrl+c`) and window size changes

## Files to Modify / Create
- `internal/app/model.go` — add `fatalError` field, health-check cmd dispatch, `View()` routing, key handling
- `internal/gateway/beads/read_gateway.go` or `runner.go` — add `HealthCheck(ctx) error` method to the gateway interface if not present
- `internal/domain/gateway.go` or equivalent gateway interface file — add `HealthCheck` to the interface
- `internal/ui/fatalerror/fatalerror.go` — new package, full-screen error view

## Acceptance Criteria
- [ ] When `bd` is not in PATH, app shows a full-screen error screen immediately (no board renders at all)
- [ ] Error screen has a clear title, explanation, and actionable advice
- [ ] `q` or `ctrl+c` quits the app from the error screen
- [ ] When `bd` is available, the health check passes and the app loads the board normally with no regression
- [ ] Health check failure is also logged via slog at error level
