The Done column loads only viewport-sized rows from the repository (`opts.ClosedLimit = sectionItemCapacity()`). On a repo with more closed issues than fit (this repo: 736 closed; dtctl-test: 89 closed at height=25) the user can see only the first slice. Header honestly reads `N of M` but there is no UI affordance — and no in-memory data — to reach the remaining closed issues.

`beads-workbench-b38b.4` covers the renderer-side selection-follows-viewport fix for the other three columns. That fix is necessary but not sufficient for Done: there is nothing in memory to scroll into, so the fix has to span repository + composer + model + UI.

## Symptom

`bwb` against this repo at any practical height shows e.g. `35 of 736` in Done. Pressing `j` past the last loaded row either stops at the slice end or wraps. The 701 closed issues older than the last visible row are unreachable.

## Root cause

- `internal/mode/board/model.go:332,349` sets `opts.ClosedLimit = m.sectionItemCapacity()` per dashboard load.
- `internal/repository/beads/lean_reads.go` builds `bd query --status=closed --sort closed --limit N` from that.
- `internal/dashboard/composer.go:193-219` builds `Done.Issues = closedIssues[:N]`, `Total = ClosedTotal`, `TotalIsExact = len(closedIssues) >= ClosedTotal`.
- There is no offset or "fetch next page" concept anywhere in the stack.

## Design decisions (committed up front so the executing agent isn't blocked)

These resolve the open questions captured in the original draft:

1. **Pagination model: growing slice.** Each load-more appends to `Done.Issues`; `ClosedTotal` does not change. Rationale: matches the iwvm precedent (cap grew with viewport rather than introducing offsets); simpler client state (one slice, one offset = `len(Issues)`); honest header `N of M` evolves naturally as N grows. Reject per-offset caching: closed lists mutate (reopens, mass-close ops) and the cache invalidation matrix is not worth the win.

2. **Repository surface: extend `DashboardOptions` with `ClosedOffset int`.** A load-more call sets `ClosedLimit = page-size` and `ClosedOffset = currentLen`. Initial load uses `ClosedOffset = 0`. The repo emits `bd query --status=closed --sort closed --limit N --offset M`.

3. **Caching strategy: pass-through on offset > 0.** When `ClosedOffset > 0`, `caching.Repository` skips its memory cache and hits the backing repo directly; the returned slice is appended to the persisted-cache snapshot under lock. This keeps the cache's "warm initial board" semantic intact while letting deep navigation always reflect live data.

4. **Refresh behavior: deep navigation resets to page 1 on `r`, focus regain, and the periodic auto-refresh tick.** Rationale: simpler invariant; deep navigation is rare; the user can scroll down again. If usability proves bad, revisit.

5. **Trigger: implicit threshold + explicit key.** When the cursor's distance from the loaded-slice end is `< loadMoreThreshold` (initial: 5), dispatch a load-more cmd. Also bind `>` (or similar; final key TBD) to "load next page" for power users. One in-flight load at a time (ule7 pattern).

6. **Page size: `max(2 * sectionItemCapacity(), 50)`.** Big enough to absorb common scroll bursts without thrashing `bd`; small enough not to feel slow.

7. **Other columns stay client-only.** `b38b.4` is the canonical fix for Ready / NotReady / InProgress. Pagination is not planned for them.

## Scope of change

| Layer | Files | What changes |
|---|---|---|
| Repository interface | `internal/repository/repository.go` | Add `ClosedOffset int` to `DashboardOptions`. |
| `beads` impl | `internal/repository/beads/lean_reads.go` | Thread `opts.ClosedOffset` into `bd query --offset M`. |
| `memory` impl | `internal/repository/memory/repository.go` | Apply offset to closed slice before limit. |
| `caching` impl | `internal/repository/caching/caching.go` | When `ClosedOffset > 0`, bypass memory cache; on success, append to persisted snapshot under lock. |
| Composer | `internal/dashboard/composer.go` | Take an explicit "merge with previous Done" path: caller passes prior `Done.Issues` and new page; composer concatenates and recomputes `TotalIsExact` from `ClosedTotal`. |
| Board model | `internal/mode/board/model.go` | Track `doneLoadedCount` and `doneLoadInFlight`. New `loadMoreClosedCmd` dispatched on cursor threshold or `>` key. In-flight guard prevents overlap. On `r` or auto-refresh, reset to page 1. |
| Renderer | `internal/ui/board/board.go` | Optional: a "more below" affordance (e.g. a dim `↓` glyph in the bottom row of Done when `Selected` is near the end and a load is queued). |
| bd argv | `internal/bd/doc.go` + pinning test | Document the new `--offset` flag in the lean argv contract; add per-operation argv fixture. |
| Runbook | `docs/RUNTIME_UI_VERIFICATION.md` | New procedure: deep Done navigation. |

## Test plan

### Tier 1 — unit
- **Composer**: feed a prior `Done.Issues` of length 35 + a new page of 50 + `ClosedTotal=736`. Assert `len(Done.Issues)==85`, `Total==736`, `TotalIsExact==false`. Edge: when the appended page lands `len == ClosedTotal`, `TotalIsExact==true`.
- **Board model — load-more dispatch**: synthesize a Done column with `len(Issues)=35`, cursor at index 31. Send `j`. Assert exactly one `loadMoreClosedCmd` is emitted with `ClosedOffset=35, ClosedLimit=PageSize`.
- **Board model — in-flight guard**: use `DelayedFakeRepository` (`internal/mode/search/model_async_test.go` pattern). Cursor at 31, press `j` ×5 while the first load is delayed. Assert exactly one cmd was dispatched; the rest are no-ops.
- **Board model — refresh resets**: with `len(Issues)=85` (after one load-more), press `r`. Assert next `loadDashboardCmd` uses `ClosedOffset=0, ClosedLimit=sectionItemCapacity()`.

### Tier 1 — argv pinning (`RecordingExecutor`)
- New `internal/repository/beads/lean_reads_argv_test.go` case: call `Dashboard(ctx, opts{ClosedLimit:50, ClosedOffset:35})` and assert the recorded argv equals exactly `["query", "status=closed", "--json", "-a", "--sort", "closed", "--limit", "50", "--offset", "35"]`. Pin the wire contract.

### Tier 2 — parity contract
- New scenario `PaginatedClosedFetch` in `internal/repository/contract/parity_integration_test.go`: seed >100 closed issues into each factory; call `Dashboard` with offset 0 / limit 50, then offset 50 / limit 50; assert no ID overlap, contiguous `ClosedAt DESC` order, total equals expected; `len(page1)+len(page2) <= ClosedTotal`. Both `memory` and `beads` factories must pass.

### Tier 1 — golden
- `internal/ui/board/testdata/done_loading_more.golden` — Done column with 60 loaded of 736, in-flight skeleton row at the bottom.
- `internal/ui/board/testdata/done_deep_navigation.golden` — Done with `Selected` at row 58 of 60 loaded; cursor `›` visible inside the windowed render (this depends on `b38b.4` having landed first).

### Runtime UI (runbook)
Add to `docs/RUNTIME_UI_VERIFICATION.md`:

1. Build, launch against this repo (`>700` closed issues) at a 25-row terminal.
2. Note initial Done header `N of 736`.
3. Press `RIGHT RIGHT RIGHT` to focus Done.
4. Press `j` past the loaded end. Confirm: header grows (e.g. `35 of 736` → `85 of 736`); no double-load (only one network/process call in `[bwb-debug]` trace per threshold crossing); cursor stays inside the visible window (depends on `b38b.4`).
5. Press `r`. Confirm: header resets to the initial viewport count; `Selected` returns to top of Done.
