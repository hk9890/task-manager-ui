## Description
Discovered while reviewing the shared persistent log at `~/.local/state/bwb/bwb.log`. That file is user/machine scoped, so it can contain sessions from multiple repositories and multiple BWB binaries. In this environment, trunk builds are typically used for this repository while other repositories may use the release binary, which makes cross-project log interpretation ambiguous unless provenance is explicit.

## Expected Behavior
Operators and agents should be able to tell whether a given log record came from this repository or another repository, and whether it came from the trunk build or a release build, without relying on guesswork.

## Actual Behavior
The persistent log mixes sessions in one file. Startup records include `cwd`, but routine gateway records do not carry enough provenance to make project/build attribution obvious when multiple repos and binary versions are in use on the same machine.

## Steps to Reproduce
1. Use BWB from more than one repository on the same machine.
2. Run one repo with a locally built trunk binary and another repo with a release binary.
3. Inspect the shared persistent log at `~/.local/state/bwb/bwb.log`.
4. Observe that records are grouped only by `session_id`, with limited per-record context and no explicit build/version provenance.

## Impact
Monitoring and debugging can attribute behavior to the wrong repository or wrong binary version. This is especially risky when comparing trunk behavior in this repo against release behavior in other repos on the same machine.

## Files to Modify
- internal/logging/logging.go
- cmd/bwb/main.go
- docs/MONITORING.md
- logging/startup coverage tests

## Acceptance Criteria
- Log records make project provenance explicit enough to distinguish repositories in a shared log sink.
- Log records or startup/session metadata make build provenance explicit enough to distinguish trunk/dev from release builds.
- Documentation states whether the log sink is shared and how to attribute sessions safely.
