## Discovered during

[[9x70.1]] (write contract audit, 2026-05-17).

## Expected vs actual

Expected: UpdateIssue with `ClearLabels=true` clears the labels array on the stored issue.

Actual: bd 1.0.4 `bd update --set-labels ''` exits 0 with a success message but leaves labels unchanged. Confirmed via audit (`/tmp/9x70-audit.log`).

## Minimal repro

```
tmp=$(mktemp -d) && cd "$tmp" && BD_NON_INTERACTIVE=1 bd init
bd create --title "test" --type task --labels "alpha,beta"
# get the id, then:
bd update <id> --set-labels ""    # exits 0, success msg
bd show <id> --json | jq '.labels'  # labels still ["alpha","beta"]
```

## Verification required

Must verify in app (`mise run bwb` against a representative DB) that removing all labels in the edit modal appears to succeed but labels persist after reload. If confirmed visually, fix at the appropriate layer.

## Suggested fix layer

Two options:
1. **Gateway adapter**: the gateway's ClearLabels path could emit a different bd flag combination that actually clears (e.g., omit `--set-labels` entirely and use `--remove-labels alpha,beta,...` enumerating each existing label). Requires fetching current labels first.
2. **FILE UPSTREAM**: this is a bd bug; the fix belongs there. Use option 1 as a workaround until the upstream fix lands.
