## Description
Direct runtime verification shows empty-query search is still broken in the real app. Entering search with no query currently produces an error because the gateway/search path attempts an official `bd search` call with an empty query, and the CLI rejects that with `search query is required`.

## Evidence
- User runtime output:
  - `Search failed`
  - `search issues: command exited with code 1`
  - `error: search query is required`
- This disproves the intended a2p UX change in real runtime.

## Required behavior
1. Entering search with an empty query must show a useful all-issues result set.
2. The implementation must not rely on an empty `bd search` call if the official CLI forbids it.
3. Use an alternate official-surface strategy for the empty-query state (for example a list-based fallback or another explicit gateway path), while preserving live filtering when the query becomes non-empty.
4. Clearing the query must restore the same all-issues result set without error.
5. Add automated coverage that proves the empty-query path uses the correct fallback behavior.
6. Re-verify with a real runtime run, not just unit tests.

## Files to Inspect
- `internal/gateway/beads/`
- `internal/mode/search/`
- `internal/ui/search/`
- `internal/app/`

## Acceptance Criteria
- [ ] Empty-query search shows all issues without runtime error
- [ ] Empty-query path avoids illegal `bd search` invocation
- [ ] Typing still filters live for non-empty query
- [ ] Clearing query restores all-issues view
- [ ] Tests cover the fallback behavior
- [ ] Real runtime verification confirms the fix
