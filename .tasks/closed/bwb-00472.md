Context: internal/mode/search/model.go, CapturesShellKey (line 551) and shellKeysPassThrough (line 573).

What I observed: When the search query field is focused, CapturesShellKey checks 'if msg.Type == tea.KeyRunes { return true }' at line 559 -- BEFORE it calls shellKeysPassThrough at line 562. shellKeysPassThrough explicitly lists ShellActionHelp and ShellActionModeDetail among the shell actions that should pass through to the shell while typing. But the default bindings for those actions are '?' and '3', which are KeyRunes. So line 559 captures them into the query first, and shellKeysPassThrough never sees them. Result: with the default config, pressing '?' or '3' in search mode while the query is focused types the character into the query and does NOT open help / detail.

Why this is unclear: Two readings, and source/docs do not disambiguate:
(a) Intentional -- shellKeysPassThrough only needs to work for NON-rune rebinds (e.g. toggle_help rebound to F1); for rune defaults, typing the literal character into the query is the desired behavior, and the list entries are just defensive coverage for rebinds.
(b) Latent bug -- the design intent is that help/detail stay reachable even while typing a query, and the KeyRunes early-return at line 559 should exclude keys that shellKeysPassThrough would pass through (i.e. check passthrough before the rune capture).

Note: help/detail ARE still reachable in search mode by moving focus off the query pane (focus != FocusQuery -> CapturesShellKey returns false -> shell switch runs), so this is not a lockout -- only a question of whether the query-focused behavior and the dead list entries are deliberate.

If (a): consider a code comment on shellKeysPassThrough explaining the rune-default entries are intentionally dead. If (b): reorder so passthrough is checked before the KeyRunes capture.
