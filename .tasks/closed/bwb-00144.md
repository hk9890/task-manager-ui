## Description
The gateway `CommandRunner` currently uses a single `sync.Mutex` (see `internal/gateway/beads/runner.go:62, 103-107`) that serializes every `bd` subprocess. Empirical sampling (~25 concurrent reads, no failures) suggests reads are concurrency-safe, but the existing comment at `runner.go:97-102` warns that "Dolt uses file locking for the local .beads database, so concurrent bd CLI calls against the same repo can contend and fail." This task declares that comment **stale** (based on observation against bd 1.0.4 + embedded Dolt) and switches to `sync.RWMutex` so reads run in parallel while writes stay serialized.

The switch is what makes the dashboard rewrite (epic beads-workbench-au54) actually faster — without it, `tea.Batch` runs 3 reads serially behind the mutex.

## Risk acknowledgment
The original mutex was a defensive choice with limited evidence. This task replaces it based on empirical evidence that ALSO has limited sample size. To reduce risk, a meaningful contention soak is required (see test plan), and the original safety comment is rewritten to capture the new rationale + the contention test result for posterity.

## Instructions
1. In `internal/gateway/beads/runner.go`:
   - Replace `runMu sync.Mutex` with `runMu sync.RWMutex`.
   - Add an `IsWrite bool` field to `CommandRequest` (default false = read).
   - In `Run`, take `runMu.Lock()`/`Unlock()` when `req.IsWrite`, and `runMu.RLock()`/`RUnlock()` otherwise.
   - Replace the existing block comment at lines 97-102 with one that explains the new contract: reads parallel; writes serialized; cite that this was verified empirically and reference the soak test added in this task.
2. In `internal/gateway/beads/writes.go`, set `IsWrite: true` on every `CommandRequest` issued for mutating bd commands (create, update, close, comment add, dep add, etc.).
3. Read-side callers (`read_gateway.go`) do nothing — they keep the default `IsWrite: false`.
4. Add a concurrency soak test in `runner_test.go` (the term "soak" here means a real-load test, not a microbenchmark):
   - **Parallel-read overlap test**: spawn N=100 concurrent `Run` calls with a fake `CommandExecutor` that sleeps 20 ms. Verify total wall time is well below `N × 20 ms` (target: < 100 ms, allowing ~5× overlap). Run the inner loop 5 times to catch flakiness.
   - **Write exclusion test**: mix 5 writers with 10 readers; verify writers wait for in-flight readers and subsequent readers wait for the writer (use a Counter pattern: increment under RLock, panic if ever > 1 writer simultaneously, allow many readers).
   - **Real-bd contention probe** (optional but recommended): if a `.beads` repo is available in the test environment, run a separate test (skippable via `-short`) that issues 100 parallel `bd list --json` invocations against the real binary and asserts all exit 0 with no stderr. If this fails, the RWMutex is unsafe and the ticket must be reopened with the failure attached.
5. Audit `writes.go` after the change: every exported call must set `IsWrite: true`. List the matches in the PR description.
