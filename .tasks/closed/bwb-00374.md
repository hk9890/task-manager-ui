## Description

`internal/gateway/beads/read_gateway.go:753` (`parentChildSiblings`) issues a second blocking `bd show` per detail load for any child issue. This doubles the latency of every detail open for a child issue.

Minor finding m3 from the 2026-05-16 audit.

> **Scope note:** This ticket originally bundled a HealthCheck improvement (Part 1) and the parent-fetch deferral (Part 2). The HealthCheck part is already done — `read_gateway.go:46-48` uses `bd ping --json` and `TestGatewayHealthCheckIssuesPingJSON` pins the contract. Only Part 2 remains in this ticket.

## Files to Modify

- `internal/gateway/beads/read_gateway.go`
- extend `internal/gateway/beads/read_gateway_test.go`

## Instructions

1. Either:
   - Cache the parent lookup per detail load (Bubble Tea command-scoped), OR
   - Move the parent fetch to a follow-up message so it does not block the main `ShowIssue` return. The detail view can render without siblings initially, then re-render when siblings arrive.
   Pick the smaller change.
2. Add a test asserting that detail load for a child issue issues at most one `bd show` synchronously.

- [ ] Detail load issues at most one `bd show` synchronously
- [ ] Existing detail tests still pass
