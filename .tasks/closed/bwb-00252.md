## Description
Discovered during acceptance review of epic beads-workbench-au54. The dashboard rewrite fires exactly 4 bd subprocesses (ping + ready_explain + query in_progress + query closed). But `internal/app/model.go:216` also batches `m.search.Init()` in the startup `tea.Batch`, which fires a 5th concurrent `bd list --json --all --limit N` from `searchIssuesFromList`.

The 5th call contends with the 4 dashboard reads on Dolt's embedded mode and adds wall time to dashboard-first-paint. Most users land on the board first; the search preload is speculative.

## Instructions
1. Move `m.search.Init()` out of `app.Model.Init`'s startup batch. Instead, trigger the search initial load lazily when the user first switches to search mode (or restore it on mode entry).
2. Confirm search mode UX still loads results promptly when entered (lazy-load latency should be near-zero compared to current preload).
3. Update relevant scenario tests if any depend on search preload at startup.

## Acceptance Criteria
- [ ] Cold dashboard startup fires exactly 4 bd subprocesses (verified via persistent log)
- [ ] Switching to search mode for the first time still loads default results within ~500 ms
- [ ] Existing search scenario tests pass; any startup-preload assertions migrated to mode-entry preload
- [ ] Full quality gate green
