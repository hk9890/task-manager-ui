## Description

Today the integration suite (95s wall, 12 tests) runs the full app stack against real `bd` — paying ~10s per test for fixture seeding and bd subprocess startup. Most of those tests assert app behavior that could equally be covered by the fake gateway in <100ms each.

This epic restructures testing around the **gateway boundary**:

1. **Unit tier** — uses `fakes.FakeBeadsGateway`. Covers app behavior. Fast (<10s for the whole tier).
2. **Gateway-integration tier** — the ONLY tests that hit real `bd`. Two subgroups:
   - **2a. Read contract** — single immutable fixture built once per package via `TestMain`, snapshot of `.beads/beads.db` copied per test. Covers every read method on the gateway against well-known data. Fast (1 seed + cheap copies).
   - **2b. Mutating scenarios** — small number (2) of multi-step end-to-end tests on dedicated fixtures: full create→update→comment→close lifecycle, and links/dependencies. Sized to be debuggable (~5–8 steps each).
3. **Fake/real parity is enforced** by a contract test suite parameterized over `{fake, real}` — both implementations run the same read test bodies and must produce identical results. This is what lets us trust the fake at the unit tier.

## Why

- Current integration cost (~95s) is dominated by per-test fixture rebuilds (~10s × 10 tests). Reads can share a fixture; mutations only need a few well-chosen scenarios.
- App-layer integration tests are *double-covered* today — most assertions could be unit tests using the fake; the integration tier should verify only the gateway's bd contract.
- Without a fake/real parity contract test, fakes silently drift from reality. The contract test closes that gap once and forever.

## Scope decisions (locked during planning)

- **Mutating scenarios = 2**: (a) single-issue lifecycle (create→update→comment→close), (b) links and dependencies (folds the two Bucket B tests). Each ~5–8 steps. Tunable later if a real coverage gap emerges.
- **Read fixture lives at `internal/testing/e2e/embeddedfixture/`** (extend existing package). One seed per `TestMain`, snapshot `.beads/beads.db` SQLite file, `cp` per sub-test.
- **Contract test suite at `internal/gateway/beads/contract/`** — parameterized over fake+real factories.
- **Tea.Tick blocking issue in unit tests** (Bucket D, ~30s) is in-scope as a separate task — it's orthogonal but blocks the <10s unit target.
- **`process_runner_integration_test.go` sleep** (2s, Bucket E) is in-scope.
- **Out-of-scope**: the launcher integration tests stay where they are (real subprocesses, not bd).
