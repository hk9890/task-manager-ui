## Description

Replace the current full-screen "Loading from gateway‚Ä¶" takeover with a non-blocking loading UX across all three top-level surfaces (Board, Search, Detail).

Today, both initial load and any background refresh (e.g. switching modes triggers `maybeAutoRefreshActiveSurfaceCmd` which sets `m.loading=true` and `View()` short-circuits to `loading.View(...)`) hide the entire screen until gateway responses arrive. Observable on:
- Board (`internal/mode/board/model.go:76,226-228,305,338`)
- Search short-circuit lives in `internal/ui/search/search.go:68-70` (NOT mode model; mode model already has `Reloading` field at `internal/mode/search/model.go:46`)
- Detail (`internal/mode/details/model.go:67,98-104`)

After this epic:
- Cold start: app shows chrome (header + tabs + column frames) immediately. Each empty region renders a skeleton placeholder built from `‚ĖĎ`.
- Refresh: stale data stays on screen and stays interactive. A small animated braille spinner in the header (left of the tab bar, glyph from `‚†č‚†ô‚†Ļ‚†ł‚†ľ‚†ī‚†¶‚†ß‚†á‚†Ź`) is the only signal that data is in flight.
- New data swaps in atomically per region (the 4-column board block / detail / search results) as gateway calls return.

App-level loading aggregation already exists at `internal/app/model.go:1258-1270` (`loadingStates()` returns scopes for board/search/detail) ‚ÄĒ the data is there; we just need to stop hiding the rest of the UI behind it.
