## What

In internal/mode/board/model.go at the BoardActionReload keyboard handler (currently line 233-234), guard against re-entering startReload while a refresh is already in flight, and emit a debug log when the guard fires.

## Why

This is the primary trigger for the refresh-corruption bug described in the parent epic. The handler currently calls m.startReload(refreshModeManual) unconditionally, while AutoRefresh() at line 323-328 already does the right thing with `if m.IsLoading() { return nil }`.

The debug log makes the suppression visible in `bwb --debug` traces and the persistent JSON Lines log per docs/MONITORING.md — silent no-ops make this class of bug invisible.

## Change

```go
case m.keys.Match(config.BoardContext, config.BoardActionReload, msg):
    if m.IsLoading() {
        m.logger.Debug("manual board refresh suppressed; refresh already in flight",
            "component", "board",
            "trigger", "board-manual")
        return nil
    }
    return m.startReload(refreshModeManual)
```

## Dependencies

- Depends on 5q6t.4 having landed first (TDD: the failing regression test for double-refresh must exist and FAIL on current code before this fix is applied).

## Acceptance

- Two consecutive BoardActionReload key events while the first refresh is in flight produce only one startReload invocation.
- pendingResults is not reset by the second keypress.
- The regression test from 5q6t.4 transitions from FAIL to PASS after this change.
- A `bwb --debug` run hammering 'r' mid-refresh shows the "manual board refresh suppressed" log entries (verified manually as part of 5q6t.5).
- Existing AutoRefresh path is unchanged (assert by running existing tests in internal/mode/board/model_test.go and internal/app/model_loading_refresh_test.go — all must still pass).
