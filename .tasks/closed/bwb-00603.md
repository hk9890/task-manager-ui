## Description
Today, moving the cursor (↑/↓) through the Dependencies pane in Detail mode immediately reloads the whole detail to the highlighted issue (`moveRelatedSelection` returns `OpenRelatedIssueIntent`, internal/mode/details/model.go:231-250; handled in internal/app/model.go:683-699). Decouple this: cursor movement should only move the highlight. The full detail should change only when the user presses Enter on the highlighted related/child issue. Because the cursor and the displayed subject become distinct, the cursor's pending selection must be visually distinguishable.

Depends on .1 (shared files; strictly sequential — plan-review decision Q1).

## Instructions
1. In `internal/mode/details/model.go` (Dependencies pane handling, ~lines 231-250): make `DetailActionScrollUp`/`DetailActionScrollDown` only call `moveRelatedSelection(...)` and return WITHOUT emitting `OpenRelatedIssueIntent`. Cursor movement no longer triggers a reload.
2. Wire Enter to open the highlighted issue using the EXISTING hardcoded Enter handling, NOT a new config action (plan-review decision Q5). Detail-context Enter is special-cased at internal/mode/details/model.go:178-199 (it is NOT keymap-driven), and a no-op stub already exists at ~187-189 for Enter while the Dependencies pane is focused. Make that stub emit `OpenRelatedIssueIntent{IssueID: <selectedRelatedIssue>.ID}` so the shell handler (internal/app/model.go:683-699) performs the reload + scroll reset + focus move it already does. Do NOT add a new DetailContext config action — that would diverge from how detail-Enter already works. Confirm line numbers against the file.
3. Visual distinction — concrete two-marker spec (plan-review decision Q4). Note: an issue with a parent already renders its OWN row inside the Structure group (parentChildSiblings includes self), and the cursor defaults to the subject's row, so today cursor == subject; after decoupling they diverge and BOTH must be legible:
   (a) Cursor / pending row: a background-highlight style (add a style in internal/ui/styles), distinct from the current `› ` chevron — it must read as "movable cursor, not yet loaded".
   (b) Subject row, when it appears in-list (e.g. in the Structure group): a separate persistent marker (a distinct glyph, or keep the existing chevron) so "loaded subject" and "pending cursor" are simultaneously distinguishable.
   (c) When the subject is NOT in-list (the Children group never contains self; a top-level epic has no Structure group): no subject marker is needed — the subject lives in the Content/Metadata panes. Document this case.
   Encode each of (a)/(b)/(c) as a golden assertion.
4. Confirm Esc still returns to the prior browse mode (internal/app/model.go ShellActionEscape) — no regression.
5. Confirm no regression to Search-mode preview: scope is Detail mode only. Verify the shared `uidetails` rendering changes don't break Search goldens.
6. Update affected golden files.
