## Context

The current `scale_parity_integration_test.go` exists because the 50-cap was hardcoded in the Repository layer (`defaultLeanClosedLimit = 50` at `internal/repository/beads/lean_reads.go:19`). To exercise the cap path, the test required a fixture with >50 closed issues — hence the scale fixture (75 closed) and the `BWB_SCALE_FIXTURE=1` gate.

After iwvm lands, the cap is a caller-controlled parameter (`opts.ClosedLimit`). Any fixture can exercise the cap path by passing a limit smaller than its closed count.

## What this task does

Once iwvm.1..iwvm.4 are merged, rework the parity test surface so the cap behavior is exercised against the **minimal fixture** (or a fresh tiny fixture) with an explicit `ClosedLimit` chosen smaller than that fixture's closed count.

### Concrete shape

1. Drop the `closedCapForTest = 50` constant in `sort_parity_integration_test.go:50`. The cap is now per-call.
2. Drop the two `t.Skipf` precondition guards in `scale_parity_integration_test.go:53, 134` — they exist only because the test was scale-only.
3. Rewrite both cap-path tests (`TestScaleParity_DoneColumnCapEngagement`, `TestScaleParity_CapEngagement_VsBdCount`) to:
   - Use a small fixture (minimal anchor, or a per-test seeded one with ~3 closed).
   - Pass an explicit `ClosedLimit` (e.g. 2) via `DashboardOptions`.
   - Assert: `len(cols.Done.Issues) <= ClosedLimit`, `cols.Done.Total == bdClosed`, `cols.Done.TotalIsExact == false`.
4. Decide whether `scale_parity_integration_test.go` should be deleted entirely (folded into regular sort/count parity files), or kept for genuinely scale-only properties (e.g. cardinality warnings on >500 active, search corpus depth). If kept, rename to clarify what makes it scale-only post-iwvm.
5. Update `BWB_SCALE_FIXTURE` env gating so that **all remaining scale-only tests** still skip cleanly when unset, but cap-engagement tests no longer require it.

## Why this is its own task

iwvm.6 ("Parity contract scenario for opts.ClosedLimit") covers the **new** scenario in the contract suite. This task covers **cleaning up the legacy cap-path test infrastructure** that was built around the hardcoded 50. They're complementary: iwvm.6 adds the new assertion, this task retires the obsolete one.

## Done when

- `closedCapForTest = 50` constant is gone or renamed to reflect that it's a test-local choice, not a contract.
- Cap-engagement tests run on `mise run test:integration` (no `BWB_SCALE_FIXTURE` requirement).
- No `t.Skipf` in the cap-path assertions.
- If `scale_parity_integration_test.go` survives, every test in it has a documented scale-only reason that is not about the 50 cap.
