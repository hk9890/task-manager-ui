## Description
User feedback confirms search mode is currently not acceptable. The mode looks visually poor, does not feel like the Perles search workflow, and is effectively broken as an operator experience. Specific problems reported: there is an unwanted search-box style screen, the overall layout looks bad, switching into search with `2` is awkward compared to the expected Perles-style quick toggle, and returning cleanly to the dashboard is not obvious/reliable enough in practice.

## Evidence
- User feedback after direct runtime use
- Current app requires `2` for mode switch instead of expected quick search toggle flow
- Search screen visual design does not meet product quality expectations
- User reports search mode is “completely broken” and not working well at all

## Instructions
1. Rework search mode to feel like a professional operator workflow rather than a rough form screen.
2. Use Perles as a UX reference for how entering/searching/exiting search should feel, without copying architecture.
3. Fix the mode-switch interaction so entering and leaving search is smooth and predictable.
4. Ensure search results, filters, focus, and keyboard flow are all usable in real runtime.
5. Add/update automated coverage plus a full-app verification path for the search workflow.

## Files to Inspect
- `internal/mode/search/`
- `internal/ui/search/`
- `internal/app/model.go`
- related search/app tests and goldens

## Acceptance Criteria
- [ ] Search mode is visually coherent and no longer presents as a broken search-box workflow
- [ ] Entering search and returning to board are smooth and discoverable in real use
- [ ] Search interaction works in runtime: query/filtering/focus/results/navigation
- [ ] Search mode feels materially closer to Perles UX expectations
- [ ] Automated tests and full-app verification cover the intended flow
