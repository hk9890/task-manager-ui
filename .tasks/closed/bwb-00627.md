## Description

Remove the full-screen loading takeover from the Board surface. Replace the single `m.loading` flag with per-column loading state so:

- Cold start: header + tab bar + 4 column frames render immediately, each column shows skeleton rows.
- Refresh: stale rows stay on screen; the only signal is the global header spinner from `beads-workbench-0x36.6`.
- Atomic swap: when the LAST of the 3 parallel gateway calls returns, all 4 columns swap to new data at once (matches current `maybeCompose` semantics — see "Out of scope" below).
- Per-call errors render inline in the affected column(s); other columns keep working.

Depends on `beads-workbench-0x36.1` (SkeletonRow helper). The header spinner from `beads-workbench-0x36.6` provides the in-flight signal but is NOT a hard prerequisite — this task can land before the spinner; the only visible effect is a missing refresh signal until the spinner ships.

## Out of Scope (explicit)

- Per-call partial composition (one of the 3 calls landing → only its columns swap) is NOT in scope. Today `maybeCompose` waits for all 3 calls then writes all 4 columns at once (`internal/mode/board/model.go:343-398`). This task preserves that semantics; "atomic per region" here means the 4-column block.

## Files to Modify

- `internal/mode/board/model.go` — drop the `m.loading` short-circuit in `View()` (lines 226-228). Replace `m.loading` with per-column `loading bool` + `err error` on `columnData`. Update `startReload` (line 305) to mark all 4 columns loading at reload start. Update `maybeCompose` (lines 343-398) to clear all 4 loading flags atomically and copy per-call errors into the affected columns. Replace the current full-board error path (lines 138-167) with per-column error rendering.
- `internal/ui/board/board.go` — render rule per column: skeleton rows when `loading && len(issues)==0`, stale issues when `loading && len(issues)>0`, inline error row when `err != nil`.
- `internal/mode/board/model_test.go` — assertions per the criteria below.

## Instructions

1. Extend `columnData` with `loading bool` and `err error`. DO NOT add `loadedOnce` — `loading && len(issues)==0` is sufficient to distinguish cold-start vs refresh per column.
2. In `Init()` and `startReload()`, set `loading=true` and `err=nil` on all 4 columns. Remove the global `m.loading` flag if no consumer remains (verify with `grep -RIn 'm\.loading' internal/mode/board/`).
3. In `maybeCompose`, when the atomic composition runs: clear all 4 `loading` flags; copy each per-call error into the relevant column(s)' `err` field (Not Ready + Ready ← ReadyExplain error; In Progress ← in_progress query error; Done ← closed query error); replace each column's rows atomically.
4. Remove the `if m.loading { return loading.View(...) }` short-circuit in `View()`.
5. In `internal/ui/board/board.go`, render rule per column:
   - `loading && len(issues)==0` → render ~6 skeleton rows via `loading.SkeletonRow`.
   - `loading && len(issues)>0` → render `issues` unchanged (header spinner from `0x36.6` is the only refresh signal).
   - `err != nil` → render an inline single-line error row (e.g. `"⚠ load failed: <short error>"`) at the top of the column's content area; render the rest as skeleton (cold) or stale issues (refresh) per the same rules.
6. Keyboard input during full cold-start (all 4 columns empty) MUST be a no-op, not a crash. Audit `moveRow` / `currentSelection` to confirm zero-length issue arrays are handled safely.
