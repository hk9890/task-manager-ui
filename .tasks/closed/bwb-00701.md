## Description
Create the annotated v0.2.0 tag, trigger the automated release workflow, and verify that the GitHub release assets are published successfully.

## Instructions
1. Confirm `git tag --list "v0.2.0"` is empty and `HEAD` is the intended release commit.
2. Re-check that the exact `HEAD` commit to be tagged has a successful `CI` workflow run; if the changelog/docs commit created a new SHA without green CI, wait for CI before tagging.
3. Create the annotated tag with `git tag -a v0.2.0 -m "bwb v0.2.0"`.
4. Push the branch if needed, then push the tag with `git push origin v0.2.0` to trigger `.github/workflows/release.yml`.
5. Watch the `Release` GitHub Actions workflow until it succeeds.
6. Verify publication with `gh release view v0.2.0`, `gh release view v0.2.0 --json assets`, and `git ls-remote --tags origin "refs/tags/v0.2.0"`.
7. Confirm the release includes the expected assets documented in docs/RELEASING.md and `.goreleaser.yaml`, including:
   - `bwb_0.2.0_linux_x64.tar.gz`
   - `bwb_0.2.0_linux_arm64.tar.gz`
   - `bwb_0.2.0_macos_x64.tar.gz`
   - `bwb_0.2.0_macos_arm64.tar.gz`
   - `bwb_0.2.0_checksums.txt`
8. If tag publication or asset generation fails, do not silently retry. Record the failure and decide explicitly whether recovery means deleting the unpublished/bad tag and retrying, or issuing a new version such as v0.2.1.

## Acceptance Criteria
- [ ] Annotated tag `v0.2.0` exists locally and on origin
- [ ] The exact tagged SHA has green `CI` provenance before publication
- [ ] The `Release` workflow completes successfully for `v0.2.0`
- [ ] The GitHub release is visible and includes the expected archives plus checksums
- [ ] Any publish/asset failures are tracked separately before the issue closes
