## Description
Add regression coverage for the rewired `e` (edit issue) flow so the goroutine-vs-`tea.Exec` bug cannot reappear. teatest drives the real Bubble Tea program; `execMsg` is processed inline by the program loop, and a fake `tea.ExecCommand` does no real TTY I/O — so teatest can drive the full flow headlessly.

Note: existing `FakeEditor`-based tests in `model_test.go` / `scenario_workflow_test.go` are migrated to the new API under beads-workbench-h3ql. This task only ADDS new teatest coverage and updates the fake — do not re-edit those migrated tests here.

## Instructions
1. Update `internal/testing/fakes/editor.go` to match the new `editor.Service` API (`PrepareDocument` + `ApplyEdits`). Provide a fake satisfying the full `tea.ExecCommand` interface (`Run`/`SetStdin`/`SetStdout`/`SetStderr` — the setters are no-ops; Bubble Tea calls all three before `Run`). `Run()` writes configured "edited" content to the temp path captured at construction and returns the configured error (nil or non-nil variant).
2. Add a teatest test in `internal/app/` that, using the fake gateway and the injected fake `ExecCommandFactory`:
   - sends an `e` keypress on a selected issue
   - asserts the fake `ExecCommand.Run` was called exactly once with the path returned by `PrepareDocument`, and `ApplyEdits` reads back the fake's edited content
   - asserts `UpdateIssue` is recorded on the fake gateway and the "Updated issue X" success toast renders with detail reload
3. Cover the no-change path ("No changes saved for issue X" toast) and the editor-error path ("Failed to edit issue X" toast).
4. Follow the fake-vs-real discipline in `internal/testing/fakes/doc.go`.
