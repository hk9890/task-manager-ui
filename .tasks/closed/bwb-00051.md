Discovered from live user review of the current detail UI after the earlier metadata-interaction follow-up was shipped. Expected: when the Metadata pane is focused, the pane border already communicates focus, so metadata field selection should use the same shared selected-row/cursor treatment as Dependencies/dashboard (same alignment and same blue cursor/selection style) without adding a redundant chevron banner. Actual: in detail mode on `beads-workbench-t7o.2`, focusing Metadata shows a top line `› Editable: Status, Priority`, which duplicates the pane-focus signal and introduces an extra chevron that does not match the rest of the UI. The selected metadata field is also rendered with an inline `›` prefix (`› Priority: P1` / `› Status ...`) rather than with the same shared selected-row treatment used in Dependencies/dashboard, so text alignment shifts and the selection affordance feels inconsistent. Repro: 1) `go build -o /tmp/bwb ./cmd/bwb` 2) run `BD_NON_INTERACTIVE=1 /tmp/bwb` from the repo root 3) open `beads-workbench-t7o.2` in detail mode (`3`) 4) focus Metadata with `RIGHT` 5) observe the `› Editable: Status, Priority` banner and inline-chevron selection styling while moving between Status/Priority. Impact: metadata interaction works functionally but feels visually inconsistent and less polished than the rest of the selection system.

## Acceptance Criteria
- [ ] The redundant `› Editable: Status, Priority` banner is removed or replaced with a non-chevron affordance that does not compete with pane focus.
- [ ] Metadata field selection uses the same shared selection/cursor component style as Dependencies/dashboard, including consistent color treatment and stable text alignment.
- [ ] Moving between Status and Priority does not shift row alignment with ad-hoc inline prefixes.
- [ ] Real-app review confirms Metadata selection feels visually consistent with the rest of the app.
