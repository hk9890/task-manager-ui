## Description
When opening `beads-workbench-2shr` in the real `bwb` app, the detail screen keeps the three-pane Dependencies / Content / Metadata layout even at common terminal widths (for example 80x24 and 100x30). The content pane collapses to a tiny sliver, headers truncate, and the main issue description becomes effectively unreadable.

## Expected Behavior
The detail view should adapt at narrower widths so the main issue content remains readable, with a responsive fallback instead of forcing the full three-pane layout.

## Actual Behavior
At 100x30 the content rail is already heavily compressed, and at 80x24 the content header truncates to `Content · 4 co...` while the description collapses into narrow wrapped fragments and ellipsis rows. The layout looks visibly broken even though the app technically renders.

## Reproduction
1. `go build -o /tmp/bwb ./cmd/bwb`
2. Run `env BD_NON_INTERACTIVE=1 /tmp/bwb` from the repo root.
3. Open epic `beads-workbench-2shr` in detail mode.
4. Observe the detail view at widths like 100x30 or 80x24.

## Impact
This is a user-visible runtime UI regression in the main detail workflow. It makes real issue inspection difficult on normal terminal sizes and means the current responsive detail layout does not meet the runtime UI readability bar.

## Instructions
1. Review the detail renderer width logic in `internal/ui/details/details.go`, especially the unconditional three-pane path in `Render` and `splitThreePaneWidths`.
2. Restore a readable responsive fallback for narrower widths instead of always forcing Dependencies + Content + Metadata together.
3. Update focused detail renderer tests and any goldens that currently encode the overly dense layout.
4. Re-verify with built-binary captures at representative widths (at minimum 80 columns and 100 columns).

## Files to Modify
- `internal/ui/details/details.go`
- `internal/ui/details/details_test.go`
- `internal/ui/details/testdata/*.golden`

## Acceptance Criteria
Opening `beads-workbench-2shr` in the built app produces a readable detail screen at representative narrow/medium terminal widths, and focused detail rendering tests reflect the intended responsive fallback behavior.
