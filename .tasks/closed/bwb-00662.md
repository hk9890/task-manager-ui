## Description

Run the expanded `RunReadContract` from [[8qw9.2]] against `FakeBeadsGateway` and fix every divergence. After this task, the fake passes the SAME contract suite as the real bd-backed implementation on the SAME fixture inputs. Future tests using the fake automatically inherit contract fidelity.

## Instructions

1. Run `go test -tags=integration ./internal/gateway/beads/contract/...` and identify failures specific to `TestFakeGatewayReadContract`.
2. For each failure, decide:
   - **Fake gap**: real bd does X, fake does Y. Update the fake to match real bd. Add a brief comment near the fix explaining what was wrong (helps future readers understand the divergence pattern).
   - **Contract gap**: assertion is too strict for behavior bd never guarantees. Soften the assertion or document the carve-out in [[8qw9.1]]'s interface doc.
   - **Real bd quirk**: assertion expected something bd doesn't honor (caught during the audit). Document in interface.go; fake matches the documented quirk.
3. Read existing fake test helpers (`internal/testing/fakes/beads_gateway.go`) — many tests construct fake state inline. Resist the temptation to add complex auto-behavior to the fake; prefer explicit "if you want this behavior, prime these fields" patterns that match real bd's observable surface.
4. Once all fake-side contract sub-tests pass, run `mise run test:integration` end-to-end. Existing tests that use the fake should still pass — if they break, the fake's prior behavior was wrong AND those tests relied on the wrong behavior. Fix both.
5. Document material fake updates in the "Fake drift log" section established by [[8qw9.5]] (if it lands first) or stub the log in this task and reference it.

## When fake-vs-real divergence reveals a real bug

Apply the "Discovered-during-audit bug discipline" from the parent epic [[8qw9]]. If during the fake audit you find that:

- The fake was returning one behavior, and bwb consumers RELIED on that incorrect behavior, then **real bd would not satisfy those consumers** — that's a user-visible bug hiding behind the fake's lie.
- OR: real bd returns something the fake didn't simulate, and a consumer would mis-handle it if it ever appeared (e.g., null fields, empty lists, unexpected sort).

In either case:

1. **File a separate bug** via `bd create --type=bug --title="Gateway fake masked real-bd behavior: <consumer> would <X> when bd returns <Y>"`.
2. **Bug body MUST include**: "Verify in app (`mise run bwb` against a representative DB) that this fake-masked behavior actually causes user-visible incorrect behavior. If it does not, close as 'not a bug' with rationale. If it does, fix at the appropriate consumer."
3. **Link the bug back to this task** via `bd dep add <bug-id> beads-workbench-8qw9.3`.
4. **Update the fake to match real bd FIRST** (part of this task), THEN the fixed consumer will pass against the updated fake in its own bug-fix ticket. This is the loop: fake reflects reality → tests against fake catch real bugs.
