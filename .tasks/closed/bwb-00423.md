## Description
Triggering the Update Status dialog from Detail mode opens it only after an async
status-catalog load. If the user presses ESC (or any superseding key) during that
sub-second window, the app (a) silently switches Detail -> Board via the Esc handler,
then (b) opens the status dialog ORPHANED over the Board once the catalog arrives.

## Repro
1. go build -o /tmp/bwb ./cmd/bwb; seed internal/testing/e2e/embeddedfixture/setup.sh.
2. Open an issue detail; focus the Metadata pane; press Enter on Status (dispatches loadStatusCatalogForIssueCmd).
3. Before the dialog appears (sub-second on embedded dolt), press ESC.
Observed: header shows "Board · Selected: <id>" with an "Update Status <id>" modal painted over the Board.
Control: with a longer wait so the dialog opens in Detail first, ESC cancels cleanly (no orphan) — confirms an async-open race.

## Root cause (internal/app/model.go — verified)
- Enter on Status -> m.detail.ConsumeOpenStatusDialogIntent() -> loadStatusCatalogForIssueCmd(...) (line 667; search-mode sibling line 726).
- The modal is opened later by the statusCatalogLoadedMsg handler (lines 558-568), which sets showActionModal=true UNCONDITIONALLY (no mode check). The create/update path has the same shape in the mutation-catalogs-loaded handler (lines 547-556).
- The ShellActionEscape handler (line 788) pops Detail->Board with no awareness of an in-flight dialog-open.
- During the load window showActionModal is still false, so ESC BYPASSES the modal intercept (model.go:370) and reaches ShellActionEscape (788). This ordering is load-bearing for the fix.

## Instructions
1. Add a pending-dialog guard on Model (a small struct/flag, e.g. pendingDialog struct{active bool; kind mutationKind; issueID string}), set wherever an async dialog-open Cmd is dispatched: the loadStatusCatalogForIssueCmd sites (667, 726) AND the loadMutationCatalogsCmd (create/update) sites.
2. CENTRALIZE guard-clearing at a single choke point. Recommended: clear the guard at the TOP of the tea.KeyMsg branch for any key that is not a catalog-loaded message; and additionally guard the modal-open so the catalog-loaded handler only opens the modal when the guard is still active. Do NOT scatter clears across the ~8 mode-change sites (model.go:749-799 + the ActionRequest path ~628-639) — a single choke point prevents missing one.
3. In the statusCatalogLoadedMsg handler (558) AND the mutation-catalogs-loaded handler (547): only open the modal if the guard is still active (and the kind/issue matches); otherwise drop the result with no modal.
4. In ShellActionEscape (788): if a dialog-open is pending, consume ESC as "cancel the pending open" (clear the guard) and DO NOT pop the mode.
5. CREATE uses an empty IssueSummary{} (empty issue ID), so the guard must key on a mode/kind flag, NOT issue-ID matching, for the create path.
6. Leave the synchronous open paths (priority/comment/close, which set showActionModal directly) unaffected.
