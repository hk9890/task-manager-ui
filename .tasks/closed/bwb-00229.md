## Outcome (post-kxci follow-up)

The caching backend's warm-restart "sub-second first paint" mechanism (kxci.7) is correct but only delivers full benefit when memory.Repository already has the dashboard's underlying issues. Currently the per-ID cache only populates on `Issue(id)` calls; the Dashboard view itself lives in a separate `dashboardCache` field that is NOT persisted.

Net effect: a fresh user (or short prior session with little navigation) gets cold Dashboard on warm restart even though the kxci.8 cross-session resolver finds the prior cache file.

## Approaches

Option A — persist dashboardCache as a sibling file:
- `repo.jsonl.dashboard.json` adjacent to `repo.jsonl.manifest.json`
- filestorage.SaveWithDashboard / LoadWithDashboard
- caching.SaveNow serialises current dashboardCache; caching.Hydrate restores it if hash matches

Option B — pre-populate per-ID cache from Dashboard():
- When caching.Dashboard fetches from backing, also fetch each issue's detail in background and store
- Higher cost; bigger files; only useful for issues that fit in the dashboard

Option C — extend Repository or add Dashboard-summary cache:
- Persist DashboardData directly (it's an IssueSummary slice, smaller than full details)
- Cheapest persistence; matches what's actually rendered

Recommended: Option C. Smallest disk footprint; matches the actual data the dashboard renders; simplest hydration logic.
