## Outcome
bd CLI calls made by the workbench stop contending under burst and stop re-running when the beads DB is unchanged. The >4s tail latency is eliminated and steady-state board refreshes become near-instant cache hits.

## Root cause (monitoring analysis)
project-analyze-monitoring-data over a ~24h window of ~/.local/state/bwb/bwb-*.log:
- Solo 'bd count' measures ~0.28s (process spawn + cold-open of embedded Dolt).
- Board auto-refresh (internal/app/model.go:32, refreshTickInterval=60s) fans out 4 bd processes per tick in one tea.Batch (internal/mode/board/model.go:357, pendingResults=4).
- The gateway runner runs reads fully concurrently and uncapped (internal/gateway/beads/runner.go:164, shared RLock).
- 4 cold Dolt opens contend super-linearly: median bd call 1.25s, p95 2.2s, max 6.1s across 2578 logged calls.
- 99% waste: session 8c657fd1 made 1063 gateway calls but only 7 distinct argv -- nearly every refresh returned byte-identical data.
- Zero real operational failures in the window; all log ERRORs were test-suite negative cases. Latency is the only genuine finding.

## Scope -- 2 implementation tasks, single PR
- wcgy.1 -- Cap concurrent bd subprocesses with a bounded semaphore in runner.go. Eliminates the 4-6s contention tail.
- wcgy.2 -- In-process gateway read cache keyed on resolved argv, invalidated by a DB change-token (stat mtime of .beads/last-touched), plus per-argv singleflight. Unchanged token -> cached bytes, no bd spawn; covers board/detail/search read surfaces.

The 60s auto-refresh tick is RETAINED -- it becomes the cheap trigger that runs the token check to catch external agent writes.

## Out of scope
Collapsing the 4 board refresh commands to fewer was considered (originally "#3") and DROPPED. 'bd query status=closed --json --limit N' returns a bare JSON array with no total-count field, so the closed-count is not derivable from a capped result set. The separate 'bd count --by-status closed' call is load-bearing -- it is the fix for closed issue beads-workbench-ssom (Done column must show the real DB count, not the capped slice size). The batch stays at 4 commands; the win comes from caching, not from cutting commands.

See NOTES for the empirical-verification audit trail (token confirmation, #3 drop rationale).

Note: the board's 4 refresh commands are 4 DISTINCT argv, so singleflight does not collapse them -- bounding that burst is #1's job (semaphore).
