## Description
A second dashboard provider is expected soon, so the dashboard boundary should be explicit, validated, and narrow before that work lands.

## Instructions
1. Add a standalone validation entry point in `internal/dashboard` (for example `ValidateDefinitions`) that validates provider output before the board uses it. Do not put validation logic on `Provider` and do not bury it inside `internal/mode/board`.
2. Define and enforce the minimum invalid states the board must reject: provider returns zero definitions; dashboard definition has empty ID or Title; dashboard has zero sections; section has empty ID or Title; section uses an unsupported `QueryType`.
3. While tightening `internal/dashboard/definition.go`, make the query contract explicit enough that invalid or unsupported query configurations are detectable without inventing new machinery. If exact payload-shape validation is not representable with the current struct shape, document and enforce the narrower supported rule instead of guessing.
4. Update `internal/mode/board` to validate definitions before use and to fail locally and predictably on invalid provider output, without adding plugin machinery or arbitrary query execution.
5. Add tests for empty, invalid, and valid multi-dashboard provider outputs.
6. Keep built-in definitions simple and aligned only to currently supported gateway query types.

## Files to Modify
- `internal/dashboard/definition.go` — clarify contract and validation entry points
- `internal/dashboard/*.go` — add standalone validation helpers and tests as needed
- `internal/dashboard/builtin_provider_test.go` — extend built-in provider coverage
- `internal/mode/board/model.go` — validate provider output before use
- `internal/mode/board/model_test.go` — add board-side contract coverage

## Acceptance Criteria
- [ ] `internal/dashboard` exposes a standalone validated contract for dashboard definitions
- [ ] Board loading fails predictably for zero-dashboard, empty-id/title, zero-section, and unsupported-query-type cases
- [ ] Built-in provider behavior remains covered and unchanged for valid definitions
- [ ] No plugin registry, BQL surface, or arbitrary query execution is added
