Finding 2 from the ipq7 audit (PERFORMANCE, not correctness). Hydrate restores dashboardCache and sets dashboardDirty=false but never restores lastDashboardClosedLimit (stays 0 from New). In production the first Dashboard() call uses ClosedLimit=sectionItemCapacity() (>=1), so limitChanged is always true -> cache miss -> re-fetch. Correct data, but the v2-header fast-paint restore is effectively dead code: the cache never serves the persisted first paint.

Fix: add lastDashboardClosedLimit to filestorage.V2Header (schema bump), persist it in SaveNow, restore it in Hydrate. This is SAFE only because ipq7's boundClosed(serve) fix now truncates any persisted superset to ClosedLimit on serve — without that, restoring the limit would re-expose the stale-superset correctness bug at hydrate time. Prerequisite: commit 12fef25.
