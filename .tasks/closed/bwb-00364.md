## Description

The detail-mode responsive layout breaks at narrow terminal widths:

1. `internal/ui/details/details.go:181-208` (`renderResponsiveLayout`) — bottom-row widths summed (`dependenciesWidth + gap + metadataWidth`) can exceed `width` when `splitResponsiveBottomWidths` clamps `available < 2` to 2 (line 256). Causes wrap below ~22 cols.
2. `internal/ui/styles/section.go:42-46` (`buildFormTopBorder`) — when `cfg.Width < 3`, the dash count goes negative and is clamped to 1; the rendered top border becomes wider than `innerWidth + 2`. Bottom border has the same issue. Column alignment breaks on terminals <40 cols.

Major finding U2 from the 2026-05-16 audit.

## Files to Modify

- `internal/ui/details/details.go`
- `internal/ui/styles/section.go`
- new test: width-parameterized golden under `internal/ui/details/testdata`

## Instructions

1. Add a minimum-width threshold (suggest 30 cols) at the top of `details.Render`. Below the threshold, return a single-line `"Terminal too narrow"` message instead of attempting the responsive layout. This is the simplest safe fallback.
2. In `styles.section.Render`, refuse to render and return a similar "too narrow" string when `cfg.Width < 6` (smallest sensible bordered box).
3. Add tests:
   - Detail render at width = 20 → assert the fallback message is returned
   - Detail render at width = 30 → assert no panic, no wrap
   - Detail render at width = 80 / 120 / 180 (existing goldens) — still pass
4. Verify no other consumer of `FormSection` is broken by the new minimum-width guard.

## Acceptance Criteria

- [ ] Detail render returns a clear fallback below minimum width
- [ ] `FormSection` does not produce malformed borders at small widths
- [ ] Tests cover at least width = 20 and the existing golden widths
- [ ] Full quality gate green
