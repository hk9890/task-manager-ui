## What

In internal/mode/search/model.go at the SearchActionReload keyboard handler (currently line 243-244), guard against re-entering triggerSearchPreservingSelection while a search is already in flight, and emit a debug log when the guard fires.

## Why

Search mode has the same bug as board mode but with a different failure mode. The keyboard handler calls m.triggerSearchPreservingSelection() unconditionally, while Reload() (line 380-385) and AutoRefresh() (line 388-396) both already check `if m.loading`.

**Search-specific failure mode:** unlike board (where pendingResults underflows), search has no counter to corrupt. Instead, when 'r' is pressed twice rapidly:
- Both keypresses invoke triggerSearchWithAnchor.
- Both set m.loading=true (idempotent).
- Both overwrite m.pendingSelectionAnchor and m.appliedQuery (the SECOND call wins).
- Both dispatch loadSearchCmd → two concurrent gateway queries.
- Their loadSearchResultMsg responses land in unknown order; the LAST one to arrive replaces page state.
- User sees results from the older/stale query, or results that don't match the displayed appliedQuery.

The debug log makes the suppression visible in `bwb --debug` traces per docs/MONITORING.md.

## Change

Route the keyboard handler through Reload() (which already has the guard) and add the debug log inside Reload (or inline both at the call site — equivalent):

```go
case m.keys.Match(config.SearchContext, config.SearchActionReload, msg):
    return m.Reload()

// in Reload():
func (m *Model) Reload() tea.Cmd {
    if m.loading {
        m.logger.Debug("manual search refresh suppressed; refresh already in flight",
            "component", "search",
            "trigger", "search-manual")
        return nil
    }
    return m.triggerSearchPreservingSelection()
}
```

Note: confirm `m.logger` exists on the search model; if not, plumb a logger through or use slog.Default() consistent with the board model pattern.
