## Description
For ~1s after a search returns FEWER results than the previously displayed list, the
Results pane briefly shows stale rows from the prior list below the new (shorter) result
set — including non-matching issues. Header count is correct; rendered rows are not.
Self-heals once the async search settles (~1.2s). Reproduced 2/2 immediate captures.

## IMPORTANT — confirm the real cause BEFORE editing (initial hypotheses were falsified)
A code review falsified the two obvious causes:
- The Results box ALREADY clamps + pads: styles.FormSection (internal/ui/styles/section.go:59-78) truncates content to innerHeight and pads short content with blank, right-padded lines. A short/unpadded body therefore CANNOT leave stale rows — do NOT "add padding", it is already done.
- The page swaps ATOMICALLY: searchLoadedMsg replaces m.page (internal/mode/search/model.go ~line 140); state.Results = m.page.Results, so a frame renders EITHER the old page OR the new one, never concatenated. The model does NOT expose old+new rows simultaneously.
Most likely the defect is a RENDERER / terminal repaint artifact when the TOTAL rendered frame height shrinks between paints (lines below the new shorter frame are not cleared) — i.e. at the Bubble Tea frame-diff / composed app View level, OUTSIDE internal/ui/search/search.go and internal/mode/search/model.go.

## Repro
1. go build -o /tmp/bwb ./cmd/bwb; seed a fixture with >12 issues.
2. Enter Search; submit a query yielding few results (from a larger/default list -> e.g. a 4-result query).
3. Capture the Results pane immediately after the new count appears (scripts/capture_bwb_screen.py). Clean after a ~1.2s settle.
Observed: top N new rows, blank rows, then a second block of stale non-matching rows (e.g. "Migrate ledger", "Unlabeled open task").
