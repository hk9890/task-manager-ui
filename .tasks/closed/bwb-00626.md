## Description

Apply the non-blocking loading pattern to the Search surface. The full-screen loading takeover lives in the UI layer (`internal/ui/search/search.go:68-70`), NOT the mode model. The mode model already exposes a `Reloading` field (`internal/mode/search/model.go:46`) that distinguishes refresh from cold-start.

After this task:
- Cold start (no prior results): search input + skeleton result rows (no full-screen takeover).
- Refresh (results present): stale results stay visible; header spinner from `beads-workbench-0x36.6` is the only signal.
- User can type/edit the query at any time (already true today — covered as a regression check, not new work).

Depends on `beads-workbench-0x36.1` (SkeletonRow helper).

## Files to Modify

- `internal/ui/search/search.go` — remove the early return at lines 68-70: `if state.Loading && len(state.Results) == 0 { return loading.View(...) }`. Replace with skeleton-row rendering in the result-list section.
- `internal/mode/search/model.go` — verify (do NOT remove) that `m.loading` continues to be set on reload, since `internal/app/model.go:1258-1270` (`loadingStates()`) depends on it to drive the header spinner.
- `internal/ui/search/search_test.go` or `internal/mode/search/model_test.go` — assertions per the criteria below.

## Instructions

1. Remove the loading short-circuit at `internal/ui/search/search.go:68-70`. The rest of `Render()` should produce the normal search layout (input + result list).
2. In the result-list rendering:
   - `state.Loading && len(state.Results) == 0` → render ~6 skeleton rows via `loading.SkeletonRow`.
   - `state.Loading && len(state.Results) > 0` (or `state.Reloading`) → render `state.Results` unchanged.
   - Idle → unchanged.
3. Verify (regression-style) the user can type/edit while a query is in flight — this should already be true since `handleKey` does not gate on `m.loading`.
4. Confirm `m.loading` is still set in the mode reload path so `loadingStates()` reports search activity to the header spinner.
