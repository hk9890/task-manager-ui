## Discovered during

[[8qw9.1]] (gateway contract audit, 2026-05-16).

## Expected vs actual

The domain model's `SortDirectionDescending` means "newest/highest first". The gateway translates this to `--reverse` passed to bd. However, **bd `--sort <field>` defaults to DESCENDING**; `--reverse` inverts it to ASCENDING. So `SortDirectionDescending` → `--reverse` → ascending order — the opposite of intent.

## Critical consumer impact

The board model calls `Query("status=closed", {SortBy: SortFieldClosedAt, SortOrder: SortDirectionDescending})` to load the Done column expecting newest-closed-first. Due to this bug it receives oldest-closed-first. **This means the Done column is currently displaying the wrong order in production.** The ssom fix landed Done.Total=452 (correct number) but the rendered list is the WRONG 50 issues (oldest, not newest).

## Verification required

Must verify in the running app (`mise run bwb` against this repo or another DB with ≥2 closed issues at different times) that the Done column shows oldest-closed-first rather than newest-closed-first. If confirmed visually, the fix is straightforward.

Orchestrator note: code-path verified by reading composer + board model + gateway argv assembly. Visual confirmation gated on user TTY.

## Suggested fix layer

Gateway (`internal/gateway/beads/read_gateway.go`, sort args logic). Invert the condition: add `--reverse` for `SortDirectionAscending`, NOT for `Descending` (since bd defaults to DESC, Descending should send no flag).

Tests that will need updating after fix:
- `TestBoardInitRealGatewaySubprocessArgvCardinality` (`internal/mode/board/model_test.go:988`) — pins the current (wrong) argv with `--reverse`.
- Any gateway test asserting argv shape for sort flags.
- The sort parity test (`internal/dashboard/parity/sort_parity_integration_test.go`) compared bwb against bd using THE SAME wrong flag — both sides agreed because both were wrong. After fix, both sides should still agree but on the CORRECT order.
