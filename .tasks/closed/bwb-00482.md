What I did: In search mode (embedded fixture, 5 issues), committed queries with Enter: 'blocked' (2 results), 'zzqqxxnomatch' (0 results), and empty/cleared (5 results). The Results pane header always showed '<n> partial' — '2 partial', '0 partial', '5 partial'.

Expected (source: docs/break-it.md Consistency; general UX): a completeness badge should help a user judge whether they are seeing all results. 'partial' intuitively means 'incomplete — there may be more'.

Actual — two coupled problems in the SearchResultCompleteness model:

1. MISLEADING LABEL. internal/gateway/beads/read_gateway.go:555-557 and 567-570 set Completeness = Partial precisely when 'returnedCount < requestedLimit || no limit' — i.e. when the result set did NOT hit the backend cap, meaning the user IS seeing everything available. The UI (internal/ui/search/search.go:365-366) renders that as the badge 'partial'. So a complete 2-of-2 result set is labelled '2 partial', which reads as 'incomplete'. The semantics are inverted from the word. Meanwhile the cap-hit case (MaybeMore) renders as 'capped'. Two badges ('partial', 'capped') both connote incompleteness and are applied to opposite situations.

2. DEAD ENUM + DEAD UI BRANCH. domain.SearchResultCompletenessExact (internal/domain/issues.go:73) and its UI branch (internal/ui/search/search.go:361-362 -> 'exact') are never reachable. The gateway only ever assigns MaybeMore or Partial (confirmed: grep of all non-test Completeness assignments; and the interface contract internal/gateway/beads/interface.go:443-446 explicitly documents only MaybeMore/Partial). 'exact' completeness can never be produced or displayed.

Severity: low. No functional break — it is a metadata badge plus an unreachable enum value. But the badge actively misinforms during normal search use, and the dead 'exact' value/branch is exactly the class of inconsistency the (now-closed) consistency-cleanup epic beads-workbench-5rvn targeted (this specific item was NOT in 5rvn's findings list).

Repro: bwb against embedded fixture -> search mode (f14 / ctrl+space) -> type 'blocked' + Enter -> Results header shows '2 partial' while displaying all 2 matches.

Suggested fix: either (a) relabel: 'partial' -> 'all' / 'complete' (and reconsider whether a badge is needed at all when complete), or (b) actually produce SearchResultCompletenessExact for the no-cap-pressure case and reserve 'partial' for a genuine partial state; either way remove whichever enum value/branch stays unreachable.
