## Description

PLACEHOLDER EPIC — captures scope so we don't lose context. Not actively planned yet.

The contract-hardening epic [[8qw9]] covers read methods only. The four write methods on `BeadsGateway` deserve the same treatment:
- `CreateIssue(input)`
- `UpdateIssue(issueID, input)`
- `CloseIssue(issueID, input)`
- `AddComment(issueID, input)`

Write methods have additional contract surface that reads don't:
- **Side effects**: a successful CreateIssue must produce an issue retrievable by ShowIssue with matching fields.
- **Idempotency / collision**: what happens if you Create with a colliding ID? Update a non-existent issue?
- **Confirmation prompts**: bd has interactive confirmations for some destructive operations; bwb passes `BD_NON_INTERACTIVE=1`. Document the assumed bypass.
- **Audit trail**: bd attributes writes to an actor (env var or git user.name); the gateway's `--actor` handling needs explicit contract.
- **Dolt write-ack**: bd's dolt backend has auto-commit policy (off/on/batch); writes may not be visible to subsequent reads in some modes. Contract must document the consistency model.

## Probable scope (when activated)

- Audit bd write behavior hands-on (mirror [[8qw9.1]] approach for the write side).
- Document each write method's contract in `interface.go`.
- Add a `RunWriteContract` suite that runs against fake AND real implementations.
- Extend `validatingGateway` (or add a write-side decorator) for post-write invariants like "the created issue is now visible to ShowIssue."
- Audit `FakeBeadsGateway`'s write methods against the new contract; close fidelity gaps.

## Out of Scope

Read-side contract (covered by [[8qw9]]).
