## Description

`internal/gateway/beads/runner.go:62, 103-107` uses a single `sync.Mutex` (`runMu`) held across the **entire** child-process lifetime — including JSON decode. Comment states this is needed because Dolt file-locks. But Bubble Tea fans out dashboard sections as concurrent `tea.Cmd`s; every one of them blocks behind the slowest read. The result is a ~4x slower dashboard load than the gateway is otherwise capable of, and overlaps with the parallel work being done in epic beads-workbench-au54 (which assumes concurrent reads are unblocked).

Note: a separate ticket (beads-workbench-au54 epic) plans to split this mutex into RWMutex as part of a larger refactor. This ticket exists so the fix is tracked under the audit epic too. May be closed as a duplicate of work landing under au54 — confirm before doing the work in isolation.

Major finding G2 from the 2026-05-16 audit.

## Files to Modify

- `internal/gateway/beads/runner.go`
- new test: `internal/gateway/beads/runner_concurrency_test.go`

## Instructions

1. Confirm with epic beads-workbench-au54 whether this is being addressed there. If yes — close this as duplicate. If not:
2. Replace `sync.Mutex` with `sync.RWMutex`. Read commands (`list`, `ready`, `blocked`, `search`, `show`, catalog) take RLock; write commands (`create`, `update`, `close`, `comment`) take Lock. Identify the read/write split via existing command names in `runner.go` / `read_gateway.go` / `writes.go`.
3. In `RunJSONInto`, release the mutex BEFORE the JSON decode — the lock is only needed for the executor invocation. Restructure if necessary.
4. Add `runner_concurrency_test.go` asserting two read invocations dispatched in parallel measurably overlap in time. Use a fake executor that sleeps deterministically and assert the wall-time is less than 2x the sleep.
