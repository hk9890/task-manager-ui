## Description
When the user presses Enter on a dependency in the Details view to drill into a related
issue, focus must follow the loaded issue's rail: stay on the Dependencies rail when the
drilled issue has dependencies, and move to the Content pane only when its rail is empty.
Today focus always ends up on Content because the optimistic placeholder (no deps) trips
`clearBrowserPanel()` during the in-flight window and nothing restores it.

## Instructions
1. In the Enter-drill handler (`internal/app/model.go` ~683-703, `OpenRelatedIssueIntent`),
   do not let the optimistic placeholder reset focus away from the Dependencies rail.
2. In `internal/mode/details/model.go`, ensure `clearBrowserPanel()` /
   `syncBrowserPanel` (lines ~511-544) do not force `FocusPane = Content` based on the
   transient empty placeholder; the focus decision must be driven by the loaded detail.
3. On the real `detailLoadedMsg` path (`internal/app/model.go` ~454-473 ->
   `ApplyLoadedDetail`), apply the rule: non-empty rail keeps focus on Dependencies,
   empty rail moves focus to Content.
4. Preserve existing behavior for non-drill loads (selection change from board/search)
   and for the cold-start skeleton.

## Files to Modify
- `internal/app/model.go`
- `internal/mode/details/model.go`
- `internal/app/model_test.go` (extend `TestAppHandlerOpenRelatedIssueIntentPerformsReloadFocusMoveAndScrollReset`)
- `internal/mode/details/model_test.go`

## Acceptance Criteria
- [ ] Drilling (Enter) into an issue WITH dependencies keeps focus on the Dependencies rail after load.
- [ ] Drilling (Enter) into a leaf issue (empty rail) moves focus to the Content pane.
- [ ] Focus is not flipped to Content during the optimistic placeholder phase when the target has deps.
- [ ] Selecting issues from board/search still lands focus per existing behavior (no regression).
- [ ] Relevant tests pass (`mise run` test/quality gates).
