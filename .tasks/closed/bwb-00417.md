## Description
Analyze the current UI verification stack and choose the stable primary strategy before adding new tooling.

## Findings
The repository already has strong, fast in-process UI verification through embedded-fixture app tests, teatest-driven mode tests, and golden render assertions. These cover real `bd` CLI gateway behavior, startup board rendering, board/detail transitions, empty-query search fallback, and launcher/editor seams without the fragility of external terminal capture.

## Scope
- Inventory what the current teatest, embedded-fixture, and model tests already verify
- Identify concrete bug classes still under-covered
- Decide whether any process-level capture is justified beyond in-process verification
- Record explicit readiness, timeout, and cleanup requirements for any process-level path
- Capture local feasibility findings so later tasks do not rediscover them

## Coverage Inventory
- `internal/app/model_test.go` already covers shell-level startup, board/detail transitions, search mode behavior, embedded-fixture board capture, startup regression checks, and launcher/editor flows
- `internal/mode/board/model_test.go` already covers real runtime-ish board rendering across widths with teatest and golden assertions
- `internal/mode/search/model_test.go` already covers teatest search rendering plus embedded-fixture empty-query fallback behavior
- `internal/testing/ui/harness.go` already provides deterministic Bubble Tea runtime harness helpers and output wait helpers

## Gap Analysis
Remaining gaps are narrower than originally assumed:
1. Combined shell workflows that span board, search, detail, and external-tool flows in a single deterministic scenario are still sparse
2. Assertion helpers are still somewhat ad hoc; later tasks should add quicker structured assertions/entrypoints for common verification needs
3. The binary entrypoint and terminal lifecycle boundary still has limited verification; this is the only area that may justify a process-level harness
4. Process-level visible-screen capture is risky for alt-screen TUIs and should not be the default strategy

## Decision
Primary strategy: extend in-process deterministic verification first.

Use embedded-fixture + teatest + golden/state assertions as the default path for UI verification work. Only add a narrow process-level check if a specific bug class cannot be caught in-process, such as binary entrypoint wiring or terminal boot/capture regressions.

## Process-Level Requirements (if later justified)
- Define a concrete bug class that in-process tests cannot catch
- Establish a readiness signal before capture/assertion
- Enforce a startup-to-capture budget under 2 seconds for the seeded fixture path
- Fail with an explicit timeout instead of hanging indefinitely
- Guarantee child-process cleanup on success and failure
- Avoid relying on raw stdout transcript capture alone for alt-screen rendering proof
