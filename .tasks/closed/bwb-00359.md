## Description

`internal/gateway/beads/read_payloads.go:175-201` (`toStatusOption`, `toTypeOption`) marks `description` as `requiredString`. Production `bd` is free to return `description: null` for user-defined statuses/types and the current test fixtures do not exercise null. If `bd` ever emits a null description for any catalog entry, the entire catalog read fails and the dashboard breaks.

Related test gaps from minor m4/m5/n6: no fixture exercises `null` or missing optional fields anywhere in gateway payloads (labels, comment text, description). This ticket folds that hygiene work in.

Major finding G6 + minor n6 from the 2026-05-16 audit.

## Files to Modify

- `internal/gateway/beads/read_payloads.go`
- `internal/gateway/beads/testdata/` — add new fixtures
- extend `internal/gateway/beads/read_gateway_test.go` or `payload_test.go`

## Instructions

1. Switch `description` in `toStatusOption` / `toTypeOption` to `optionalString`.
2. Add a fixture covering: `null` description on a custom status; `null` description on a custom type; `null` labels list; `null` comment text.
3. Add a "real-shape" production-noise fixture with at least one `null` optional, one unknown top-level field, and a multi-paragraph description, to exercise version-skew tolerance.
4. Add tests asserting the gateway decodes these without erroring, and that downstream consumers (board/details) handle empty/nil values correctly.

## Acceptance Criteria

- [ ] Catalog decoders accept `null`/missing descriptions
- [ ] Fixtures exist for null optional fields across catalog, issues, comments, labels
- [ ] Tests assert decode + consumer behavior
- [ ] Full quality gate green
