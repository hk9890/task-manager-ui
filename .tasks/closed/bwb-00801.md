## Description
Implement the standalone launcher framework that starts external tools from the current issue context. This task creates the foundation for editor, shell, and `opencode` launch actions while keeping launchers thin and explicitly non-orchestrating.

## Instructions
1. Create launcher abstractions under `internal/launcher/`.
2. Support structured issue-context interpolation including issue ID, title, labels, assignee, and project root.
3. Put process launching behind a replaceable seam so tests can verify constructed commands without spawning real tools.
4. Keep launcher behavior thin: fire-and-forget subprocess launches only. Do not add workflow coordination, polling, or orchestration behavior.
5. Integrate launcher invocation into the app shell in a way that later built-in launchers can reuse.

## Files to Modify
- `internal/launcher/` (create or update framework)
- `internal/domain/` (add launcher action/context models if needed)
- `internal/app/` or `internal/mode/details/` (integrate launcher triggers if needed)
- launcher-related `*_test.go` files if needed

## Acceptance Criteria
- [ ] A launcher framework exists under `internal/launcher/`
- [ ] Launchers can interpolate issue context values
- [ ] Process launching is behind a replaceable seam for automated tests
- [ ] Launcher behavior is explicitly limited to fire-and-forget subprocess launches
- [ ] The app can trigger launcher actions from issue context
