## Description

The keyboard help modal renders its top border (`╭` at top-left) but the matching bottom border (`╰`) is never painted. Help content runs out of vertical room and is truncated mid-line with no overflow indicator; the board's own bottom edge then renders below the modal content, with no visible modal closing edge.

Confirmed real bug — the pinning test `TestHelpModal_RenderedFrameHasClosingBottomBorder` in `internal/app/help_modal_render_test.go` reproduces it at multiple terminal sizes (120x34, 160x40) and is currently marked `t.Skip` referencing this issue.

(The same exploration originally claimed `Esc` did not dismiss the help modal — that claim was a PTY harness artifact and is **withdrawn**. The current ticket scope is **rendering only**.)

## Expected vs Actual

- **Expected**: When the help modal is open, the rendered frame has `╭` at top-left and a matching `╰` at the same column on a later row, closing the modal box. Either all content fits inside, or content is clipped with a visible scroll / overflow affordance inside the frame.
- **Actual**: At 120x34 and 160x40, only `╭` is painted. No matching `╰` appears anywhere below it. Help content overflows the viewport and the board's own bottom frame is what closes the visible area.

## Suspected Root Cause

Not yet code-traced. The help modal is rendered via the overlay primitive under `internal/ui/modal/` (uses the shared `internal/ui/overlay/` composer). The overlay composer likely paints the modal's lines onto the base view position-by-position; when the modal's natural height exceeds the viewport's available rows, the bottom-border lines fall off the bottom and are simply not painted (rather than the content being clipped while the closing border is preserved).

## Instructions

1. Read `internal/ui/modal/modal.go` (View / Overlay methods) and `internal/ui/overlay/` to identify exactly where the modal's lines are composed onto the base view.
2. Decide on a behavior — clipped scroll vs hard cap on modal height. The simplest correct fix is:
   - Compute the modal's effective height as `min(naturalHeight, viewportHeight - margin)`.
   - When the natural content exceeds the available height, render the visible slice of content **inside** the frame, leaving the last row for the closing `╰…╯` border. Add a single-line overflow indicator (e.g. `… N more lines (scroll)`) on the row above the bottom border.
3. Update or remove the `t.Skip` in `TestHelpModal_RenderedFrameHasClosingBottomBorder` so the test runs by default.
4. Confirm the test passes at all sizes it asserts (120x34, 160x40, and any others it covers).
5. Add a follow-up assertion to the same test (or a sibling test) confirming the overflow indicator appears when content is clipped, so the failure mode is exercised explicitly.
