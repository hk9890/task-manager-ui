## Description
Build a small ad-hoc verification toolkit that helps the agent/operator test the real UI properly during manual/runtime review. This is **not** a generic automated UI test framework. The toolkit should be centered on the existing PTY capture script plus a tiny persisted-state wrapper script.

## Intent / Non-Goals
### Goal
- help the agent run better real-app checks during implementation and review
- support flows like: open detail, move to Dependencies / Metadata, press up/down/enter/escape, inspect the resulting UI, and optionally compare `bd show --json` before/after

### Non-goal
- do not build a large new automated UI framework
- do not add branching flow logic, retries-as-a-framework, or general orchestration abstractions
- do not replace deterministic unit/integration tests

## Reviewer-Guided Reliability Requirements
- Step execution must be condition-based for multi-step flows, not only blind sleeps.
- The PTY helper must support waiting for text to appear and disappear.
- Waiting must use explicit per-step timeouts with clear failure messages identifying the step that hung.
- Polling during wait phases should be frequent enough to stay responsive (target around 50ms, not coarse 250ms waits).
- Cleanup must remain guaranteed on timeout/failure.

## Proposed Shape
### Script 1 — extend `scripts/capture_bwb_screen.py`
Add only the minimum reliable ad-hoc capabilities:
- send-key steps
- wait-for-text steps
- wait-for-no-text steps
- optional checkpoints / intermediate screen captures
- per-step timeout reporting
- structured JSON output with step/checkpoint context

### Script 2 — add a tiny wrapper (separate script)
Purpose:
- snapshot `bd show --json` before a mutation flow
- run a PTY flow via the capture script
- snapshot `bd show --json` after the flow
- report changed/unchanged state for selected fields

Keep this wrapper very small and recipe-oriented.

## Scope Boundary
If a flow can be described as:
- send key
- wait for text / wait for no text
- optionally checkpoint
then it is in scope.

If it requires general branching logic, retries as a framework feature, or broad orchestration semantics, it is out of scope.

## Concrete Design Deliverables
The design/implementation should specify:
- exact PTY step syntax
- exact PTY JSON output shape
- exact wrapper usage pattern for before/after `bd show --json`
- example command for verifying status save
- example command for verifying cancel-does-not-save

## Success Criteria
- [ ] A small helper script/toolkit exists for ad-hoc real-app UI checks.
- [ ] The PTY helper supports condition-based multi-step flows reliably enough for ad-hoc use.
- [ ] A tiny wrapper supports before/after `bd show --json` verification for mutation flows.
- [ ] `docs/RUNTIME_UI_VERIFICATION.md` documents the workflow concisely with practical recipes.
