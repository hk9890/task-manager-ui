The Done column on the board hardcodes the fetch limit to 50 in the Repository layer, even though the UI already knows how many rows it can display.

## Symptom

Running `mise run bwb -- --repo beads` against this project shows "50 of 679" in the Done column. With a tall terminal that can render many more rows, the user only ever sees 50; the rest of the closed issues are silently unavailable.

## User-facing workaround (until this ships)

**None.** The 50-cap lives in the Repository layer (`internal/repository/beads/lean_reads.go:19`), present in BOTH `--repo beads` AND `--repo caching` paths. Resizing the terminal or pressing `r` does not increase the cap. Closed-issue browsing beyond 50 requires `bd list --status=closed` directly.

## Root cause

`Repository.Dashboard` (`internal/repository/repository.go:53`) accepts no closed-fetch limit. The beads impl uses a hardcoded constant:

```go
// internal/repository/beads/lean_reads.go:17-19
const defaultLeanClosedLimit = 50
```

…passed at lean_reads.go:69. The board model already computes the right number based on terminal height (`internal/mode/board/model.go:249-258`) but only feeds it to `dashboard.Compose` (model.go:341), never to the Repository.

## Interface change

Adds `DashboardOptions{ClosedLimit int}` (landed by `beads-workbench-s4r7`). This epic implements `opts.ClosedLimit` honoring in `beads.Repository`, `memory.Repository`, `caching.Repository`, and wires `m.closedLimit()` through `loadDashboardCmd`.

**`ForceFresh × ClosedLimit` interaction:** `ForceFresh=true` honors the caller's `ClosedLimit`; it does not over-fetch a high-water max. A separate high-water optimization can land later if needed.

## Success Criteria

1. **Unit test (CI-verifiable):** board-model test that injects `height` for a range of values and asserts the value passed to a stub Repository's `Dashboard(opts)` matches `closedLimit()`. Without this, the resize-then-refresh behavior is not testable in `mise run quality` (terminal resize is not driveable from CI).
2. **Manual verification** of resize-then-refresh fetches more closed issues; entry in `docs/RUNTIME_UI_VERIFICATION.md`.
3. **"N of M" display correctness:** M is the real DB total via the existing `ClosedTotal` path, regardless of the requested limit.
4. **Parity contract scenario** in `internal/repository/contract/parity_integration_test.go` pinning that both memory and beads impls honor `opts.ClosedLimit`.
5. **`mise run quality` green.**

## Scope of change

- `internal/repository/beads/lean_reads.go` — use `opts.ClosedLimit`; treat 0 as "use default" or as "no limit" — decide and document.
- `internal/repository/memory/repository.go` — honor `opts.ClosedLimit`.
- `internal/repository/caching/caching.go` — propagate to backing; decide caching strategy when limit varies (re-fetch vs slice from high-water).
- `internal/mode/board/model.go` — pass `m.closedLimit()` into the Dashboard call via `loadDashboardCmd`.
- **bd argv pinning:** per `internal/bd/doc.go`, the lean `bd query --limit <variable>` argv now varies by caller. Add per-operation argv fixtures and contract tests.

## Out of scope

- Caching layer correctness bug (tracked in fbea).
