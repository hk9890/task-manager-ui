# Resize-triggered refetch — board + search

## Symptom

When the terminal window is resized larger, the board's `closedLimit()` and search's `searchItemCapacity()` increase (both are height-derived), but **no refetch is triggered**. The UI continues showing the smaller dataset until a manual reload, auto-refresh tick, or mode re-entry.

## Root cause

Both modes handle `tea.WindowSizeMsg` by updating dimensions and returning `nil`:

- `internal/mode/board/model.go:137-139` — `SetSize(w,h); return nil`
- `internal/mode/search/model.go:113-115` — same shape

The caching layer (`internal/repository/caching/caching.go:487-509`) treats a different `ClosedLimit` as a cache miss that round-trips to backing (option-a strategy, by design). So a resize that grows past the last-fetched limit *should* fan out to backing — but the model never asks.

## Design (confirmed with reviewer)

- **Debounce ~250ms**: drag-resize fires many `WindowSizeMsg`/sec; without debounce, a 30→80-row drag would issue ~50 `bd` round-trips. Implementation: generation counter + `tea.Tick` settled message. Tests inject `resizeSettledMsg` directly, no wall-clock dependency.
- **Grow-only**: refetch only when new derived limit > `lastFetchedLimit`. Shrink keeps existing data.
- **In-flight collision**: settled tick no-ops when `inflight`. Next resize/auto-refresh recovers.
- **`lastFetchedLimit` advances on successful load only**: a failed load leaves the threshold unchanged so the next resize still retries.
- **Search pre-submit gate**: settled tick no-ops until `hasLoadedPage` is true.
- **Board floor honored**: `closedLimit()` has a min-50 floor, so heights below ~53 won't actually grow the limit and shouldn't trigger refetches.

## Acceptance criteria

1. **Board: grow past last-fetched limit → exactly one refetch with the new `closedLimit()`** — unit test injects `WindowSizeMsg` then `resizeSettledMsg`, asserts `optRecordingRepo` saw a new call with the larger `DashboardOptions.ClosedLimit`.
2. **Board: shrink → no refetch** — `WindowSizeMsg` with smaller height does not produce a tick; if one is injected, it no-ops.
3. **Board: three rapid grows + one settled tick of the final gen → exactly one refetch with the final limit** — debounce via generation counter.
4. **Board: settled tick while `inflight=true` → no refetch**.
5. **Board: failed load → `lastFetchedLimit` unchanged → next grow retries**.
6. **Search: grow past last-fetched limit after first load → one refetch with new `searchItemCapacity()`** — assert `domain.SearchIssuesQuery.Limit` on the recorded call.
7. **Search: grow before first load (`hasLoadedPage=false`) → no refetch**.
8. **Search: same debounce + inflight + failed-load invariants as board**.
9. **`mise run quality` green**.
10. **Manual verification** entry in `docs/RUNTIME_UI_VERIFICATION.md`.

## Scope

- `internal/mode/board/model.go` — add `resizeGen`, `lastFetchedLimit`; `WindowSizeMsg` handler returns `tea.Tick` on grow; add `resizeSettledMsg` handler; `compose` updates `lastFetchedLimit` on success.
- `internal/mode/search/model.go` — same shape, gated on `hasLoadedPage`.
- New tests: `internal/mode/board/model_resize_refetch_test.go`, `internal/mode/search/model_resize_refetch_test.go`.
- `docs/RUNTIME_UI_VERIFICATION.md` — append resize-grows-window procedure.

## Out of scope

- Changing caching strategy from "different `ClosedLimit` = miss" to high-water + slice.
- Refetching on shrink.
- Auto-refresh interval changes.
