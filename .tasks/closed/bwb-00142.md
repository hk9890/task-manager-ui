## Description
Add a pure, table-tested component that takes the four data groups returned by the new dashboard fetch plan (Ready, Blocked, InProgress, Closed) and produces the typed column data the board renderer consumes. No I/O, no gateway dependency, **no logger** — fully unit-testable. Required by the dashboard rewrite (epic beads-workbench-au54). The component is intentionally dashboard-specific (lives in `internal/dashboard`); the gateway stays generic.

## Cardinality safety rail (locked during planning)
Active populations (Ready, Blocked, InProgress) are uncapped on the read side because <100 is expected. The composer must SURFACE — but not emit — a cardinality warning when any of those three groups exceeds 500 items. Logging is the caller's responsibility, not the composer's. This keeps the function pure and the warnings independently testable as data.

## Instructions
1. Create `internal/dashboard/composer.go` with:
   - Input struct:
     ```go
     type Inputs struct {
         Ready       []domain.IssueSummary
         Blocked     []domain.BlockedIssueView
         InProgress  []domain.IssueSummary
         Closed      []domain.IssueSummary
         ClosedLimit int // the cap that was sent to bd; used to compute "N+" badge
     }
     ```
   - Output struct:
     ```go
     type Columns struct {
         NotReady   ColumnData // from Blocked
         Ready      ColumnData
         InProgress ColumnData
         Done       ColumnData
         Warnings   []CardinalityWarning
     }
     type ColumnData struct {
         Issues       []domain.IssueSummary
         Total        int
         TotalIsExact bool // false when "N+" should be rendered (closed only)
     }
     type CardinalityWarning struct {
         Group     string // "Ready", "Blocked", "InProgress"
         Count     int
         Threshold int
     }
     ```
2. Sort rules (stable, deterministic):
   - NotReady: by Priority asc, then UpdatedAt desc, then ID. Map `BlockedIssueView` → `IssueSummary` for the column rows.
   - Ready: same sort.
   - InProgress: same sort.
   - Done: keep the order bd already returned (already sorted by closed_at desc at the backend); do not resort.
3. Total computation:
   - Active columns: `Total = len(Issues)`, `TotalIsExact = true`.
   - Done: `Total = len(Issues)`; `TotalIsExact = false` if `len(Issues) >= ClosedLimit && ClosedLimit > 0`, else `true`.
4. Cardinality warnings:
   - Threshold = 500.
   - If `len(Ready) > 500`, append `{Group:"Ready", Count:len(Ready), Threshold:500}` to `Warnings`. Same for Blocked and InProgress.
   - Tests assert `Warnings` slice contents directly — no slog capture, no global state.
5. Defensive sort check (for Done):
   - If `len(Closed) >= 2` and the first two items have non-descending `ClosedAt` (i.e. backend sort assumption is violated), append a `CardinalityWarning{Group:"Closed", Count:len(Closed), Threshold:-1}` so the caller can log a "backend sort assumption broken" alert. Document the `-1` sentinel in code.
6. Create `internal/dashboard/composer_test.go` with table-driven cases covering: all empty; single item per group; mixed; cardinality warning fires at >500 and does NOT fire at exactly 500 or 499; Done "N+" indicator at and below ClosedLimit; stable sort ordering when priorities tie; defensive sort warning when Closed input is misordered.
7. Do NOT import `bubbletea`, `gateway`, or `logging` from this file. Composer must be importable from a unit test with no I/O.
