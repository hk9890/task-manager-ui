In internal/testing/ui/scenarios.go, drainCmd (lines 84-108) recursively flattens BatchMsg commands. If a Cmd returns a BatchMsg that contains itself (or creates a cycle), this loops forever with no guard. In production Bubble Tea, the runtime handles this with its own scheduler. In the test harness, you're manually draining commands synchronously.

This is unlikely to trigger in practice because the project's commands are well-behaved, but the harness has no depth/iteration limit. A single misbehaving Init() or Update() return during development could hang the test suite silently.

Suggested action: add a max-iteration guard (e.g., 1000 messages) with a clear panic message. This is cheap insurance and makes debugging easier when it does happen.
