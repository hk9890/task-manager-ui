## Description

Production tripwire for write methods. Mirror the read-side `validatingGateway` decorator pattern from 8qw9.4: track writes and verify subsequent reads are consistent. When a write succeeds but the next read doesn't reflect it (dolt batch-commit lag, bd cache staleness, schema drift), emit `slog.Warn` so the divergence is captured in the persistent log.

## Decision needed during implementation

Two design options — pick the cleaner one after reading the existing validatingGateway:

- **Option A (extend)**: add write-method delegations + invariant checks to the existing `validatingGateway` in `internal/gateway/beads/validating_gateway.go`. Single decorator, all-in-one. Cross-method state (recently-written IDs) lives next to the existing ssom state.
- **Option B (separate decorator)**: new `validatingWriteGateway` in a sibling file. Composable with read decorator. Cleaner separation but requires `NewCLIGateway` to wrap twice.

I lean Option A — keeps related state together and avoids double-wrapping overhead. Validate before committing.

## Instructions

1. Read `internal/gateway/beads/validating_gateway.go` to understand the existing structure + state model.
2. Add new validators to `internal/gateway/beads/contractcheck/`:
   - `ValidateCreateResultShape(result domain.CreateIssueResult) []Violation` — returned ID non-empty, etc.
   - `ValidateWriteObservable(method, writtenID string, observedDetail domain.IssueDetail) []Violation` — fields the write claimed to set actually appear in the observed detail.
3. Cross-method invariant: track the last N (e.g., 16) write operations and their inputs in an ephemeral ring buffer. When a subsequent `ShowIssue` or `ListIssues` runs against an ID in the buffer, run `ValidateWriteObservable` and warn on mismatch.
4. **Critical limitation to document**: dolt batch-commit mode means writes may legitimately not appear in immediate reads. The invariant must either (a) skip when dolt is in batch mode, (b) wait briefly before warning, or (c) document the warning as advisory under batch mode. Pick the approach that minimizes noise.
5. Add unit tests in `validating_gateway_test.go`:
   - Each write invariant has a deliberate-break test (mock inner returns success from Create but next ShowIssue returns nothing → warning fires).
   - Test the dolt-batch-mode exemption if implemented.

## Acceptance Criteria

- [ ] Each write method has a corresponding invariant validator in `contractcheck`
- [ ] `validatingGateway` (or new `validatingWriteGateway`) delegates writes and runs validators
- [ ] WriteObservable cross-method invariant exercised by deliberate-break test
- [ ] Dolt batch-mode case handled (no false positives) and documented
- [ ] `mise run quality` passes
- [ ] Decision A vs B documented in the validating_gateway.go header
