## Summary

When `SearchIssues` is called with non-empty Text and len(Statuses) > 1, the gateway routes through `bd search`. The contract (interface.go INERT quirk) documents that `bd search --status` accepts only a SINGLE token, and the gateway passes `--status all` instead of comma-joining when multi-status + text. Real-bd audit confirms STILL INERT in bd 1.0.4.

No unit test pins this argv shape. A regression that changes the gateway to pass `--status open,closed` to bd search would be invisible to the test suite until real-bd integration runs.

## Discovered during

`beads-workbench-vhp2` Read-search audit (parent epic: `beads-workbench-yspw`).

## Fix

Add a unit test in `internal/gateway/beads/read_gateway_test.go` that calls `SearchIssues(ctx, SearchIssuesQuery{Text:"foo", Statuses:["open","closed"]})` and asserts the executor receives `["search", "foo", "--json", "--status", "all", ...]` — NOT `--status open,closed`.

## Acceptance criteria

- [ ] Test fails if the routing logic is changed to pass csv to bd search.
