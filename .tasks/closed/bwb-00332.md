## Description

`internal/repository/caching/caching.go:559` (`AddComment`) Forgets the per-ID cache but does NOT set `dashboardDirty`. The current comment says "Dashboard is NOT marked dirty because comments do not affect the Dashboard projection" — but that is wrong: Dashboard slot summaries (InProgress, Blocked, Ready, Closed) carry `UpdatedAt`, and bd advances UpdatedAt when a comment is added. Sort orders by UpdatedAt and any "recently active" affordance render with the pre-comment timestamp until 60s tick or another write.

User-visible effect: user posts a comment, dashboard list-row position does not update; user re-comments thinking the first post failed.

## Instructions

1. In `AddComment` after `c.memory.Forget(id)`, set `c.dashboardDirty = true` (under c.mu.Lock).
2. Update the godoc comment at the AddComment site to reflect the new behavior.

- `internal/repository/caching/caching.go` (AddComment ~line 559)

## Test

Add `TestAddComment_MarksDashboardDirty`:
1. Build CachingRepository over a fake backing whose Dashboard returns InProgress with issue I at UpdatedAt=T0, then after AddComment returns InProgress with I at UpdatedAt=T1.
2. Call Dashboard() — assert UpdatedAt==T0; cache is warm.
3. Call AddComment(ctx, I, "comment body").
4. Call Dashboard() — assert UpdatedAt==T1 (cache refetched).
