## Description
BWB currently starts the Bubble Tea UI immediately from `cmd/bwb`, which means operators cannot use standard CLI affordances like `--help`, `--version`, config inspection/validation, or startup overrides from a non-TTY context. Add a small pre-TUI CLI layer so BWB can expose a focused flag surface without turning into a Perles-style multi-command tool.

## Decisions
- Use the Go standard library `flag` package with explicit aliases for `-h`/`--help` and `-v`/`--version`; do not add `cobra`/`pflag` for this minimal CLI layer.
- Keep `bwb` as a single-command TUI entrypoint with flags, not a subcommand-heavy CLI.
- Use `--cwd <path>` as the repo-targeting flag instead of `--beads-dir`, because BWB is repo-root oriented today and uses the project root for gateway working dir and launcher interpolation.
- Non-interactive commands (`--help`, `--version`, `--print-config`, `--check-config`) must succeed without opening a TTY.
- Exit codes are: `0` success, `1` validation/runtime error, `2` usage error (unknown flag, malformed usage, missing required flag value).
- Revisit `--beads-dir` only if BWB ever needs to target a beads tracker that is not co-located with the project root.

## Success Criteria
- [ ] `bwb` supports `-h/--help`, `-v/--version`, `-c/--config`, `--cwd`, `-d/--debug`, `--no-auto-refresh`, `--print-config`, and `--check-config`
- [ ] Non-interactive flag paths return before Bubble Tea starts and work in non-TTY environments
- [ ] `--config` overrides the default XDG config path cleanly
- [ ] `--cwd` changes the repo/project target cleanly for gateway and launcher startup behavior
- [ ] `--debug` and `--no-auto-refresh` change runtime behavior without affecting default startup
- [ ] CLI help/docs/release metadata reflect the shipped flag surface
- [ ] Documented exit-code behavior matches the shipped CLI behavior
- [ ] `go test ./...`, `go build ./cmd/bwb`, and `go vet ./...` pass
