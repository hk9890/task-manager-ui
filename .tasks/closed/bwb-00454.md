Context: Verifying detail-view data parity vs bd on the embedded fixture (bwf-2).

What I observed:
  - bd comments bwf-2 lists, in order:
    1. 'Needs dependency resolution'   (2026-05-24 14:37)
    2. 'Second deterministic comment'  (2026-05-24 14:37)
  - The bwb detail pane displays them in the inverse order:
    [1/2] Hans Kohlreiter · 2026-05-24 14:37 — Second deterministic comment
    [2/2] Hans Kohlreiter · 2026-05-24 14:37 — Needs dependency resolution
  - Both comments share the same creation minute, so the secondary sort isn't visually distinguishable from the timestamp alone, but the name 'Second deterministic comment' strongly implies it was added after 'Needs dependency resolution'.

Why this is unclear:
  - Newest-first is a reasonable UX choice for high-traffic discussions, and 'bd comments' default ordering is undocumented in the docs I've read so far (docs/CONFIGURATION.md, docs/RUNTIME_UI_VERIFICATION.md, internal/repository/repository.go method docstrings would be the place but I haven't checked).
  - But the documented intent of bwb is to mirror what bd shows; if bd's canonical order is oldest-first and bwb silently inverts, that's an undocumented divergence that may confuse operators who use both CLIs.
  - The [N/M] index suggests a stable indexing for keybind selection — if the order ever changes (e.g. cache rehydration vs fresh fetch), the index may drift relative to bd's own.

Resolution paths:
  - If newest-first is policy: document it (docs/CODING.md or docs/user-guide), and ideally label the indices accordingly so 'comment [1/N]' is obviously the most recent.
  - If it's accidental: switch to bd's natural order to remove the divergence.

Where discovered: project-explore session beads-workbench-czkq, iteration 4 (2026-05-24).
