## Description
Discovered while inspecting runtime logs for this project. Search queries can hit `bd` result limits, and current monitoring evidence shows truncation warnings in command stderr. This issue now tracks the search-specific product gap and the decision needed for how BWB should handle capped search results.

## Expected Behavior
When search relies on capped `bd` queries, BWB either uses a strategy that is known-safe for expected repo sizes or explicitly surfaces that search results may be partial.

## Actual Behavior
Runtime logs captured successful search-related `bd list` commands whose stderr reported hidden matches due to `--limit`. The current model uses the returned item count as the total, so the app cannot distinguish complete vs truncated search results.

## Steps to Reproduce
1. Run the app in a repo with more matching issues than the search query limit.
2. Start `bwb` with debug logging enabled.
3. Inspect the persistent log at `$XDG_STATE_HOME/bwb/bwb.log` or `~/.local/state/bwb/bwb.log`.
4. Observe command records whose stderr reports hidden search results due to `--limit`.

## Impact
Search can appear complete when the underlying query returned only a capped subset. This can hide work during investigation and makes monitoring reveal a search completeness gap.

## Files to Modify
- internal/gateway/beads/*
- internal/mode/search/*
- internal/domain/* if result metadata changes
- docs/MONITORING.md or other product docs if truncation behavior is documented

## Acceptance Criteria
- The team chooses one search-result strategy (for example: raise the cap, surface partial-result state, or redesign the query path).
- The implementation and tests match that single chosen strategy.
- Documentation reflects the final search-result behavior clearly.
