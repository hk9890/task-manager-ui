## Contract restated (key clauses — see interface.go for full text)

### StatusCatalog
- Result never empty; bd always returns at least built-in statuses.
- Built-ins always include at least: open, in_progress, blocked, deferred, closed, pinned, hooked.
- Custom statuses appear after built-ins.
- Each `Name` non-empty; `Description` may be empty.
- 2026-05-16: `bd statuses --json` OMITS the "custom_statuses" key entirely when no custom statuses exist (rather than empty array). Decoder handles via nil slice. Disposition: ACCEPT.
- 2026-05-16: `bd statuses --json` includes "category" and "icon" fields not captured by decoder. Disposition: ACCEPT — not needed by consumers.

### TypeCatalog
- Result never empty.
- Core types always include at least: task, bug, feature, chore, epic, decision, spike, story, milestone.
- Custom types appear after core types.
- Each `Name` non-empty.
- 2026-05-16: `bd types --json` omits "custom_types" key when none exist. Disposition: ACCEPT.

### LabelCatalog
- Each `Name` non-empty (gateway strips whitespace, skips blank entries).
- Order is bd-provided (typically alphabetical or by usage; unspecified).
- Returns empty slice (NOT error) when no labels.
- 2026-05-16: `bd label list-all --json` returns array entries with "label" (NOT "name") and "count" fields. Decoder uses `bdLabelCatalogEntryPayload` with `json:"label"` — INTENTIONALLY different from statuses/types pattern.
- 2026-05-16: Labels from closed issues are included. Disposition: ACCEPT.

**Error semantics (all three):**
- `ErrorCodeCommandFailed` for non-zero bd exit.
- `ErrorCodeDecodeFailed` for unparseable stdout.
