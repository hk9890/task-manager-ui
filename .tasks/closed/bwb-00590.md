## Description
Rework issue inspection UX around a true side panel model instead of treating related issue navigation as a sub-surface inside the issue detail view.

## Updated Product Direction
- Related issue browsing should behave like its own side panel, analogous to browsing issues in the dashboard while a detail pane updates beside it.
- Selecting issues in that side panel should update the main issue detail without forcing a separate nested focus model inside the detail body.
- Focus/navigation behavior should become consistent with the rest of the app rather than introducing a detail-only interaction pattern.
- Detail metadata fields may become directly actionable in the future, with Enter-driven edits for fields like status and priority.
- Layout should use the available screen width more effectively; the current implementation under-uses horizontal space and feels boxed in.

## Current State
The shipped v1 left rail works functionally and is verified, but it does not match the preferred UX direction above. Treat the current implementation as an intermediate step, not the final interaction model.

## Next UX Questions to Resolve
- How should the inspector/side-panel layout be partitioned across board, search, and dedicated detail so focus behavior is uniform?
- Should the issue list panel become a persistent browser context while the detail pane updates live on selection?
- Which metadata fields should support direct Enter-to-edit actions in v1 of inline editing?
- How should the layout expand to better use full terminal width without making content harder to read?
