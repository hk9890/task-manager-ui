## Description
Direct runtime feedback shows current search empty-state UX is still wrong. When entering search with an empty query, the app currently shows an instructional empty state instead of a useful result list. Better operator UX is to show all searchable issues by default, then narrow the list live as the user types. Clearing the query should restore the full result list.

## Evidence
- User feedback after direct runtime test: search still does not feel better
- Current search mode shows an empty/help state with no issue list until the user types
- This creates extra friction and makes search feel less useful than a browseable filtered list

## Instructions
1. Search mode should show all issues by default on entry when query is empty.
2. Typing should live-filter that list.
3. Clearing the query should restore the full issue list.
4. Results/preview/navigation should remain coherent across empty and non-empty query states.
5. Update tests and snapshots to cover default-all-results behavior.

## Files to Inspect
- `internal/mode/search/`
- `internal/ui/search/`
- `internal/app/model.go`
- related app/search tests and goldens

## Acceptance Criteria
- [ ] Entering search with an empty query shows a useful all-issues result list
- [ ] Typing narrows the result list live
- [ ] Clearing the query restores the full result list
- [ ] Search navigation and preview remain usable in both empty-query and filtered states
- [ ] Tests cover default-all-results and clear-to-restore behavior
