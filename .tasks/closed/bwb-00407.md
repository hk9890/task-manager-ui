## Description
`internal/gateway/beads/runner.go` (around line 92-94) holds a global mutex for the entire subprocess duration in `CommandRunner.Run`. The TUI loads 4 dashboard sections concurrently, but the mutex forces them to execute sequentially — defeating the purpose of concurrent loading.

## Instructions
1. Open `internal/gateway/beads/runner.go`
2. Find the mutex lock in the `Run` method (around line 92)
3. **Default approach: document and keep the mutex.** Dolt uses file locking on its local database, so concurrent `bd` CLI invocations against the same `.beads/` database may cause file lock contention. Test fakes don't exercise real file I/O so won't catch this.
4. Add a clear comment above the mutex explaining:
   - WHY: Dolt file locking means concurrent bd CLI calls against the same DB can contend
   - IMPACT: Dashboard sections load sequentially (~4x slower than parallel)
   - FUTURE: If bd adds a long-running server mode or read-only concurrent access, this can be relaxed
5. **Only remove the mutex if you can confirm** (via bd/Dolt docs or testing with real bd) that concurrent read-only CLI calls are safe. If removing, replace with a semaphore allowing bounded concurrency (e.g. 4 concurrent calls) and verify with `go test -race ./internal/gateway/beads/...`

## Files to Modify
- `internal/gateway/beads/runner.go` (add documentation comment, or relax mutex if safe)
