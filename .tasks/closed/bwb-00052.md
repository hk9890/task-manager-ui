Discovered during verification of beads-workbench-x2u in the built binary against the embedded fixture repo. Expected: after running `bd dep relate <issue-a> <issue-b>`, the linked issue appears only under Related in detail view. Actual: `bd show --json` includes the relates-to issue in both dependencies and dependents with `dependency_type=relates-to`; the gateway normalization was fixed on the dependency side but the dependents path still maps entries straight into Blocks, so the live app shows the same issue under Blocks and Related.

## Root Cause
Gateway handling for dependents still treats all dependent refs as Blocks. A `relates-to` dependent must be classified as Related, not Blocks.

## Acceptance Criteria
- [ ] Gateway/detail normalization classifies `relates-to` dependents as Related instead of Blocks.
- [ ] Unit tests cover the dependent-side `relates-to` case.
- [ ] In the real app, an issue whose dependents include a `relates-to` entry shows that issue only under Related, not under Blocks.
