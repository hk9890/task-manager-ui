## Description
Consolidate Beads Workbench issue-list rendering so board and search share the same compact issue-row presentation model instead of drifting through separate hand-built row formatting paths.

## Goals
- Introduce a shared compact issue-row renderer used by multiple list-style issue views.
- Decide whether a lightweight shared issue-list component should also exist above the row level.
- Keep existing detail rendering as a separate, richer component while clarifying its relationship to row/list/panel building blocks.
- Preserve deterministic truncation, selection semantics, and width-aware layout behavior across board and search.

## Decisions
- `internal/ui/details` already acts as the issue detail component and should remain separate from compact list rendering work.
- The most urgent gap is the missing shared issue-row component; issue-list abstraction should only be introduced if it reduces real duplication after row extraction.
- Generic bordered panel rendering is already provided by `internal/ui/styles.FormSection`; do not replace it unless a concrete limitation is found.
- Board and search should share one compact row contract for type/priority/status/id/title/selection rendering.

## Success Criteria
- [ ] Architecture decisions for row/list/panel/detail responsibilities are explicitly documented in code-facing project docs.
- [ ] Board and search no longer maintain separate hand-built compact issue-row rendering logic.
- [ ] Any introduced list abstraction has a clear payoff and does not add speculative complexity.
- [ ] Existing detail rendering continues to work as the dedicated issue detail component.
- [ ] Tests cover shared row/list behavior and board/search integrations.
- [ ] `go test ./...`, `go build ./cmd/bwb`, and `go vet ./...` pass.
