## Description

Move the contract-validation surface from `validatingGateway` (gateway boundary) to a new `validatingRepository` decorator at the **Repository** boundary. It wraps any `repository.Repository` implementation and runs validation rules against domain return types (`DashboardData`, `IssueDetail`, `SearchResultPage`, etc.). The legacy `validatingGateway` is NOT deleted here — that is E4.4. This task delivers the replacement; deletion follows.

## Always-on in production

Match today's behaviour: `validatingRepository` wraps the lean `beads.Repository` in production wiring (wired in E4.3). On violation it warn-logs through the existing slog logger; it does NOT fail the call.

## Instructions

1. Create `internal/repository/beads/validating_repository.go` (or `internal/repository/validating.go` if it makes sense at the parent package level — choose where it composes most naturally with the Repository interface). Constructor: `func NewValidating(inner repository.Repository, logger *slog.Logger) repository.Repository`.
2. For every rule currently in `validatingGateway` + `contractcheck`, identify the equivalent post-condition on Repository return types and implement it. Examples to consider:
   - Status / type / label values returned by `Catalogs()` are non-empty and well-formed
   - `Dashboard` columns contain no duplicate issue IDs across sections
   - `Issue` returned by `Issue(id)` has matching `ID` field
   - `Search` page metadata is consistent with returned issues
3. Rule migration is one-to-one with `contractcheck.Violation` cases where possible. Where a gateway-level rule no longer makes sense at the Repository level (e.g. it was about an intermediate type that no longer exists), drop it and note the drop in a brief comment.
4. Mirror the existing warn-logging path through slog. Do not return errors on violation — match current always-warn behaviour.
5. Add unit tests in `validating_repository_test.go` that inject a stub `repository.Repository` returning known-bad responses and assert the violation is detected and logged. Aim for one test per migrated rule (positive + negative case).

## Acceptance Criteria

- [ ] `NewValidating(inner, logger)` returns a `repository.Repository` that delegates calls and warn-logs on violations
- [ ] Every rule from `validatingGateway` / `contractcheck` has either a Repository-level equivalent OR an explicit decision-comment recording why it was dropped
- [ ] Unit tests cover each migrated rule (positive + negative)
- [ ] `go test ./internal/repository/...` green
