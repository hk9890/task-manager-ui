## Description
Search results and board rows currently render issue metadata through two separate row-formatting paths. The board now applies colored type/priority/status tokens, muted IDs, and stronger selection emphasis, but search still builds plain text rows manually.

## Steps to Reproduce
1. Launch Beads Workbench on any repo with visible board rows and search results.
2. Observe that board rows render colored type/priority/status tokens, muted IDs, and a stronger selected-row indicator.
3. Switch to Search and inspect the result list for the same or similar issues.
4. Observe that search rows still use a separate plain-text assembly path and do not share the board row styling/renderer.

## Why
This creates a visible UX inconsistency and shows that Beads Workbench does not yet have a single shared compact issue-row renderer for list-style issue views. As a result, improvements landed in the board do not automatically carry into search, and future list-row changes risk drifting again.

## Scope
- Analyze the current board/search row rendering split.
- Introduce or extract a shared compact issue-row rendering helper/component suitable for both board and search result lists.
- Apply the shared renderer to search so it inherits the same compact metadata styling semantics as board where appropriate.
- Preserve width-aware truncation and mode-specific layout differences.

## Files Likely Involved
- `internal/ui/board/board.go`
- `internal/ui/search/search.go`
- `internal/ui/shared/renderhelpers/helpers.go`
- optionally a new small shared UI helper under `internal/ui/shared/`
- related board/search renderer tests

## Acceptance Criteria
- [ ] Board and search no longer maintain separate hand-built compact issue row formatting logic.
- [ ] Search results render the same metadata hierarchy as board for type, priority, status, ID, and selected-row indicator unless an intentional mode-specific exception is documented.
- [ ] Width/truncation behavior remains deterministic in both board and search tests.
- [ ] Tests cover the shared renderer behavior and both call sites.
