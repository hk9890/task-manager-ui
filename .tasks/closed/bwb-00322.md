The caching repository layer silently serves a degenerate dashboard built from a partial in-memory store. The root cause is structural, not a refresh-timing issue.

## Observed symptom

Running `mise run bwb` (default --repo caching) against this project: the Done column shows ONLY `znri.2`, all other lanes empty even though `bd list --status open` returns 7 and `bd list --status closed` returns 600+. Running with `--repo beads` shows correct data.

## User-facing workaround (until this ships)

Run `mise run bwb -- --repo beads` (or `go run ./cmd/bwb --repo beads`). This bypasses the caching decorator entirely and wires the lean `beads.Repository` directly to `bd`. Disables the fast-paint benefit but restores correctness.

## Root cause

`caching.Dashboard()` writes only to `c.dashboardCache` (caching.go:441-443). It **never seeds `c.memory`**. Only `Issue(id)` seeds `c.memory` via `SeedDetail` (caching.go:489). `SaveNow` snapshots `c.memory` only (caching.go:389).

**Captured evidence:** cache files captured into `internal/repository/caching/testdata/known-bad-cache-fbea/` from a real user session contain exactly the IDs the user clicked into — `znri.*` and `kb8e*` in this case. Nothing else. The persisted file is just a click trail.

On the next Hydrate with matching bd-hash, `memory.Repository.Dashboard` returns `ClosedTotal=len(closed)` from the tiny in-memory store (memory/repository.go:404-410). The dashboard the user sees is built from their prior click history, not from bd.

The bug usually masks itself because a new bd commit between sessions changes the hash, triggering the `confirmedMismatch=true` full fan-out branch (caching.go:343). It surfaces only when the bd hash is unchanged across sessions.

## Two defects

### Defect #1 — Dashboard does not seed memory (root cause)

`caching.Dashboard` populates `c.dashboardCache` but never feeds the summaries into `c.memory`. SaveNow can therefore never persist a useful Dashboard view. On Hydrate with matching hash, the memory-backed Dashboard recomputation returns a falsely "complete" tiny dashboard.

**Chosen fix candidate: (b)** — `SaveNow` persists `dashboardCache` + `catalogsCache` alongside the issue snapshot. Schema-version bump on the persisted file format; `Hydrate` restores all three caches. This preserves correctness AND fast-paint (cached Dashboard is served on cold start before any backing call). Rejected alternatives are documented in design notes.

### Defect #2 — Manual reload (`r`) does not bypass the cache

`internal/mode/board/model.go:186` → `startReload(refreshModeManual)` → `loadDashboardCmd(m.repo)` → `repo.Dashboard(ctx)`. The Repository interface has no force-fresh knob; the caching decorator returns `c.dashboardCache` when `!dashboardDirty`. The user's `r` keystroke is silently a no-op.

Independent of Defect #1.

## Success Criteria

1. **Parity invariant:** after `r.Dashboard(opts)` returns N closed issues, `r.Issue(id)` for each MUST NOT cause a backing call. This pins Dashboard-seeds-memory regardless of fix.
2. **`r` bypasses cache:** manual reload passes `DashboardOptions{ForceFresh: true}` and the caching decorator honors it by skipping the cache fast-path.
3. **Deterministic repro test:** integration test using the captured fixtures at `internal/repository/caching/testdata/known-bad-cache-fbea/` confirms launching `--repo caching` against a known-bad file results in a correct board within one refresh cycle.
4. **File-present-but-degenerate test:** add to `internal/repository/caching/cold_start_test.go` (currently only covers file-missing per `cold_start_test.go:25`).
5. **Persisted schema version bumped** and a migration path documented for older snapshots.
6. **`mise run quality` green.**

## Interface change

Adds `ForceFresh bool` to the `DashboardOptions` struct introduced by `beads-workbench-s4r7`. This epic implements honoring in `caching.Repository` and wires `r` through it.

## Persisted-state model (candidate b — chosen)

`SaveNow` now serializes a richer file:
- Schema version bumped (e.g. v2).
- Header line carries `dashboardCache` snapshot + `catalogsCache` snapshot.
- Issue lines unchanged (memory snapshot).

`Hydrate` reads the header before issue lines and restores all three caches. Cold-start fast-paint is now correct even when the bd hash matches across sessions. Backing call still fires lazily on the next interaction or `r`.

Migration: when reading a v1 file, restore only the memory snapshot; mark dashboardCache dirty so the next Dashboard() call fetches from backing.

## Out of scope

- Adding the `DashboardOptions` struct itself (tracked in s4r7 — this epic depends on s4r7).
- The hardcoded closed-issue limit (tracked in iwvm).
- 5 open UI bugs.
