## Description

Add an in-process app-level integration test that proves non-blocking refresh works end-to-end without relying solely on PTY captures. PTY is slow and prone to timing flakes; this in-process test is cheap, deterministic, and gives every CI run a regression guard.

The test must drive the app model through a Board → Search → Board cycle and assert:
1. The Board view's output contains the same set of issue id rows BEFORE and AFTER the switch-out (data did not blank).
2. The header spinner glyph appears in the View() output during the in-flight reload.

Depends on `beads-workbench-0x36.1` (SkeletonRow), `beads-workbench-0x36.2` (Board non-blocking), `beads-workbench-0x36.3` (Search non-blocking), `beads-workbench-0x36.6` (Spinner). Blocks `beads-workbench-0x36.5` acceptance.

## Files to Modify

- `internal/app/model_test.go` (or new sibling `model_loading_refresh_test.go`) — new test per the criteria below. Use the existing fake gateway + `withSpinnerTickScheduler` + `withRefreshTickScheduler` seams so the test is deterministic.

## Instructions

1. Use the existing fake gateway and result-injection helpers in `internal/app/model_test.go` (see test setup around lines 27, 391-1070 for patterns).
2. Construct the app model, drive Init() and inject the 3 board gateway results so the board has known issue rows.
3. Capture the View() output and extract the set of issue ids visible on the board.
4. Trigger the Board → Search switch (`maybeAutoRefreshActiveSurfaceCmd` path). Capture View() output BEFORE the next gateway results arrive.
5. Assert: the captured view still contains the same set of issue id rows from step 3 (stale data is preserved).
6. Inject one spinner tick so the spinner frame advances. Assert: View() contains a glyph from the pinned `SpinnerFrames` set.
7. Switch back to Board. Inject new gateway results. Assert: View() now contains the new rows and no longer contains the spinner glyph (idle).
