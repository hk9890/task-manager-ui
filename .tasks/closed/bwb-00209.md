## Description
Launcher behavior is currently fire-and-forget for non-editor tools. This leaves unclear runtime UX for interactive terminal tools: how the user gets back to the app, whether launcher commands should take over the terminal, and what the expected post-launch workflow is. The current implementation starts launcher subprocesses immediately and does not provide a structured round-trip back into the app.

## Evidence
- User explicitly asked how to go back to the app after launcher use and what happens after saving changes
- `internal/launcher/process_runner.go` starts subprocesses and returns immediately
- current launcher UX is under-specified for interactive terminal tools

## Instructions
1. Define the intended UX for launchers vs rich editor flow.
2. Decide whether built-in launchers are meant for non-interactive fire-and-forget commands, terminal takeover, or a managed return flow.
3. Implement the chosen behavior or, at minimum, make the UX explicit and coherent in-app/docs.
4. Add tests for the intended launcher lifecycle behavior where practical.

## Files to Inspect
- `internal/launcher/`
- `internal/app/model.go`
- `docs/CODING.md`
- related launcher tests

## Acceptance Criteria
- [ ] Launcher lifecycle/return behavior is clearly defined
- [ ] The app/docs no longer leave ambiguity about returning to the app after launcher use
- [ ] Interactive launcher behavior is either supported properly or explicitly constrained/documented
- [ ] Tests and docs reflect the intended launcher UX
