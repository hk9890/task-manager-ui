Discovered during verification of beads-workbench-59z while augmenting the embedded fixture with a `bd dep relate` link. Expected: a relates-to link created with `bd dep relate <id1> <id2>` appears in detail-mode under the Related group. Actual: `bd show --json` returns the link with dependency_type `relates-to`, but the gateway/detail pipeline only treats dependency_type `related` as related work, so the app renders the linked issue under Blocked by/Blocks and leaves Related empty.

## Root Cause
Gateway normalization only recognizes `related` and misses `relates-to`, even though `bd dep relate` emits `relates-to`.

## Acceptance Criteria
- [ ] Gateway normalization classifies both `related` and `relates-to` dependency types as related work.
- [ ] Unit tests cover both dependency type strings.
- [ ] Real-app detail view shows `bd dep relate` links under Related rather than Blocked by/Blocks.
