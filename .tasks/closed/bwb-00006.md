## Description
Define the default triage dashboard grouping and startup focus behavior so opening the app immediately shows a useful 4-column overview similar in usefulness to Perles. This task owns both the exact lane strategy and the deterministic startup focus needed for a stable initial dashboard.

## Explicit 4-Column Target
Implement this default 4-column dashboard in order:
1. **Not Ready** — work that is blocked or otherwise not ready to act on yet
2. **Ready** — actionable work ready to start now
3. **In Progress** — currently active work
4. **Done** — recently completed / recent finished work suitable for review context

These are the intended startup columns for the redesign effort. Do not substitute the old `Ready / In Progress / Blocked / Recent Work` grouping.

## Instructions
1. Implement the explicit 4-column default dashboard above in `internal/dashboard/`.
2. Use Perles as a UX benchmark only when choosing how to map those lanes to supported gateway query types.
3. Keep all grouping/query definitions backed by supported gateway query types only.
4. Fold startup focus stability work from `beads-workbench-3tm` into this task.
5. Ensure startup focus is deterministic and stable in dashboard order during async loads.
6. Add/update tests for dashboard definitions, query mapping, stable startup focus behavior, and rendered startup dashboard behavior using the new multi-width/runtime test infrastructure.
7. Include at least one runtime/render assertion for the startup dashboard shape, not only model-state assertions.
