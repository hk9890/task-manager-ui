## Description
Two related improvements to the full Detail view (Detail mode, the 3-pane Dependencies/Content/Metadata surface):

1. **Epic children are fetched but never rendered.** `IssueDetail.Children` is populated from `bd show` dependents with `dependency_type=parent-child` (internal/repository/beads/lean_payloads.go:452) but `dependencyGroups()` (internal/ui/details/details.go:501) omits them, so an epic's detail shows no child tasks.

2. **Related-list navigation is coupled to full-detail reload.** In the Dependencies pane, ↑/↓ moves the cursor AND immediately reloads the whole detail (moveRelatedSelection emits OpenRelatedIssueIntent, internal/mode/details/model.go:231-250). The user wants cursor movement to only move a highlight; the detail should change only on Enter. Because cursor and shown-issue become distinct, the UI must visibly distinguish the pending cursor selection from the currently-displayed subject.

## Outcome
- Opening an epic in detail shows a "Children" group listing its child tasks, counted and navigable like other related refs.
- ↑/↓ in the Dependencies pane moves only the highlight; Content/Metadata do not change. Enter opens the highlighted related/child issue as the new full detail and moves focus to it. Esc still returns to the browse mode.
- The cursor's pending selection is visually distinct from the loaded subject.

## Success Criteria
- [ ] Epic detail renders a Children group with correct count
- [ ] Related/child navigation decoupled from detail reload; Enter gates the reload
- [ ] Visual cursor-vs-subject distinction is clear
- [ ] Goldens updated; mise run test green; runtime UI verification passes
