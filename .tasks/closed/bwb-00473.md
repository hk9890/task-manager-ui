What I did: read internal/app/model.go, then verified at runtime with /tmp/bwb against the embedded fixture.

Expected: toggle_help is a TOGGLE — the configured key should both open and close the help overlay. docs/CONFIGURATION.md even ships an example config (lines ~81-82) that rebinds 'toggle_help: [F1]'.

Actual: The OPEN path (model.go line ~672) uses 'm.keys.Match(config.ShellContext, config.ShellActionHelp, msg)' and respects the config. The CLOSE path (model.go line ~376) is hardcoded: 'if k, ok := msg.(tea.KeyMsg); ok && k.String() == "?"'. So when toggle_help is rebound away from '?', the configured key opens help but does NOT close it; the only key that closes help is the hardcoded '?', which is now unbound for opening. Under the documented example ('toggle_help: [F1]') the help overlay becomes effectively non-toggleable from the configured key.

Verified via scripts/capture_bwb_screen.py with config 'keybindings: {shell: {toggle_help: [t]}}':
  - press 't' -> help overlay visible (opened)
  - press 't' again -> help overlay STILL visible (rebound key does not close)
  - press '?' -> help overlay closed (hardcoded close key)

Severity: medium. The documented customization is half-broken; an operator who rebinds toggle_help loses the toggle-off and may think the app is stuck (esc also closes it, but that is not the documented toggle contract).

Repro:
  printf 'keybindings:\n  shell:\n    toggle_help: [t]\n' > /tmp/c.yaml
  /tmp/bwb --config /tmp/c.yaml --check-config   # config OK
  # run against fixture: t opens help, t does not close it, ? closes it

Fix: change the close check at model.go ~376 to 'm.keys.Match(config.ShellContext, config.ShellActionHelp, msg)' so the same configured key toggles both directions.
