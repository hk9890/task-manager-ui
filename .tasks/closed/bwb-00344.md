What I did: Launched bwb against the 590-issue scale fixture and read the Done column header; cross-checked with 'bd' counts and the composer/renderer source.

Expected (source: internal/dashboard/composer.go comment — ClosedTotal 'overrides len(Closed) as Done.Total so the header shows the actual count'): the Done column header communicates the real closed-issue count clearly.

Actual: Done header renders '75+'. The real closed count is exactly 75 (bd count status=closed = 75; CountIssues feeds Done.Total = 75 correctly — the ssom fix holds). The trailing '+' is appended whenever the rendered row list is capped: closedLimit()=50 at this terminal height, so 50 rows < 75 total => TotalIsExact=false => '+' (internal/ui/board/board.go:88-91).

Problem: the number (75) is an EXACT count, but '75+' uses the conventional notation for a lower bound ('75 or more'). The '+' here actually means 'the row list below is truncated', not 'the count is approximate'. The two meanings are conflated in one badge. A reasonable user reading '75+' concludes 'at least 75 closed issues' when in fact exactly 75 exist. This is the same column that bug ssom already corrected for showing a wrong total; the total is now right, but the badge semantics undercut it.

Severity: low. Count is correct; this is a UX/semantics ambiguity (FEW HICCUPPS: Explainability, Familiarity — '+' after a number conventionally means '≥'). It only surfaces when closed count > visible row cap.

Suggested fix direction: since Done.Total is always the exact DB count now, drop the '+'-as-lower-bound notation and use a truncation indicator that does not attach to the number (e.g. 'showing 50 of 75', or a separate '⋯'/scroll affordance), so the exact count is not misrepresented.
