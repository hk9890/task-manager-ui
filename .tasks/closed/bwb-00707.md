## Description

Make the argv-contract testing pattern discoverable so future agents/maintainers add tests for new bd calls without having to reinvent the discipline.

## Instructions

1. Extend `internal/gateway/beads/doc.go` (created/extended in [[ppja.2]]) with a section "Argv contract testing" explaining:
   - The dual-contract design: `RunReadContract`/`RunWriteContract` covers what the gateway PARSES from bd (incoming); per-call argv-cardinality tests cover what the gateway SENDS to bd (outgoing).
   - When to use `RecordingExecutor`: every test that asserts a specific bd argv shape. Show the canonical pattern (similar to `TestBoardInitRealGatewaySubprocessArgvCardinality`).
   - The argv inventory at `internal/gateway/beads/ARGV_CONTRACT.md` (or wherever [[ppja.2]] placed it) is the single source of truth — any new bd call site must add an entry.
2. Link from AGENTS.md to the doc section so a new agent making changes that affect gateway argv has the discipline in scope.
3. Cross-link from `internal/testing/fakes/doc.go` (the fake-update discipline doc from 8qw9.5) — the argv contract is the outgoing-contract twin of the read/write contract.
