Discovered from live user review of the current detail UI on `beads-workbench-t7o.2`. Expected: dependency selection should traverse grouped rows in a predictable linear order, so moving down from the last row of one non-empty group should enter the first row of the next non-empty group, and moving up should behave symmetrically in reverse. Actual: in the Dependencies pane for `beads-workbench-t7o.2`, when selection is in `Blocks (2)`, pressing Down on the last Blocks row does not advance into the `Structure (5)` list below it. However, pressing Up from the top of the Blocks list can move into the Structure list, so traversal is asymmetric and confusing. Repro: 1) `go build -o /tmp/bwb ./cmd/bwb` 2) run `BD_NON_INTERACTIVE=1 /tmp/bwb` from the repo root 3) navigate to `beads-workbench-t7o.2` and open detail mode 4) focus Dependencies with `LEFT` 5) move selection within `Blocks (2)` to the second row 6) press Down and observe selection does not enter `Structure (5)` 7) from the top Blocks row, press Up and observe selection can move into Structure. Impact: dependency browsing feels broken because list traversal order is not intuitive or consistent across group boundaries.

## Acceptance Criteria
- [ ] Dependency selection traverses non-empty grouped dependency rows in a consistent linear order across `Blocked by`, `Blocks`, `Related`, and `Structure`.
- [ ] Pressing Down on the last row of a group moves to the first selectable row of the next non-empty group.
- [ ] Pressing Up on the first row of a group moves to the last selectable row of the previous non-empty group.
- [ ] Real-app verification on `beads-workbench-t7o.2` confirms symmetric traversal between `Blocks` and `Structure`.
