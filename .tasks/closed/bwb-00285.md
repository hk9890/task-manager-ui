Discovered while implementing beads-workbench-0mj1. metadataGroups() in internal/ui/details/metadata.go builds the "Counts" group with Comments / Blocked by / Blocks / Related but omits the Children count, in BOTH the skeleton branch (~lines 168-174) and the real branch (~lines 175-182). An epic with children therefore shows no children-count in the metadata rail, inconsistent with the Dependencies pane (which renders a Children group) and the compact summary line (fixed in 0mj1).

EXPECTED: Counts group shows a "Children: N" row using len(detail.Children), ordered after "Related" to mirror the Dependencies-pane group order (Blocked by -> Blocks -> Related -> Children). Always shown, including "Children: 0", matching the always-show-all-counts behavior of the other relationship counts. Skeleton branch shows the skeletonCountValue ("░░") placeholder for Children too.

ACTUAL: No Children row appears in the metadata-rail Counts group.

REPRO: Open the details view for an epic that has child issues; the metadata rail Counts block lists Comments/Blocked by/Blocks/Related but no Children.

FIX: Add a metadataField{label: "Children", ...} after "Related" in both branches of the Counts group in metadata.go -- skeletonCountValue in the skeleton branch, fmt.Sprintf("%d", len(detail.Children)) in the real branch. Mirrors the 0mj1 fix in details.go.

ACCEPTANCE CRITERIA:
- Counts group renders "Children: N" after "Related" in the real branch, and "Children: ░░" in the skeleton branch.
- A golden test pins the Children row for an issue with children > 0 (mirroring 0mj1's TestRenderCompactWithChildrenGolden), and affected goldens under internal/ui/details/testdata/ are regenerated.
- `go test ./internal/ui/details/...` passes and `mise run quality:fast` is green.

IMPACT: Low -- cosmetic/parity gap in a summary surface; no data or behavior loss.
