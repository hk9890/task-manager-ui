## Description

The edit-document parser is vulnerable to marker confusion. The renderer emits `## Description` BEFORE the status/type/priority/assignee/labels fields inside the editable block, and `issueEditExtractField` locates each field with `strings.Index` (first occurrence). Because description content appears earlier in the editable block, a description containing forged markers like `<!-- BWB:FIELD:LABELS:BEGIN -->malicious<!-- BWB:FIELD:LABELS:END -->` is picked up as the labels payload on save — silently overwriting the real value the operator sees later in the document.

A description containing `<!-- BWB:EDITABLE:END -->` also truncates the entire editable section, causing the parser to fail field extraction with no specific guidance to the operator (DoS variant).

This is a Major finding from the 2026-05-16 audit (L1).

## Files to Modify

- `internal/domain/issue_edit_document.go:48-205`
- new test: `internal/domain/issue_edit_document_security_test.go`

## Instructions

1. On render: scan the description body for any `BWB:` marker tokens (`BWB:EDITABLE:BEGIN/END`, `BWB:FIELD:*:BEGIN/END`). If detected, EITHER:
   - Reject the render with a clear error (preferred — keeps parse symmetric).
   - Zero-width escape them (insert a sentinel char that is stripped on parse).
   Pick rejection unless there is a real use case for legitimate marker-shaped text.
2. On parse: search for status/type/priority/assignee/labels markers AFTER locating their corresponding section heading, not by first match across the whole editable block. Alternatively, find the LAST occurrence inside the editable block (more conservative).
3. On parse: if multiple `BWB:EDITABLE:END` markers appear, fail explicitly rather than silently truncating.
4. Add `issue_edit_document_security_test.go` covering:
   - Description containing `BWB:FIELD:LABELS:BEGIN ... END` — assert parse rejects or the real labels (not the injected ones) win
   - Description containing `BWB:FIELD:STATUS:BEGIN ... END` — same
   - Description containing `BWB:EDITABLE:END` — assert parse fails with a specific error mentioning the conflict
   - Round-trip property test: render → parse should fail closed if description contains any `BWB:` token
