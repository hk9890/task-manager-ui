## What

Add an in-flight guard inside the destructive functions themselves, using a dedicated `inflight bool` field, so future callers cannot accidentally bypass the protection added at the call sites.

- internal/mode/board/model.go: startReload (line 333) — early-return if `m.inflight` is true.
- internal/mode/search/model.go: triggerSearchWithAnchor (line 306) — early-return if `m.inflight` is true (or reuse `m.loading` if the semantics align cleanly).

## Why

The call-site guards (5q6t.1, 5q6t.2) are the user-facing fix. But startReload is a single function that wipes pendingResults, partial buffers, focus, and selection — any future caller that forgets to check IsLoading first will re-introduce the bug. Push the invariant down to where the state is mutated.

## Design (pinned)

Introduce a dedicated `inflight bool` field on the board Model:

- `inflight` starts as `false` on Model construction.
- `startReload` sets `inflight = true` after passing its guard.
- `maybeCompose` sets `inflight = false` when `pendingResults == 0` and composition runs.
- The guard at the top of `startReload` is `if m.inflight { return nil }` — and ALSO emits a debug log so internal re-entry attempts are visible.

**Why a dedicated bool instead of reusing IsLoading():**

`IsLoading()` returns true when ANY column has `loading=true`. But `initialLoadingColumns()` (called once at construction) sets all columns to loading=true. So at Init() time, `IsLoading()` is already true — guarding startReload with IsLoading would block the cold-start load. A dedicated `inflight` bool, false at construction, true only while a startReload is genuinely in flight, has no Init/cold-start ambiguity.

For search, `m.loading` is already set/cleared correctly by triggerSearchWithAnchor and the load-result handler, so search can reuse `m.loading` for the internal guard — no new field needed.

## Update IsLoading() to align

Optionally update `IsLoading()` to delegate to `m.inflight` (single source of truth) so the AutoRefresh guard and the new internal guard share the same signal. This is a small refactor that prevents drift.

## Dependencies

- Best landed AFTER 5q6t.1, 5q6t.2, and 5q6t.4 — defense-in-depth on top of working call-site guards.

## Acceptance

- A second `startReload(refreshModeManual)` call mid-flight does not reset pendingResults or partial buffers and does not call `initialLoadingColumns()` a second time.
- A second `startReload(refreshModeAuto)` call mid-flight is also a no-op (auto-refresh callers should respect the same invariant).
- Init() still loads data on first run (pendingResults populated, all 4 columns get data after gateway responses land).
- pendingResults never goes negative under any sequence of legitimate or rogue caller invocations (extend 5q6t.4's test to also drive `m.startReload` directly to exercise the internal guard).
- Search's triggerSearchWithAnchor early-returns when m.loading is already true (assert via fake gateway's Calls slice).
- All existing tests in internal/mode/board and internal/mode/search continue to pass.
