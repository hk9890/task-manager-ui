What I did: built /tmp/bwb, ran it against the embedded fixture via scripts/capture_bwb_screen.py, opened the in-app help overlay with '?', and compared it to docs/user-guide/key-bindings.md and internal/config/keybindings.go DefaultKeyBindings().

Expected (source: docs/user-guide/key-bindings.md lines 13-17): mode switching documented as 'f13 = board, f14 = search, f15 = cycle next, f16 = cycle prev', presented as ordinary keys an operator can press.

Actual:
1. The in-app help overlay ('?') does NOT mention f13-f16 at all. It presents mode switching as only 'ctrl+space = toggle Board/Search' and '3 = open selected issue detail'. So the documented mode keys and the in-app help disagree on the mental model.
2. f13-f16 are Shift+F1..F4 terminal sequences. docs/user-guide/key-bindings.md lists them as plain 'f13' etc. with no hint that an operator must press Shift+F1-4; many keyboards/terminals cannot produce them at all.
3. mode_cycle_next / mode_cycle_prev are bound ONLY to f15/f16 in DefaultKeyBindings() (internal/config/keybindings.go). There is no alternative default binding and the help overlay never mentions mode cycling. On a standard keyboard with default config, the mode-cycle feature is effectively unreachable/dead.

Not a lockout: Board stays reachable via ctrl+space toggle and esc-from-detail. Severity: low-medium (rough edge + doc/UI inconsistency, plus one functionally-dead default feature).

Repro:
  go build -o /tmp/bwb ./cmd/bwb
  # seed an embedded fixture, run /tmp/bwb, press '?' -> help overlay shows no f13-f16
  # compare to docs/user-guide/key-bindings.md Shell/Global section

Fix options: (a) change the default mode-switch / mode-cycle bindings to keys a normal keyboard can produce and that the help surfaces; (b) if f13-f16 are deliberate, document them in key-bindings.md as Shift+F1-F4 and add them (incl. mode cycling) to the in-app help, OR give mode_cycle_next/prev a usable secondary default binding.
