modal/modal.go:229-233 forwards any non-KeyMsg (including WindowSizeMsg which is handled above) to the focused text input's Update method. While WindowSizeMsg is caught first, any other future message types (custom msgs, tick msgs) will leak into textinput.Update when focusedInput >= 0. The textinput will silently ignore them now, but this is a latent coupling. Not critical but worth noting. Suggested action: consider guarding the textinput forwarding to only pass tea.KeyMsg types, making the intent explicit.
