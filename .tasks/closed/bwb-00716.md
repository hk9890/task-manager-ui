## Outcome

A `CachingRepository` decorator at `internal/repository/caching/caching.go` wraps `memory.Repository` (fast-path) + `repository.Repository` (source of truth, expected to be a `beads.Repository`). Implements the full Repository interface.

## Reads (per-method)
- `Issue(id)`: if the issue is in memory, return it. Otherwise `backing.Issue(id)`, store in memory, return.
- `Dashboard()`: if a non-stale cached DashboardData exists, return it. Otherwise `backing.Dashboard()`, cache it, return. Staleness is set by kxci.3's tick or by mutations; for THIS task, a single in-memory bool "dirty" is enough (cleared on fetch, set on any mutation).
- `Search(q)`: ALWAYS pass through to `backing.Search(q)` (no caching — search is queryable and not easily keyed).
- `Catalogs()`: TTL-cached (default 5min). If cached and `time.Since(fetchedAt) < ttl`, return cached. Otherwise `backing.Catalogs()`, store with new `fetchedAt`, return.
- `HealthCheck()`: pass through to backing.

## Writes (all synchronous: backing first, memory on success)
- `CreateIssue`: `backing.CreateIssue` → on success, mark Dashboard dirty AND seed the new issue into memory (best-effort; if seed fails, leave dirty so next Dashboard refetches)
- `UpdateIssue(id, ...)`: `backing.UpdateIssue` → on success, drop the cached `Issue(id)` AND mark Dashboard dirty
- `CloseIssue(id, ...)`: same as UpdateIssue
- `AddComment(id, ...)`: `backing.AddComment` → on success, drop the cached `Issue(id)` (Dashboard need not be dirty unless comments affect Dashboard projection — they don't)

## Constructor
```go
func New(backing repository.Repository, opts ...Option) *CachingRepository
```
Options: `WithCatalogsTTL(d time.Duration)`, `WithClock(func() time.Time)`. Defaults: TTL=5min, clock=time.Now.

Internal memory uses a per-decorator `*memory.Repository` instance (do NOT reuse the caller's memory instance — keep ownership clean).

## Concurrency
- All methods safe for concurrent use; use `sync.RWMutex` around cache state
- Per the Repository interface godoc: reads may overlap; writes serialize

## Files (new)
- `internal/repository/caching/caching.go`

## Test coverage
- Each read method: cache miss → backing called → cache populated → second call hits cache (no second backing call)
- Each write method: backing called, on success cache updated/invalidated; on failure cache untouched
- Catalogs TTL: before TTL → cached; after TTL → refetched
- Race-clean: tests run with `-race`
- Concurrent reads + concurrent writes do not data-race (table-driven goroutine test)

## Out of scope for THIS task

- File persistence → kxci.4
- App wiring → kxci.5
- Dirty flag is a single bool; per-ID Dashboard reconstruction is a future optimization
