What I did: Launched bwb against the embedded seed fixture (5 issues bwf-1..5) and read all 4 board columns via PTY capture (scripts/capture_bwb_screen.py).

Expected (source: project-plan triage-quality browsing intent + docs/OVERVIEW.md): every non-trivial issue in the repo is visible somewhere on the board so an operator can triage it.

Actual: Board columns total 1+2+0+1 = 4 of the 5 fixture issues. bwf-5 ('Stored-blocked task with no dependency', status=blocked, assignee=eve, open work) appears in NO column:
- Not Ready (1): bwf-2 only — fed by ReadyExplain .Blocked, which lists only dependency-blocked issues.
- Ready (2): bwf-1, bwf-4 — ReadyExplain .Ready.
- In Progress (0): Query(status=in_progress).
- Done (1): bwf-3 — Query(status=closed).
bwf-5 has status=blocked but no dependency blocker, so bd ready/blocked both exclude it; it is not in_progress and not closed. It falls through all four board buckets.

Root cause: internal/mode/board/model.go composes the board from ReadyExplain + Query(in_progress) + Query(closed). There is no bucket for stored status=blocked issues that are not dependency-blocked. The seed fixture deliberately includes bwf-5 as this edge case; its comment only asserts absence from 'bd blocked', and nothing covers board visibility.

Severity: medium. Reproducing requires a stored status=blocked issue with no dependency, which 'bd update <id> --status blocked' produces. For a triage board the consequence is a real open issue silently invisible to the operator (FEW HICCUPPS: Product purpose, World/user mental model).

Repro:
1. repo=$(mktemp -d); sh internal/testing/e2e/embeddedfixture/setup.sh "$repo" internal/testing/e2e/embeddedfixture/seed.json
2. bwb --cwd "$repo"  (or capture_bwb_screen.py)
3. Read all 4 columns (h/l to scroll): totals are 1/2/0/1; bwf-5 is absent everywhere.
4. (cd "$repo" && bd list) shows 4 issues incl. bwf-5.
