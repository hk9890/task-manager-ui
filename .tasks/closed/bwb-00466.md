## Description
The five mutation branches in `internal/app/model.go` (UpdateIssue full, CloseIssue, AddComment, status-only UpdateIssue, priority-only UpdateIssue) each wrap a gateway call in an error path that emits a documented toast prefix. None of these error paths have tests. The `FakeBeadsGateway.SetError(method, err)` seam already exists for all 16 methods — this task uses it.

**Test seam (recommended):** Call the unexported `submitMutationCmd(services, state, values)` directly from `package app` tests (it's at `internal/app/model.go:1653`). It takes plain inputs, returns a `tea.Cmd` that yields `mutationResultMsg`. This is ~15 lines per test vs. ~50 lines if you drive through `Update + modal.SubmitMsg`. The toast string originates inside `submitMutationCmd`; routing through `Update → showToast(err.Error())` at `model.go:548` is already covered by the existing success-path tests.

## Files
- `internal/app/model.go` (production, read-only here):
  - `:1732` `mutationUpdate`   → "update issue failed: %w" (Gateway.UpdateIssue)
  - `:1739` `mutationClose`    → "close issue failed: %w" (Gateway.CloseIssue)

## Instructions
1. Declare a sentinel `var errInjected = errors.New("injected mutation failure")` once at the top of the test file.
2. For each of the five mutation kinds, write a test that:
   - Builds a `FakeBeadsGateway` via the existing test helpers.
   - Calls `gateway.SetError(<MethodConstant>, errInjected)`:
     - mutationUpdate, mutationStatus, mutationPriority → `fakes.MethodUpdateIssue`
     - mutationClose → `fakes.MethodCloseIssue`
     - mutationComment → `fakes.MethodAddComment`
   - Builds the minimal `services` value the existing tests use (look at `internal/app/model_test.go:1017` for the synthesis pattern, or `:1595–1614` for the modal-driven pattern — copy whichever helper builder fits the new test file).
   - Builds the appropriate `mutationDialogState` (look at `submitMutationCmd`'s signature at `internal/app/model.go:1653` and the call sites for `kind` and `issue` shape).
   - Builds the appropriate `values map[string]string` (e.g. `{"status": "in_progress"}` for mutationStatus, `{"priority": "1"}` for mutationPriority, etc. — copy from the success-path call sites).
   - Invokes the cmd: `msg := submitMutationCmd(services, state, values)()`.
   - Type-asserts to `mutationResultMsg`.
   - Asserts BOTH:
     - `strings.HasPrefix(res.err.Error(), "<documented prefix>")` for the specific prefix (`"update issue failed:"`, `"close issue failed:"`, `"add comment failed:"`, `"update status failed:"`, `"update priority failed:"`).
     - `errors.Is(res.err, errInjected)` — this catches a silent `%w → %v` regression (which would still leave `Error()` looking identical but break the unwrap chain).
3. All three UpdateIssue-backed paths (update, status, priority) use the same `MethodUpdateIssue` injection. The injected error surfaces identically; what differs is the wrapper string. The three separate tests each pin one wrapper.
4. Falsifiability check (manual, once): temporarily edit `internal/app/model.go` to change one wrapper from `fmt.Errorf("update status failed: %w", err)` to `fmt.Errorf("update status failed: %v", err)`. Confirm the `errors.Is` assertion now fails. Restore. Then temporarily delete one wrapper entirely (return the raw error). Confirm the prefix assertion now fails. Restore. Do not commit either change.
