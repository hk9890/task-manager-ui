## Description
The Detail Content pane's loading skeleton renders the description body as board issue-row
shapes ("X XX XXX ░░░ ░░░" via issuerow.RenderCompactSkeleton) instead of prose-like
placeholder lines, so it reads as "a list of issues loading", not a paragraph body.

## Repro
1. Open any issue detail and observe the Content pane during the async load window (~2.5s on embedded dolt), or on a preview/refresh.
Observed: rows shaped like compact board rows where description paragraphs should be.

## Root cause (verified)
internal/ui/details/details.go: the skeleton branch in renderContentPaneLines (~lines 709-722)
uses issuerow.RenderCompactSkeleton with n := max(1, availableHeight-len(upper)). Correct for
the Dependencies rail (a list); wrong for the Content/Description body (prose).

## Instructions
1. Add a prose-style skeleton for the Content body: a short heading bar, then a few paragraph blocks of varying-width ░ lines with blank-line gaps, pulsing on the same styles.SkeletonShades phase (keep the animation language consistent).
2. Apply it ONLY to the Content description-body skeleton path; keep the Dependencies rail using the row skeleton.
3. Keep the header (meta row + title + rule) rendering during skeleton as-is, and PRESERVE the skeleton line-count/height: the binding check is TestColdStartSkeletonLayoutStabilityMatchesLoadedDetail (internal/ui/details/details_test.go:779), which compares lipgloss.Width/Height of skeleton vs loaded detail. The prose skeleton must keep the SAME line count/height. Also keep TestColdStartSkeletonContainsAllThreePaneSectionHeaders (~848) passing.

## Files to Modify
- internal/ui/details/details.go
- internal/ui/details/details_test.go
NOTE: there is NO skeleton golden file (no details golden contains skeleton glyphs) — there is nothing to "regenerate". Coverage is the width/height stability test above plus a new shape assertion (below).
