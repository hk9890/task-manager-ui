## Description

`internal/dashboard/definition.go:88-94` (`ValidateQuery`) only checks that `Type` is a known enum. It does NOT detect provider definitions that combine `Type: QueryTypeListIssues` with all-zero filters and `Limit == 0` — which silently issues an unbounded list query at startup. The doc comment acknowledges payload-field validation is hard ("value types, so this contract cannot reliably distinguish unset from zero") but `Limit == 0` is treated as "uncapped" in `runSectionQuery` / `applyQueryLimit`, so the result is a real risk.

Major finding U4 from the 2026-05-16 audit.

## Files to Modify

- `internal/dashboard/definition.go`
- new test: extend `internal/dashboard/definition_test.go`

## Instructions

1. Add a soft validation path (separate from the hard `ValidateQuery` rejection) that warns when a definition has zero filters AND `Limit == 0`. Two options:
   - Make `ValidateQuery` reject it (breaking change for any consumer that intentionally wants uncapped).
   - Add a `WarnQueryRisk` function returning structured warnings; have the board provider log them via slog at startup.
   Pick option 2 unless there is a clear reason to harden. Document the choice in code + `docs/CODING.md` if you change rule #11.
2. Either way: explicitly document in the godoc that `Limit == 0` means "uncapped".
3. Add a test asserting an empty-filter + zero-limit definition produces a warning (or rejection).

## Acceptance Criteria

- [ ] Unbounded query risk is either rejected or logged as a structured warning at startup
- [ ] `Limit == 0` semantics documented explicitly
- [ ] Test covers the empty-filter + zero-limit case
- [ ] Full quality gate green
