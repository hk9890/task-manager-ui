## Description
Redesign the issue detail screen so it reads as a structured operator workspace instead of a flat metadata dump. The new layout should keep Dependencies on the left in a boxed section, place the main issue details in the center, and use a simple grouped metadata/action rail on the right separated by a vertical rule.

## Decisions Captured From Discussion
- Left side stays dedicated to dependency information.
- Rename the left section to `Dependencies`.
- The dependencies section must use the same bordered section treatment as the dashboard (`styles.FormSection` / same rounded border styling).
- The border/boxed surface should be implemented as one reusable component approach rather than ad-hoc one-off box rendering.
- Dependency issues inside that section should render with the same compact issue-row language used in the dashboard, and should reuse the dashboard/shared issue rendering component where possible.
- The same dependency rendering approach should also be used anywhere else in issue detail that shows structured issue relationship data / side-panel issue structure data.
- Center column is a single content component/pane for this pass.
- Within that center pane, issue details/body take the dominant vertical share and comments are rendered as a smaller lower subsection; comments are NOT a separate focus pane in this pass.
- Comments should render with markdown-aware coloring/styling via a reusable markdown-render component.
- The same markdown-render component should be used for both issue details/body text and comment text.
- Right side should not use separate boxes; it should be a grouped metadata rail separated by a vertical line.
- Add a `Quick actions` group in the right rail.
- The layout must use the full available screen height instead of leaving large dead areas.
- Dependencies should use effectively the full available vertical span on the left side.
- The main issue details area should use effectively the full available vertical span in the center.
- Comments should be visually bounded to a smaller portion of vertical space (roughly ~20% of the usable height, not half the screen).
- Height must be an explicit renderer input so vertical allocation is implementable and testable.
- UX/interaction details must be explicit enough that the reviewer and implementer do not need to guess navigation or focus behavior.
- After implementation, manually open the real app and inspect the detail screen to ensure it actually looks good and uses space well.

## Interaction / Overflow Rules
- Detail mode remains a 3-pane model for this pass: `Dependencies` | `Content` | `Metadata`.
- Comments belong to the `Content` pane and do not receive separate pane focus.
- Content-pane scrolling covers the full center pane as one scrollable region (issue details first, comments below).
- Dependencies pane scrolls independently if dependency content exceeds visible height.
- Metadata rail scrolls independently only if needed; otherwise it remains statically visible.
- Dependency row rendering is visual in this pass; direct dependency-row navigation/open behavior is out of scope unless it falls out naturally with minimal extra work.
- Responsive fallbacks must remain usable: narrower layouts may collapse to simpler arrangements, but the wide/detail target layout should honor the dense full-height design.

## UX / Interaction Requirements
- Dependencies groups (`Blocked by`, `Blocks`, `Related`) must remain readable and clearly grouped.
- The right rail must communicate available actions clearly enough that operators understand what can be done from detail mode.
- The new composition must not make comments dominate the screen when description/dependency context is the primary working surface.
- Focus, scrolling, and panel emphasis behavior are intentionally defined above and should not be reinvented during implementation.
- If dependency-row interaction is later desired, capture it as follow-up work instead of silently expanding scope here.

## Technical Reuse Requirements
- Reuse the shared bordered section component already used by the dashboard.
- Reuse the shared compact issue renderer used by the dashboard for dependency issue rows where possible.
- Reuse/extend the shared markdown-render component for both issue details and comments.
- Avoid duplicating near-identical rendering logic for issue relationship groups or markdown text blocks when an existing shared component can be extended.
- Prefer extending the current detail layout machinery rather than rewriting it from scratch.

## Success Criteria
- [ ] Detail renderer accepts explicit height so vertical allocation is deterministic and testable.
- [ ] Detail layout uses a left Dependencies box, center Content box, and a right grouped metadata rail separated by a vertical divider.
- [ ] Dependencies rendering reuses the dashboard section border style and the shared compact issue-row visual language.
- [ ] The same structured issue rendering approach is usable for dependency/relationship panels elsewhere in detail mode.
- [ ] A reusable markdown-render component is used for both issue details and comment text.
- [ ] Comments render with markdown-aware styling/colors instead of plain wrapped text.
- [ ] The right rail includes grouped metadata plus a Quick actions group without separate boxes.
- [ ] The layout uses the full available terminal height without leaving large empty dead zones.
- [ ] Comments are visually constrained to a smaller vertical share than the main details and dependencies regions.
- [ ] UX/interaction details are explicit enough for implementation and verification.
- [ ] Automated UI coverage is updated for the new layout.
- [ ] Built-binary runtime verification is performed against the embedded fixture repo, including opening an issue detail screen and visually confirming the layout looks good and uses space well.
