## Description
Implement the minimal v1 configuration model needed by the standalone app so editor commands, launcher definitions, and basic UI preferences do not emerge as ad-hoc constants scattered across features.

## Instructions
1. Define the configuration model for:
   - editor command / `$EDITOR` fallback
   - launcher definitions or launcher defaults
   - basic UI preferences needed by the app shell
2. Keep the configuration surface intentionally small and aligned with the v1 architecture.
3. Create a clear package boundary for config loading and access.
4. Document the configuration model and defaults in `docs/CODING.md` or `docs/TESTING.md` where appropriate.
5. Do not introduce a large legacy-style config system.

## Files to Modify
- configuration package under `internal/` (create)
- `internal/app/` and/or `internal/launcher/` (integrate config access as needed)
- `docs/CODING.md` (update)

## Acceptance Criteria
- [ ] A minimal v1 configuration model exists
- [ ] Editor command and `$EDITOR` fallback behavior are defined
- [ ] Launcher defaults are defined without a large custom configuration surface
- [ ] Basic UI preferences have a clear configuration home
- [ ] Documentation reflects the configuration model and defaults
