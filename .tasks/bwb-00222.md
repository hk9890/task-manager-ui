## Description
Follow-up from beads-workbench-42hk. The parity-contract package (~175s) is dominated by ONE scenario: S16 `PaginatedClosedFetch` at ~137s (78% of wall time). It seeds 110 issues via 220 sequential `bd` subprocess calls (110 create + 110 close) built inline at `internal/repository/contract/parity_integration_test.go:1326-1350`, bypassing `impl.build`. All other scenarios run in parallel behind it, so S16 is the only lever that moves wall time.

## Open Questions
- [ ] Can `seedCount` drop below 110 while still exercising real two-page pagination? (needs >=100 issues for two non-overlapping 50-issue pages as currently asserted)
- [ ] Alternatively, reduce `pageSize` (e.g. 10) so `seedCount` ~25 still yields multi-page coverage — is a smaller page size an acceptable pagination contract?
- [ ] Or parallel-seed S16's bd calls via goroutines — acceptable given real bd concurrency behavior and test reliability?

## Why This Needs Discussion
Every option trades pagination-contract coverage or test-reliability for speed. This is a coverage-vs-runtime decision, not a mechanical refactor. Decide the tradeoff before implementing.
